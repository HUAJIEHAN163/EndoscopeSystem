## 四、问题时间线

按开发顺序，FPS 问题的发现和应对过程：

### 阶段 1：首次暴露（问题 #3）

**触发**：实现临床模式预设，同时开启白平衡+CLAHE+去雾+锐化+降噪

**现象**：FPS 从 ~30 骤降到个位数

**应对**：
- 去雾 CV_64F→CV_32F（速度翻倍）
- 降噪缩小后处理再放大（4x 提升）
- 锐化 sigma=3.0 大核改 3x3 固定核（30x 提升）

**认知**：算法本身太慢，需要逐个优化

### 阶段 2：OOM 危机（问题 #11）

**触发**：开算法后主线程处理慢，信号队列堆积，QImage 拷贝耗尽内存

**现象**：程序被 OOM Killer 杀掉（864MB）

**应对**：经历 4 个版本迭代，最终用 atomic 计数器限制队列深度

**认知**：FPS 不足不仅影响体验，还会导致系统级故障（OOM）

### 阶段 3：系统性测量（问题 #12）

**触发**：需要量化各算法耗时，找到真正瓶颈

**应对**：
- 降分辨率处理 640→320（4-5x 提升）
- CLAHE 对象复用 + 启动预热
- 管线开销优化：去掉 frameBuffer 深拷贝（省 5ms）、去掉重复 scaled（省 16ms）

**结果**：白平衡 FPS 8→16-18，CLAHE FPS 5→11-13

**认知**：算法之外的管线开销（拷贝、缩放、转换）同样是大头

### 阶段 4：Overrun 根因定位（问题 #14）

**触发**：开 CLAHE 后 dmesg 持续报 overrun（15% 错误率）

**关键发现**：采集线程 YUYV→BGR 从 37.5ms 涨到 48.4ms——采集线程本身没变，是 CPU 被其他线程抢占

**认知升级**：**问题不是某个算法慢，而是单核 CPU 上多线程争抢导致所有环节都变慢**

### 阶段 5：三线程管线（P3，13_6）

**目标**：算法移到独立线程，UI 不再被阻塞

**结果**：UI 流畅了，但 FPS 未提升（处理线程成为新瓶颈）

**副作用**：不开算法 FPS 从 22-23 降到 ~20（多了一个线程的 CPU 调度开销）

**为什么不撤回**：
- P3 解决的是产品可用性问题：开算法时 UI 完全卡死（按钮无响应、画面停顿），这在产品中是不可接受的
- 算法设计不再受性能妃协：之前锐化不能用 Lab 空间（多 20ms 不可接受），P3 后算法耗时变成延迟而不是卡顿，可以自由选择最优方案
- 三线程是后续所有优化的基础架构：P7.1 全程 BGR、P16 零拷贝、P6 NEON 加速都基于三线程架构
- 2-3fps 的损失远小于 UI 流畅性的收益

**新发现**：
- 不开算法时 QImage::scaled 耗时 25-55ms（应该 5-10ms）
- 开算法时 QImage↔Mat 转换占 3-22ms/帧
- 颜色转换链路冗余：每帧 4-5 次转换

**认知**：三线程解决了 UI 卡顿，但 FPS 瓶颈从"主线程阻塞"变成了"处理线程速度"

### 阶段 6：全程 BGR（P7.1，13_7）

**目标**：消除冗余颜色转换，每帧从 4 次降到 1 次

**结果**：不开算法时 FPS 仍 17-20fps，resize 仍 22-59ms

**副作用**：FrameQueue 从 QImage（隐式共享，赋值近乎零开销）改为 cv::Mat（clone 每帧 2×900KB 真拷贝），导致不开算法 FPS 从 ~20 降到 17-20

**为什么不撤回**：
- 开算法时有实际收益：CLAHE 耗时波动从 36-80ms 收窄到 54-65ms，处理更稳定（省掉了 QImage↔Mat 转换的不确定性）
- 录像零转换：直接用 BGR Mat 写入 VideoWriter，之前每帧需要 QImage→Mat 转换
- 拍照零转换：直接用 cv::imwrite 保存 BGR Mat
- 架构清晰：全程 BGR 消除了数据在 BGR/RGB 之间来回转换的混乱，代码更易维护
- 为 P16 零拷贝铺路：clone 开销是可以通过 FrameQueue 重新设计消除的，而颜色转换的消除是永久收益
- 为 P5 NEON 铺路：NEON 写 YUYV→BGR 直接输出 BGR Mat，不需要再转 RGB（虽然 P5 最终失败，但架构决策是正确的）

**关键认知**：P7.1 省掉的 10-20ms 被 CPU 调度波动掩盖了。在单核 650MHz 上，三个线程的 CPU 密集操作互相拖慢，单个环节的优化效果被系统级波动淹没。但 clone 开销是新引入的确定性损失，需要通过 P16 解决。

### 阶段 7：NEON 加速尝试（P5，13_9）

**目标**：手写 NEON intrinsic 替换 cvtColor，预期 cvtColor 40ms→10ms

**结果**：
- v1（vld1 + 标量 U/V 展开）：~65ms，比 cvtColor 慢 63%
- v2（vld4 + vzip）：~57ms，比 cvtColor 慢 43%

**关键发现**：OpenCV ARM 版本已启用 NEON 优化（`CV_NEON 1`），cvtColor 内部的 YUYV→BGR 已经是 NEON 优化过的。手写版本是在和 OpenCV 多年优化的实现竞争，无法超越。

**已回退到 OpenCV cvtColor。P5 方向关闭。**

**为什么不保留 NEON 版本**：
- v1 和 v2 都比 OpenCV cvtColor 慢，没有任何场景有收益
- 保留会降低 FPS，且增加代码复杂度
- neon_accel.h/cpp 文件保留但标注为未使用，供后续 P6 NEON 加速 CLAHE/去雾时复用

**认知修正**：之前假设“cvtColor 是未优化的浮点实现”是错误的。采集线程的 40ms 已经是 NEON 优化后的结果。这也说明采集线程的优化空间不在颜色转换，而在减少内存拷贝（P16）。

### 总结：问题演进脉络

```
算法太慢 → 降分辨率/优化算法 → 管线开销大 → 去掉冗余拷贝/缩放
    → 信号队列堆积 → 限制队列深度 → 主线程阻塞 UI → 三线程分离
        → 颜色转换冗余 → 全程 BGR（引入 clone 开销）
            → 手写 NEON 加速 → 失败（OpenCV 已内置 NEON）
                → CPU 调度争抢 + 硬件能力边界
```

每次解决一个瓶颈，就暴露出下一层瓶颈。当前到达的底层瓶颈是：**Cortex-A7 单核 650MHz 的物理算力限制 + 多线程调度开销 + clone 内存带宽消耗**。

当前可行的优化方向：减少 clone 次数（P16 FrameQueue 零拷贝）和加速处理线程算法（P6 NEON CLAHE/去雾）。

## 五、根因分析

### 5.1 核心结论

**FPS 不足的根本原因是 Cortex-A7 单核 CPU 上三个 CPU 密集线程争抢时间片，导致每个环节的实际耗时远超理论值。**

### 5.2 证据链

**证据 1：采集线程被处理线程拖慢**（问题 #14）

| 场景 | 采集线程 YUYV→BGR | 变化 |
|---|---|---|
| 不开算法 | 16-33ms（平均 37.5ms 含全流程） | 基准 |
| 开 CLAHE | 41-47ms（平均 48.4ms 含全流程） | +29% |

采集线程只做 cvtColor，不受算法影响。耗时增加只能是 CPU 被处理线程抢占。

**证据 2：处理线程简单操作耗时异常**（13_6 第九章、13_7 第八章）

640×480 → 460×360 的缩放，理论 5-10ms：

| 方案 | 实测耗时 | 说明 |
|---|---|---|
| QImage::scaled（P3 时期） | 25-55ms | 13_6 第九章 |
| cv::resize + QImage↔Mat（P3 尝试） | 34-87ms | 13_6 第十章，已撤回 |
| cv::resize 全程 BGR（P7.1） | 22-59ms | 13_7 验证 |

三种方案耗时范围相近，说明瓶颈不在算法选择或格式转换，而在 CPU 调度。

**证据 3：Overrun 率与处理负载正相关**（测试指南_Overrun手动测试.md）

| 场景 | 采集线程耗时 | Overrun 率 |
|---|---|---|
| 不开算法 | 37.5ms | 0% |
| 开 CLAHE | 48.4ms | 15.06% |

4 个 buffer 缓冲 132ms，采集+处理合计 92.4ms，理论不应 overrun。实际 15% overrun 说明线程调度导致了额外的不可预测延迟。

**证据 4：优化效果被调度波动掩盖**（13_7 验证）

P7.1 理论省 10-20ms/帧（去掉 3 次颜色转换），但实测 FPS 无变化。说明省下的时间被 CPU 调度的随机波动（±20ms）吞掉了。

### 5.3 问题模型

```
Cortex-A7 单核 650MHz
    ├── 采集线程: grabFrame(select等待) + cvtColor(YUYV→BGR) + clone
    ├── 处理线程: pop + resize/算法 + clone
    └── 主线程:   displayTimer + cvtColor(BGR→RGB) + QImage copy + paintEvent + UI

三个线程都是 CPU 密集型，在同一个核上轮转调度。
任何一个线程的 CPU 占用增加，都会拖慢其他两个线程。
```

Linux 默认调度器（CFS）对三个普通优先级线程平均分配时间片。当总 CPU 需求超过单核能力时：
- 每个线程的实际执行时间 = 理论时间 × (总需求/单核能力)
- 上下文切换本身也消耗 CPU（Cortex-A7 上约 5-10μs/次，但频繁切换累积可观）

### 5.4 为什么之前的优化有效但不够

| 优化 | 效果 | 局限 |
|---|---|---|
| 降分辨率 320×240 | 算法耗时降 4-5x | 处理线程仍占大量 CPU，影响采集线程调度 |
| 管线开销优化（省 21ms） | 白平衡 FPS 8→16-18 | 省的是固定开销，CPU 争抢的随机波动仍在 |
| 三线程分离 | UI 不卡顿 | FPS 未提升，反而多了一个线程争抢 CPU |
| 全程 BGR（省 10-20ms） | 理论有效 | 被 CPU 调度波动掩盖；且引入了 clone 开销 |
| P5 NEON 加速 YUYV→BGR | 失败 | OpenCV cvtColor 已内置 NEON 优化，手写无法超越 |

**共同局限**：都是减少单个环节的工作量，但没有减少 CPU 总负载。三个线程的总 CPU 需求仍然超过单核能力。

### 5.5 Cortex-A7 的硬件能力边界

STM32MP157 的 Cortex-A7 是 ARM 最低端的应用处理器核心（2011 年设计）：

| 参数 | Cortex-A7 (STM32MP157) | Cortex-A53 (i.MX8M) | 说明 |
|---|---|---|---|
| 主频 | 650MHz 单核 | 1.5GHz 四核 | 算力差 ~10x |
| 流水线 | 顺序 8 级 | 乱序 8 级 | A7 无法指令级并行 |
| NEON | 单发射 | 双发射 | A7 NEON 吞吐量只有 A53 的 1/4 |
| L1 Cache | 32KB | 32KB | 相同 |
| 内存带宽 | DDR3 ~2GB/s | DDR4 ~6GB/s | 内存密集操作差 3x |

**这不是设计方向错误**。STM32MP157 的定位是低功耗 IoT/工业控制，不是图像处理。在这个芯片上做实时图像处理是学习项目的性质，遇到性能瓶颈是正常的。

实际内窥镜产品的做法：
- 用更强的芯片（Cortex-A53/A72 四核 + 专用 ISP/NPU）
- 颜色转换由硬件 ISP 完成，不占 CPU
- 图像增强由专用硬件加速器或 FPGA 完成
- CPU 只负责 UI 和控制逻辑

本项目的价值在于学习完整的嵌入式图像处理链路（采集→处理→显示），而不是达到产品级性能。

### 5.6 解决方向

要突破当前瓶颈，必须**降低 CPU 总负载**，而不是在线程间挪移工作量：

| 方向 | 手段 | 预期效果 | 优先级 |
|---|---|---|---|
| 减少内存拷贝 | P16 FrameQueue 零拷贝（避免 clone） | 每帧省 1-2 次 900KB 拷贝，恢复不开算法 FPS 到 22+ | ★★★ 最高 |
| 降低处理线程 CPU | P6 NEON 加速 CLAHE/去雾 | 算法耗时降 2-3x，处理线程释放 CPU | ★★ |

**已排除的方向：**

| 方向 | 原因 |
|---|---|
| P5 NEON 加速 YUYV→BGR | 已验证失败：OpenCV cvtColor 已内置 NEON 优化，手写无法超越。详见 13_9 |
| 线程优先级调整（SCHED_FIFO） | 问题是 CPU 总量不够，不是分配不均，设高优先级会饿死其他线程 |
| 采集线程提前 resize | CPU 总负载不变，只是挪移耗时位置，且影响拍照全分辨率 |

其中 P16（FrameQueue 零拷贝）是当前最优先的，因为：
1. 直接减少 CPU 总负载（每帧少 1-2 次 900KB memcpy）
2. 恢复 P7.1 引入 clone 前的 FPS 水平
3. 改动范围小（只改 FrameQueue 和采集线程），风险低

### 7.2 采集线程各环节耗时（P7.1 后）

| 环节 | 耗时 | 数据来源 |

### 7.3 处理线程各环节耗时（P7.1 后）

| 环节 | 不开算法 | 开 CLAHE | 数据来源 |
|---|---|---|---|
| QImg→Mat | — | ✔ 已省掉 | P7.1 |
| 算法 | — | 54-65ms | 13_7 开发板日志 |

对比 P7.1 前：不开算法 25-55ms，开 CLAHE 52-98ms。开算法时波动范围收窄（省掉了 QImage↔Mat 转换）。

### 7.4 关键观察

- 不开算法时，处理线程只做 cv::resize，理论 5-10ms，实测 22-59ms——CPU 调度波动是主因
- 开 CLAHE 时，算法耗时 54-65ms，比 P7.1 前的 36-80ms 波动更小，说明去掉格式转换后处理更稳定
- 采集线程平均 40ms/帧，即采集帧率上限 ~25fps，这是不开算法时 FPS 的天花板

### 7.5 不开算法 FPS 逐步下降的原因追溯

| 阶段 | 不开算法 FPS | 变化原因 |
|---|---|---|
| 原始两线程（Demo） | 24-28fps | 基准，采集线程 + 主线程，两个线程争 CPU |
| 问题 #11 OOM 修复后 | ~22fps | atomic 计数器限流，丢弃部分帧 |
| 问题 #12 管线优化后 | 20-23fps | 持平，管线优化主要改善开算法场景 |
| P3 三线程改造后 | ~20fps | **多了一个处理线程争抢 CPU**，三线程上下文切换开销增加 |
| P7.1 全程 BGR 后 | 17-20fps | **FrameQueue 从 QImage 改为 cv::Mat**，clone 每帧 2×900KB 真拷贝，替代了 QImage 的隶开销隐式共享 |

**关键转折点及不撤回的理由详见第四章阶段 5（P3）和阶段 6（P7.1）。**

**可能的改善方向：**
- P4 内存池：预分配 Mat，用 copyTo 复用内存替代 clone 重新分配，减少内存分配开销
- 采集线程直接写入队列槽位（避免一次 clone）：需要重新设计 FrameQueue 的内存管理
