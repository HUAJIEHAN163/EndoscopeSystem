# 13_5 开发记录：三线程管线优化（P3）

## 一、背景

### 当前架构（两线程）

```
采集线程                              主线程
├── grabFrame                        ├── onFrameReady
├── cvtColor YUYV→BGR→RGB            │   ├── processFrame（算法）← 阻塞主线程
├── emit frameReady ──────────→      │   ├── update
└── 循环                             │   └── paintEvent（显示）
                                     ├── 按钮响应
                                     └── 其他 UI 事件
```

### 问题

1. 算法在主线程执行，阻塞 UI 刷新和按钮响应
2. 开 CLAHE（~28ms）+ 去雾（~87ms）时，主线程每帧耗时 ~130ms，FPS 降到 ~8
3. 算法设计被迫妥协性能（如锐化不能用 Lab 空间，因为多 20ms 不可接受）
4. m_pendingFrames 限制导致大量帧丢弃

```
*对第3点的详细说明
如果在 Lab 空间做锐化：
BGR → Lab          ~10ms
split              ~2ms
对 L 通道做 USM     ~7ms
merge              ~2ms
Lab → BGR          ~10ms
合计               ~31ms

当前在 BGR 空间直接做：
GaussianBlur       ~5ms
addWeighted        ~2ms
合计               ~7ms
多了 20ms 的颜色转换开销，在 Cortex-A7 上这个代价太大了。

效果区别：
BGR 空间锐化：对 B、G、R 三个通道同时锐化
  → 边缘处可能出现彩色伪影（颜色溢出）

Lab 空间锐化：只对 L（亮度）通道锐化，a、b 不动
  → 边缘更干净，不会出现彩色伪影

但在我们的场景下（320×240 降分辨率 + 内窥镜画面），BGR 锐化的彩色伪影不明显，7ms 和 31ms 的差距更重要。


CLAHE 为什么愿意付这个代价：
CLAHE 如果在 BGR 空间做，三个通道分别均衡化，颜色会严重失真——这是不可接受的。所以 CLAHE 必须在 Lab 空间做，颜色转换的开销是必须付的。
锐化在 BGR 空间做效果"还行"，所以选择省时间。这就是算法设计中的画质-性能权衡。
```

### 目标架构（三线程）

```
采集线程              处理线程（新增）          主线程
├── grabFrame         ├── processFrame        ├── paintEvent（只显示）
├── cvtColor          ├── CLAHE/去雾          ├── 按钮响应
├── push 到队列A ──→  ├── push 到队列B ──→    ├── 取队列B最新帧显示
└── 循环              └── 循环                └── 不再被算法阻塞
```

### 预期收益

| 指标 | 当前 | 优化后 |
|---|---|---|
| UI 响应 | 开算法时卡顿 | 始终流畅 |
| 显示 FPS | 受算法耗时限制（~8fps） | 接近采集帧率（~27fps） |
| 算法自由度 | 受性能妥协 | 可自由选择最优方案 |
| 延迟 | 算法耗时直接叠加 | 算法耗时变为延迟，不影响帧率 |

## 二、设计方案

### 2.1 数据流

```
采集线程：
  grabFrame → cvtColor(YUYV→BGR) → returnBuffer → cvtColor(BGR→RGB)
  → QImage copy → 写入 m_captureQueue（环形队列A）

处理线程（新增）：
  从 m_captureQueue 读取 → processFrame（CLAHE/去雾等）
  → 写入 m_displayQueue（环形队列B）

主线程：
  定时器/信号 → 从 m_displayQueue 读取最新帧 → paintEvent 显示
```

单生产者单消费者，无锁设计：

```cpp

    std::atomic<int> writeIdx{0};
    std::atomic<int> readIdx{0};
public:
    bool push(const QImage &frame);  // 生产者调用
    bool pop(QImage &frame);         // 消费者调用
```

### 5.1 Step 1：新增 FrameQueue 环形队列

新增文件：`src/utils/framequeue.h`

无锁环形队列，单生产者单消费者，用于线程间传递 QImage 帧数据。

设计要点：
- 固定大小数组（模板参数 N），不动态分配内存
- writeIdx 只有生产者写，readIdx 只有消费者写，用 atomic 保证可见性
- 队列满时覆盖最旧帧（移动 readIdx），保证生产者不阻塞
- 提供 latest() 方法，跳过中间帧只取最新帧（主线程显示用）

```cpp
#ifndef FRAMEQUEUE_H
#define FRAMEQUEUE_H

#include <QImage>
#include <atomic>

template<int N>
class FrameQueue {
    QImage m_frames[N];
    std::atomic<int> m_writeIdx{0};
    std::atomic<int> m_readIdx{0};

public:
    // 生产者调用：写入一帧
    // 队列满时覆盖最旧帧（移动 readIdx），保证生产者不阻塞
    void push(const QImage &frame) {
        int w = m_writeIdx.load(std::memory_order_relaxed);
        int next = (w + 1) % N;
        if (next == m_readIdx.load(std::memory_order_acquire)) {
            m_readIdx.store((m_readIdx.load(std::memory_order_relaxed) + 1) % N,
                            std::memory_order_release);
        }
        m_frames[w] = frame;
        m_writeIdx.store(next, std::memory_order_release);
    }

    // 消费者调用：读取一帧，返回 false 表示队列空
    bool pop(QImage &frame) {
        int r = m_readIdx.load(std::memory_order_relaxed);
        if (r == m_writeIdx.load(std::memory_order_acquire))
            return false;
        frame = m_frames[r];
        m_readIdx.store((r + 1) % N, std::memory_order_release);
        return true;
    }

    // 消费者调用：取最新帧，跳过中间所有帧
    // 用于主线程显示——只关心最新的处理结果
    bool latest(QImage &frame) {
        int r = m_readIdx.load(std::memory_order_relaxed);
        int w = m_writeIdx.load(std::memory_order_acquire);
        if (r == w) return false;
        int newest = (w - 1 + N) % N;
        frame = m_frames[newest];
        m_readIdx.store(w, std::memory_order_release);
        return true;
    }

    int size() const {
        int w = m_writeIdx.load(std::memory_order_relaxed);
        int r = m_readIdx.load(std::memory_order_relaxed);
        return (w - r + N) % N;
    }

    bool empty() const { return size() == 0; }
};

#endif // FRAMEQUEUE_H
```

### 5.2 Step 2：新增 ProcessThread 处理线程类

新增文件：`src/processing/processthread.h`

从 captureQueue 取帧 → 执行算法 → 写入 displayQueue。主线程从 displayQueue 取最新帧显示。

```cpp
#ifndef PROCESSTHREAD_H
#define PROCESSTHREAD_H

#include <QThread>
#include <QImage>
#include <atomic>
#include "utils/framequeue.h"
#include "utils/imageconvert.h"
#include "processing/imageprocessor.h"

class ProcessThread : public QThread {
    Q_OBJECT
public:
    explicit ProcessThread(FrameQueue<4> *captureQueue,
                           FrameQueue<4> *displayQueue,
                           QObject *parent = nullptr)
        : QThread(parent),
          m_captureQueue(captureQueue),
          m_displayQueue(displayQueue),
          m_running(false),
          m_quit(false) {}

    void pause() { m_running = false; }
    void resume() { m_running = true; }
    void stop() { m_quit = true; m_running = false; }

    // 算法配置（主线程通过 UI 控件更新，处理线程读取）
    ImageProcessor::Config config;

    // 畸变校正映射表（只读）
    cv::Mat undistortMap1, undistortMap2;

    // 显示区域尺寸
    int displayWidth = 640;
    int displayHeight = 480;

protected:
    void run() override {
        m_running = true;
        while (!m_quit) {
            if (!m_running) {
                QThread::msleep(10);
                continue;
            }

            QImage frame;
            if (!m_captureQueue->pop(frame)) {
                QThread::msleep(1);
                continue;
            }

            bool anyEnabled = config.clahe || config.undistort ||
                              config.dehaze || config.sharpen ||
                              config.denoise || config.edgeDetect ||
                              config.threshold;

            QImage result;
            if (!anyEnabled) {
                result = frame.scaled(displayWidth, displayHeight,
                                      Qt::KeepAspectRatio, Qt::FastTransformation);
            } else {
                cv::Mat src = ImageConvert::qimageToMat(frame);
                if (src.empty()) continue;
                cv::Mat dst;
                ImageProcessor::process(src, dst, config,
                                        undistortMap1, undistortMap2);
                cv::Mat display;
                cv::resize(dst, display, cv::Size(displayWidth, displayHeight),
                           0, 0, cv::INTER_LINEAR);
                result = ImageConvert::matToQImage(display);
            }

            m_displayQueue->push(result);
        }
    }

private:
    FrameQueue<4> *m_captureQueue;
    FrameQueue<4> *m_displayQueue;
    std::atomic<bool> m_running;
    std::atomic<bool> m_quit;
};

#endif // PROCESSTHREAD_H
```

### 7.2 FPS 未提升的原因分析

FPS 统计的是**处理线程输出的帧率**，不是显示刷新率。

```
采集线程：~27fps 产出帧 → 写入 captureQueue
处理线程：从 captureQueue 取帧 → 跑算法 → 写入 displayQueue
主线程：  16ms 定时器轮询 displayQueue → 取到帧才计数

处理线程是瓶颈：
  不开算法：处理 ~5ms/帧 → 输出 ~20fps（受采集线程限制）
  开 CLAHE：处理 ~28ms + 转换开销 → 输出 ~13fps
  开 CLAHE + 去雾：处理 ~115ms/帧 → 输出 ~8fps

主线程 60fps 轮询，但 displayQueue 里每秒只有 8 帧新数据
→ FPS 显示 ~8，和优化前一样
```

### 7.3 三线程优化的实际收益

三线程的目标不是提升 FPS，而是**解耦算法和 UI**：

| 收益 | 说明 |
|---|---|
| ✅ UI 不卡顿 | 主线程只做显示和 UI 事件，不跑算法 |
| ✅ 按钮响应流畅 | 即使算法耗时 115ms，按钮点击也能立即响应 |
| ✅ 算法设计自由 | 算法耗时不再直接影响 UI，可以用 Lab 空间做锐化 |
| ❌ FPS 不提升 | 处理线程是瓶颈，这是算法本身的物理限制 |

### 7.4 要提升 FPS 需要的后续优化

FPS 取决于处理线程的速度，要提升需要加速算法本身：

| 优化方向 | 预期效果 |
|---|---|
| P7.1 减少颜色转换次数 | 省 10-15ms/帧 |
| P5 NEON 加速 YUYV→RGB | 采集线程省 10ms/帧 |
| P6 NEON 加速 CLAHE/去雾 | 处理线程加速 2-3 倍 |
| 降低处理分辨率 | 已做（640→320），进一步降会影响画质 |

## 九、三线程管线性能基准数据（开发板实测）

### 9.1 采集线程（CAP）各环节耗时

| 环节 | 不开算法 | 开 CLAHE | 开 CLAHE+去雾 |
|---|---|---|---|
| YUYV→BGR | 13-33ms | 41-47ms | 17-28ms |
| BGR→RGB | 4-12ms | 5-9ms | 3-7ms |
| copy+push | 5-7ms | 5-10ms | 4-19ms |
| 平均合计 | ~32ms | ~33ms | ~35ms |

### 9.2 处理线程（PROC）各环节耗时

| 环节 | 不开算法 | 开 CLAHE | 开 CLAHE+去雾 |
|---|---|---|---|
| QImg→Mat | - | 2ms | 2-15ms |
| 算法（process） | - | 36-80ms | 143-161ms |
| resize→显示尺寸 | - | 12-23ms | 17-18ms |
| Mat→QImg | - | 1-7ms | 1ms |
| **合计** | 25-55ms（scaled） | 52-98ms | 176-182ms |

### 9.3 实际 FPS

| 场景 | 处理线程输出速率 | 屏幕显示 FPS |
|---|---|---|
| 不开算法 | ~24fps | ~20fps |
| 开 CLAHE | ~13fps | ~13fps |
| 开 CLAHE+去雾 | ~5fps | ~5fps |

### 9.4 发现的优化点

**优化点 1：不开算法时 QImage::scaled 耗时 25-55ms**

处理线程不开算法时用 `QImage::scaled(640×480 → 显示尺寸)` 缩放，耗时 25-55ms，是不开算法时的主要瓶颈。改用 `cv::resize` 预计可降到 5-10ms。

**优化点 2：CLAHE 算法波动大（36-80ms）**

之前测的 28ms 是 320×240 分辨率，但处理线程拿到的是 640×480 原图，`process()` 内部先 resize 到 320×240 再处理。波动可能来自 CPU 调度和缓存。

**优化点 3：采集线程开算法时 YUYV→BGR 变慢（16ms → 44ms）**

开 CLAHE 后采集线程的 YUYV→BGR 从 16ms 增加到 44ms，说明处理线程的 CPU 占用影响了采集线程的调度（与问题 #14 overrun 的根因一致）。

**优化点 4：颜色转换链路冗余**

```
采集线程：YUYV → BGR → RGB → QImage
处理线程：QImage → Mat(BGR) → 算法 → Mat → QImage
```
共 4-5 次颜色转换，每次 5-15ms。全程使用 BGR 可以省 2-3 次转换（P7.1 的内容）。

### 9.5 后续优化优先级

| 优先级 | 优化内容 | 预期收益 |
|---|---|---|
| 1 | 不开算法时 scaled 改为 cv::resize | 不开算法 FPS 提升到 ~27fps |
| 2 | P7.1 全程 BGR，减少颜色转换 | 每帧省 10-20ms |
| 3 | P5 NEON 加速 YUYV→BGR | 采集线程省 10ms |
| 4 | P6 NEON 加速 CLAHE/去雾 | 处理线程加速 2-3 倍 |

## 十、scaled 改 cv::resize 尝试（已撤回）

### 10.1 改动内容

处理线程不开算法时，将 `QImage::scaled` 改为 `cv::resize`，预期减少缩放耗时。

### 10.2 实测结果

| 方案 | 不开算法 resize 耗时 | 实际 FPS |
|---|---|---|
| QImage::scaled（原方案） | 25-55ms | ~20fps |
| cv::resize（新方案） | 34-87ms | 16-18fps |

**反而更慢了。**

### 10.3 原因

`cv::resize` 本身比 `QImage::scaled` 快，但改动后多了两次格式转换：

```
原方案：QImage → scaled → QImage（一步完成）
新方案：QImage → Mat（转换） → resize → QImage（转换）← 多了两次转换
```

`QImage → Mat` 涉及 RGB→BGR 颜色转换（~5-15ms），`Mat → QImage` 涉及 BGR→RGB 转换（~1-7ms），加起来比 scaled 省下的时间还多。

### 10.4 结论

已撤回改动。这个案例说明：**格式转换的开销比算法本身还大**，印证了 P7.1（全程 BGR 减少转换）的重要性。只有在全程 BGR 的架构下，cv::resize 才能发挥优势。

## 十一、后续 FPS 提升预估

### 11.1 当前瓶颈分析（全开 CLAHE + 去雾，5fps）

```
处理线程每帧耗时 ~180ms：
  QImg→Mat:     2-15ms   ← P7.1 全程 BGR 后可省掉
  算法(process): 140-160ms ← 内部包含 resize + CLAHE + 去雾
  resize→显示:   12-18ms
  Mat→QImg:      1-2ms    ← P7.1 全程 BGR 后可省掉
```

算法内部耗时拆分（320×240 分辨率）：
```
  resize 640→320:  ~2ms
  CLAHE:           ~28ms（BGR→Lab→均衡→Lab→BGR）
  去雾:            ~87ms（convertTo + split + erode + divide + merge）
  resize 320→显示: ~2ms
  合计:            ~119ms
```

实际测到 140-160ms，多出的 20-40ms 来自 QImage↔Mat 转换和 CPU 调度波动。

### 11.2 各优化手段的预期收益

| 优化 | 省多少 | 优化后每帧 | 预期 FPS |
|---|---|---|---|
| 当前 | - | ~180ms | ~5fps |
| P7.1 全程 BGR（省掉 QImg↔Mat 转换） | -20ms | ~160ms | ~6fps |
| P5 NEON 加速 YUYV→BGR（采集线程） | 采集线程省 10ms | 不直接影响处理线程 | ~6fps |
| P6 NEON 加速 CLAHE | CLAHE 28ms→10ms | ~142ms | ~7fps |
| P6 NEON 加速去雾 | 去雾 87ms→30ms | ~95ms | ~10fps |
| 以上全部 | | ~75ms | ~13fps |

### 11.3 各场景预估

| 场景 | 当前 | 全部优化后（预估） |
|---|---|---|
| 不开算法 | ~20fps | ~27fps（接近采集帧率上限） |
| 只开 CLAHE | ~13fps | ~20fps |
| 开 CLAHE + 去雾 | ~5fps | ~13fps |

### 11.4 说明

- 以上为理论预估，实际效果取决于 NEON 实现的优化程度
- 采集帧率上限约 27fps（受 YUYV→BGR 转换和 select 等待限制）
- 即使全部优化后，开 CLAHE + 去雾也很难超过 15fps，因为算法本身的计算量在那里
- 如果需要更高 FPS，需要进一步降低处理分辨率（320→160）或简化算法
