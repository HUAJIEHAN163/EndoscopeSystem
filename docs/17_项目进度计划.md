## 五、风险登记册

| 风险 | 可能性 | 影响 | 应对策略 | 实际发生 |
|---|---|---|---|---|
| Cortex-A7 算力不足 | 高 | FPS 无法达到 30fps | 降分辨率、减少算法、接受硬件极限 | ✅ 发生，FPS 上限 21-22fps |
| NEON 手写不如 OpenCV | 中 | P5 失败，浪费时间 | 先小规模验证，失败及时止损 | ✅ 发生，2 天后关闭 |
| WiFi 连接不稳定 | 高 | NFS 传输中断 | USB 串口作为备用、U 盘传输 | ✅ 发生，延迟 8-818ms |
| V4L2 STREAMOFF 后无法恢复 | 中 | 模式切换后采集失败 | 改用 pause/resume 代替 stop/start | ✅ 发生，问题 #16 |
| 触屏 X 轴偏移 | 低 | 按钮点击错位 | 按钮移到右侧避开偏移区域 | ✅ 发生 |
| M4 开发时间不足 | 中 | 面试时缺少硬件联动经验 | 只做 GPIO+UART 基础，不做 FreeRTOS | 待观察 |

## 六、项目管理方法总结

### 6.1 用了什么方法

| 方法 | 怎么用的 |
|---|---|
| 需求分解（WBS） | 16 个优化项（P0-P16），每个有明确目标和交付物 |
| 优先级排序 | 按依赖关系和收益排序，高收益先做 |
| 迭代开发 | 每个优化项是一个迭代：设计→编码→测试→文档 |
| 变更管理 | 每次调整都记录原因和影响（第四章） |
| 风险管理 | 识别风险→制定应对→跟踪结果（第五章） |
| 经验教训 | 失败的尝试也记录（16_FPS 总跟踪 6.5 节） |
| 知识管理 | 学习资料和项目实践关联（docs/learning/） |

### 6.2 没用什么（以及为什么）

| 没用的 | 为什么 |
|---|---|
| Jira/Trello | 单人项目用 markdown 文档足够，工具反而增加维护成本 |
| 正式的甘特图工具 | 用文本甘特图 + 里程碑表格替代，够用 |
| 代码审查流程 | 单人项目无法自审，用编译验证 + 开发板实测替代 |
| 自动化测试 | 时间有限，用手动测试 + 日志验证替代（已识别为改进点） |

### 6.3 最重要的三个管理实践

1. **及时止损**：P5 NEON 失败后 2 天内关闭，不在沉没成本上纠结
2. **决策留痕**：每个"为什么这样做"和"为什么不那样做"都有记录
3. **范围控制**：知道什么不做比知道做什么更重要——砍掉 AI、患者管理、NBI 等超出范围的需求
