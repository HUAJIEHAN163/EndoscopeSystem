# Overrun 错误原因验证

**目的**：系统地验证 overrun 错误的触发条件和根本原因

**相关问题**：问题记录.md #13

---

## 当前已知信息

✅ **已确认**：
- 驱动实际分配了 4 个 buffer
- 4 个 buffer 理论缓冲能力：4 × 33ms = 132ms
- 程序运行时设备被独占，v4l2-ctl 无法访问

❓ **待验证**：
- 什么情况下会发生 overrun？
- 处理时间和 overrun 的关系？
- 实际缓冲能力是否真的是 132ms？

---

## 验证方案

### 方案 1：测量不同场景下的 overrun 频率

**目的**：找出导致 overrun 的关键因素

**方法**：
1. 在不同处理负载下运行程序
2. 记录 overrun 错误次数和总帧数
3. 计算 overrun 率
4. 对比分析

**测试场景**：
1. 不开任何处理（基准）
2. 只开白平衡
3. 白平衡 + CLAHE
4. 白平衡 + CLAHE + 锐化
5. 白平衡 + CLAHE + 锐化 + 降噪

**执行步骤**：

```bash
# 1. 编译最新代码（包含性能监控）
cd /home/chris/build_arm
make -j$(nproc)
cp /home/chris/build_arm/endoscope /home/chris/nfs_share/

# 2. 部署到开发板
ssh root@192.168.7.2
umount /mnt/nfs
mount -t nfs 192.168.0.130:/home/chris/nfs_share /mnt/nfs -o nolock
cp /mnt/nfs/endoscope /tmp/endoscope
cp /mnt/nfs/scripts/verify_overrun.sh /tmp/
chmod +x /tmp/endoscope
chmod +x /tmp/verify_overrun.sh

# 3. 运行验证脚本
/tmp/verify_overrun.sh
```

**预期输出**：

```
场景                           | Overrun    | 总帧数     | Overrun率   | 平均FPS
--------------------------------------------------------------------------------
不开任何处理                    | 0          | 900        | 0.00%       | 30.0
只开白平衡                      | 5          | 900        | 0.56%       | 28.5
白平衡+CLAHE                   | 50         | 900        | 5.56%       | 25.0
白平衡+CLAHE+锐化              | 150        | 900        | 16.67%      | 20.0
白平衡+CLAHE+锐化+降噪         | 400        | 900        | 44.44%      | 12.0
```

**分析方法**：
- Overrun 率 < 1%：性能可接受
- Overrun 率 1-5%：轻微问题
- Overrun 率 > 5%：严重问题
- 找出 overrun 率突然上升的场景

---

### 方案 2：测量实际处理时间

**目的**：验证处理时间和 overrun 的关系

**方法**：
代码中已添加性能监控，每 100 帧输出一次平均处理时间

**查看日志**：

```bash
# 运行程序并查看性能日志
/tmp/endoscope -platform linuxfb 2>&1 | grep "性能"
```

**预期输出**：

```
[性能] 已处理 100 帧，平均耗时 25.3 ms/帧
[性能] 已处理 200 帧，平均耗时 26.1 ms/帧
[性能] 已处理 300 帧，平均耗时 25.8 ms/帧
...
[性能总结] 总帧数 900，平均耗时 25.9 ms/帧
```

**对比分析**：

| 场景 | 平均处理时间 | 理论最大帧率 | 实际 FPS | Overrun 率 |
|------|-------------|-------------|---------|-----------|
| 不开处理 | ~20ms | 50fps | 30fps | 0% |
| 开白平衡 | ~45ms | 22fps | 28fps | <1% |
| 开多个算法 | ~100ms | 10fps | 12fps | >10% |

**关键发现**：
- 如果处理时间 < 33ms（帧间隔），不应该有 overrun
- 如果处理时间 > 132ms（4 个 buffer 缓冲），必然 overrun
- 如果处理时间在 33-132ms 之间，overrun 率应该逐渐上升

---

### 方案 3：监控 dmesg 实时输出

**目的**：观察 overrun 发生的实时情况

**方法**：

```bash
# 终端 1：清除并监控 dmesg
dmesg -c > /dev/null
dmesg -w | grep dcmi

# 终端 2：运行程序
/tmp/endoscope -platform linuxfb
```

**观察内容**：
- overrun 错误的频率
- errors 和 buffers 的比例
- 是否有其他错误信息

**示例输出**：

```
[  100.123] stm32-dcmi 4c006000.dcmi: Some errors found while streaming: errors=5 (overrun=5), buffers=900
[  200.456] stm32-dcmi 4c006000.dcmi: Some errors found while streaming: errors=15 (overrun=15), buffers=1800
[  300.789] stm32-dcmi 4c006000.dcmi: Some errors found while streaming: errors=50 (overrun=50), buffers=2700
```

**分析**：
- 计算 overrun 率：errors / buffers
- 观察 overrun 是否持续增加
- 判断是偶发还是持续问题
