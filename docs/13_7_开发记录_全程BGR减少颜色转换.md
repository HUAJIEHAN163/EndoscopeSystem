# 13_7 开发记录：全程 BGR 减少颜色转换（P7.1）

## 一、背景

### 问题来源

三线程管线性能基准测试（13_6 第九章）发现，当前每帧存在 4-5 次颜色转换，每次 2-33ms，是仅次于算法本身的第二大耗时来源。

### 当前颜色转换链路

```
采集线程:
  YUYV → BGR (16-33ms)
  BGR → RGB (4-12ms)          ← 冗余转换 ①
  RGB → QImage copy (5-10ms)  ← QImage 深拷贝

处理线程:
  QImage → Mat(RGB→BGR) (2-15ms)  ← 冗余转换 ②
  算法处理（全程 BGR）
  Mat(BGR→RGB) → QImage (1-7ms)   ← 冗余转换 ③

主线程:
  QImage → 显示
```

共 4-5 次颜色/格式转换，其中 ①②③ 是冗余的——数据在 BGR 和 RGB 之间来回转换，只因为 QImage 要求 RGB 格式。

### 核心思路

**全程使用 BGR cv::Mat 传递帧数据，只在最终显示时做一次 BGR→RGB→QImage 转换。**

### 预期收益

| 环节 | 改动前耗时 | 改动后 |
|---|---|---|
| 采集线程 BGR→RGB | 4-12ms | ❌ 省掉 |
| 采集线程 QImage copy | 5-10ms | 改为 Mat clone（更快） |
| 处理线程 QImage→Mat(RGB→BGR) | 2-15ms | ❌ 省掉 |
| 处理线程 Mat→QImage(BGR→RGB) | 1-7ms | ❌ 省掉 |
| 主线程 BGR→RGB→QImage | 无 | ✅ 新增（唯一转换） |
| **预计每帧节省** | **10-20ms** | |

### 附带收益

- 不开算法时，处理线程可直接用 cv::resize（之前因 QImage↔Mat 转换开销反而更慢，见 13_6 第十章）
- 录像时直接用 BGR Mat 写入 VideoWriter，省掉 QImage→Mat 转换
- 拍照时直接用 cv::imwrite 保存 BGR Mat，省掉 QImage 转换

## 二、设计方案

### 2.1 优化后数据流

```
采集线程:
  YUYV → BGR → push cv::Mat 到 captureQueue

处理线程:
  pop cv::Mat(BGR) → 算法处理(BGR) → resize → push cv::Mat(BGR) 到 displayQueue

主线程:
  pop cv::Mat(BGR) → BGR→RGB → QImage → 显示
                   ↑ 整个管线唯一的颜色转换
```

### 2.2 关键设计决策

**FrameQueue 从存 QImage 改为存 cv::Mat**

这是最核心的改动。队列中的数据格式决定了上下游的转换需求：
- 存 QImage(RGB) → 上游必须转 RGB，下游必须转回 BGR → 冗余
- 存 cv::Mat(BGR) → 上游直接 push，下游直接用 → 零冗余

push 时使用 `frame.clone()` 确保深拷贝，与之前 `image.copy()` 语义一致。

### 2.3 改动范围

| 文件 | 改动 |
|---|---|
| `src/utils/framequeue.h` | QImage → cv::Mat |
| `src/capture/videosource.h` | 添加 opencv include |
| `src/capture/v4l2capture.cpp` | 去掉 BGR→RGB，直接 push BGR Mat |
| `src/capture/filecapture.cpp` | 直接 push BGR Mat |
| `src/processing/processthread.h` | 去掉 QImage↔Mat 转换，全程 cv::Mat |
| `src/ui/mainwindow.h` | 新增 m_latestRawMat 成员 |
| `src/ui/mainwindow.cpp` | displayTimer 改为取 Mat 并转换；拍照/录像用 Mat |

## 四、代码修改详细记录

### 4.1 Step 1：FrameQueue 改为存 cv::Mat

修改文件：`src/utils/framequeue.h`

**改动前：**

```cpp
#include <QImage>
#include <atomic>

template<int N>
class FrameQueue {
    QImage m_frames[N];
    // ...
    void push(const QImage &frame) {
        // ...
        m_frames[w] = frame;
        // ...
    }
    bool pop(QImage &frame) { /* ... */ }
    bool latest(QImage &frame) { /* ... */ }
};
```

**改动后：**

```cpp
#include <opencv2/core.hpp>
#include <atomic>

template<int N>
class FrameQueue {
    cv::Mat m_frames[N];
    // ...
    void push(const cv::Mat &frame) {
        // ...
        m_frames[w] = frame.clone();  // 深拷贝，脱离原始内存
        // ...
    }
    bool pop(cv::Mat &frame) { /* ... */ }
    bool latest(cv::Mat &frame) { /* ... */ }
};
```

**关键变化：**
- `QImage` → `cv::Mat`，include 从 `<QImage>` 改为 `<opencv2/core.hpp>`
- push 中 `m_frames[w] = frame` 改为 `m_frames[w] = frame.clone()`
  - QImage 赋值是隐式共享（copy-on-write），自带深拷贝语义
  - cv::Mat 赋值是浅拷贝（共享数据指针），必须显式 clone

## 五、设计决策记录

### 5.1 原设计为什么用 QImage 传递

原始两线程架构中，线程间通过 Qt 信号槽传递数据：

```
采集线程: emit frameReady(QImage) ───→ 主线程: onFrameReady(QImage)
```

Qt 信号槽（QueuedConnection）要求参数是 Qt 可识别的类型。cv::Mat 不是 Qt 类型，不能直接通过信号槽传递（需要 qRegisterMetaType 注册）。所以 QImage 作为线程间传递格式是当时的合理选择。

冗余是在 P3 三线程改造时引入的：P3 把信号槽换成了 FrameQueue，此时队列已经不受 Qt 类型限制，但队列里存的仍然是 QImage。当时的重点是把算法从主线程移出去，没有同时改数据格式，属于分步迭代。

### 5.2 当前改动的风险评估

**风险 1：cv::Mat clone 的内存开销**

QImage 赋值是隐式共享（引用计数，不立即拷贝），cv::Mat clone 是真正的深拷贝。FrameQueue push 时每帧都 clone 一次 640×480×3 ≈ 900KB。两个队列各 4 槽 = 最多 8 帧 ≈ 7.2MB 常驻内存。

在 STM32MP157（512MB DDR）上不是问题，但比之前 QImage 的 copy-on-write 多用一些内存。

**风险 2：主线程新增 BGR→RGB + QImage copy 开销**

之前主线程直接拿 QImage 显示，零转换。现在多了一次 `cvtColor(BGR2RGB)` + `QImage copy`。在显示尺寸（460×360）下预计 3-5ms，在 16ms 定时器周期内完成，不会阻塞 UI。

**结论：两个风险都不大，上板实测时关注即可。**

## 六、架构对比

### 改动前（QImage 传递）

```
采集线程              处理线程                    主线程
├── YUYV→BGR          ├── pop QImage              ├── displayTimer
├── BGR→RGB ①         ├── QImage→Mat(RGB→BGR) ②   │   ├── latest QImage
├── QImage copy       ├── 算法(BGR)               │   ├── 录像: QImage→Mat ④
├── push QImage ──→   ├── Mat→QImage(BGR→RGB) ③   │   ├── 显示
└── 循环              ├── push QImage ──────────→  └── 循环
                      └── 循环

颜色转换: ①BGR→RGB ②RGB→BGR ③BGR→RGB ④RGB→BGR = 4次
```

### 改动后（cv::Mat BGR 传递）

```
采集线程              处理线程                    主线程
├── YUYV→BGR          ├── pop Mat(BGR)            ├── displayTimer
├── push Mat(BGR) ──→ ├── 算法(BGR)               │   ├── latest Mat(BGR)
└── 循环              ├── resize                  │   ├── BGR→RGB→QImage ①
                      ├── push Mat(BGR) ────────→ │   ├── 录像: 直接用 BGR Mat
                      └── 循环                    │   ├── 显示
                                                  └── 循环

颜色转换: ①BGR→RGB = 1次
```

### 关键变化

| 项目 | 改动前 | 改动后 |
|---|---|---|
| 队列数据格式 | QImage (RGB888) | cv::Mat (BGR) |
| 颜色转换次数 | 4 次/帧 | 1 次/帧 |
| 采集线程 | YUYV→BGR→RGB + QImage copy | YUYV→BGR + Mat clone |
| 处理线程 | QImage→Mat→算法→Mat→QImage | Mat→算法→resize→Mat |
| 主线程显示 | 直接用 QImage | BGR→RGB→QImage |
| 录像 | QImage→Mat→resize→write | Mat→resize→write |
| 拍照 | QImage::save | cv::imwrite |

## 九、与 13_6 第十章的关联

13_6 第十章记录了"scaled 改 cv::resize 尝试（已撤回）"：

> 处理线程不开算法时，将 QImage::scaled 改为 cv::resize，反而更慢（34-87ms vs 25-55ms），因为多了 QImage↔Mat 转换开销。

P7.1 全程 BGR 后，处理线程不开算法时直接用 `cv::resize` 操作 BGR Mat，无任何格式转换开销。这是 P7.1 的附带收益之一。

预期不开算法时处理线程耗时从 25-55ms（QImage::scaled）降到 5-10ms（cv::resize 无转换），FPS 从 ~20 提升到接近采集帧率上限 ~27fps。
