## 3.3 关键设计决策

### 决策 1: VideoSource 抽象接口

**问题**: 虚拟机上没有摄像头，如何开发和测试？

**方案**: 定义 VideoSource 抽象基类，V4l2Capture 和 FileCapture 都继承它。
程序启动时自动检测: 有 `/dev/video0` 就用 V4L2，否则读配置文件中的测试视频。

```cpp
VideoSource *source;
// 自动选择:
if (设备存在) source = new V4l2Capture("/dev/video0");
else          source = new FileCapture("test_data/test.mp4");
```

**好处**: 虚拟机上开发的所有代码 (UI、算法、拍照录像) 不需要任何修改就能在开发板上运行。

### 决策 2: 去掉 C/S 网络架构

**问题**: 原 VideoMonitor 是 Server 采集 + TCP 传输 + Client 显示，内窥镜不需要网络。

**方案**: 合并为单机程序。采集、处理、显示全部在同一台设备上完成。

**好处**:
- 消除网络延迟 (原方案有 JPEG 压缩→传输→解压的开销)
- 代码更简单，减少出错点
- 符合内窥镜实际使用场景

### 决策 3: std::atomic 替代 volatile bool

**问题**: 原 VideoMonitor 用 `volatile bool isRunning` 控制线程，这在 C++ 中不保证线程安全。

**方案**: 使用 `std::atomic<bool> m_running`。

### 决策 4: OpenCV cvtColor 替代手写颜色转换

**问题**: 原 VideoMonitor 手写了 YUYV→RGB 转换函数，代码冗长且无优化。

**方案**: 使用 `cv::cvtColor(yuyv, bgr, cv::COLOR_YUV2BGR_YUYV)`。

**好处**: OpenCV 内部有 ARM NEON 优化，比手写快；代码只需一行。

### 决策 5: 配置文件驱动

**问题**: 原项目硬编码了设备路径、分辨率、端口号等参数。

**方案**: 所有硬件相关参数放在 `config/endoscope.conf` 中，用 QSettings 读取。
虚拟机和开发板使用不同的配置文件即可，代码不需要改。

## 3.4 线程模型

| 线程 | 职责 | 同步方式 |
|------|------|---------|
| GUI 线程 | Qt 界面渲染、用户交互、图像处理 | 主事件循环 |
| 采集线程 | V4L2 mmap 采集 / 视频文件读取 | QThread, 通过 frameReady 信号传递帧 |

帧数据通过 Qt 信号槽 (`Qt::QueuedConnection`) 从采集线程传递到 GUI 线程，
Qt 自动处理跨线程排队，无需手动加锁。

> **后续优化**: 如果图像处理算法耗时导致 GUI 卡顿，可以增加独立的处理线程，
> 形成 采集→处理→显示 三级流水线。当前 demo 阶段暂不需要。

---

## 3.5 数据流

```
采集线程:
  V4L2 DQBUF → mmap 缓冲区 (YUYV) → cvtColor → QImage (RGB888) → emit frameReady

GUI 线程:
  onFrameReady → QImage→Mat → ImageProcessor::process → Mat→QImage
  → 旋转 → 录像写入 → 缩放 → update() → paintEvent 绘制
```
