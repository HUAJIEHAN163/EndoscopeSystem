# 项目开发流程总结

## 一、本项目的开发阶段

```
阶段 1：调研收集
  ↓ 收集开源程序、分析技术方案、确定技术栈
阶段 2：需求设计
  ↓ 讨论软件功能、确定硬件设备、界面原型
阶段 3：创建 Demo
  ↓ 最小可用版本 + 学习计划 + 学习文档
阶段 4：迭代开发
  ↓ 功能开发 → 测试验证 → 文档记录 → 持续解决和添加
阶段 5：持续优化
  ↓ 性能优化、算法改进、架构重构
阶段 6：文档沉淀
    设计文档、问题记录、学习笔记、部署手册
```

### 1.1 调研收集

- 收集 V4L2 摄像头采集的开源项目
- 收集 Qt + OpenCV 图像处理的开源项目
- 确定技术栈：Qt5 + OpenCV + V4L2，运行于 STM32MP157 嵌入式 Linux

### 1.2 需求设计

- 软件功能：实时采集、图像处理（CLAHE/去雾/锐化/降噪）、拍照录像、冻结画面、模式切换
- 硬件设备：STM32MP157 开发板 + OV5640 摄像头 + 800×480 触摸屏
- 后续硬件：LED 冷光源模块 + MOSFET 驱动 + SHT30 温度传感器

### 1.3 创建 Demo

- 实现最小可用版本：V4L2 采集 → OpenCV 处理 → Qt 显示
- 创建学习计划（`docs/learning/00_学习规划与跟踪.md`）
- 按模块创建学习文档（V4L2 采集流程、C++ 语法等）

### 1.4 迭代开发

在 Demo 基础上持续添加功能：

| 迭代 | 内容 | 文档 |
|---|---|---|
| P0 | 调试/临床模式 + 预设管理 | 13_1_开发记录 |
| 冻结优化 | 冻结帧清晰度选择 | 13_2_开发记录 |
| 硬件参数 | 硬件参数区 + 白平衡卸载 | 13_3_开发记录 |
| 架构分离 | 实时处理与离线处理分离 | 13_4_开发记录 |

每次迭代包含：开发文档 → 代码实现 → 测试验证 → 问题记录

### 1.5 持续优化

后续优化计划记录在 `docs/10_后续优化计划.md`，按优先级排列：
- P3 三线程管线（性能）

## 二、待完善的环节

### 2.1 测试验证（当前缺失）

目前所有测试都是手动的（在开发板上看效果、看 FPS），没有自动化测试。

**应该有的测试层次：**

| 层次 | 内容 | 当前状态 |
|---|---|---|
| 单元测试 | 每个算法函数的输入输出验证 | ❌ 没有 |
| 集成测试 | 多个模块组合后的功能验证 | ❌ 没有 |
| 回归测试 | 改了代码后确认之前的功能没坏 | ❌ 手动验证 |
| 性能基准测试 | 每次优化后自动跑耗时对比 | ⚠️ 有手动计时，无自动化 |

**后续建议：**

- 为 `ImageProcessor` 的每个算法写单元测试（输入固定图片，验证输出不崩溃、清晰度分数在预期范围）
- 用脚本自动跑性能基准，输出各算法耗时表格，每次优化后对比
- 不需要追求 100% 覆盖率，优先覆盖核心算法和容易出 bug 的地方

### 2.2 版本管理策略（当前简单）

目前所有改动都在 main 分支上直接提交。对于单人开发且功能之间有依赖的项目，这样做是合理的——用分支反而增加复杂度。

**当前建议（单人开发）：**

继续在 main 上开发，但养成两个习惯：

1. 每个功能完成后打 tag，相当于存档点：
```bash
git tag v1.0-demo
git tag v1.1-edit-mode
git tag v1.2-hw-params
# 出问题时可以 git diff v1.1-edit-mode 看改了什么
# 或者 git checkout v1.1-edit-mode 回到那个版本
```

2. 大改动前先 commit，确保当前状态干净，改坏了可以 `git checkout -- .` 恢复

**后续建议（多人协作时）：**

```
main          ← 稳定版本，随时可部署
  ↑ 合并
feature/xxx   ← 功能开发分支
bugfix/xxx    ← 问题修复分支
```

### 2.3 需求追溯（当前已基本满足）

当前的文档结构已经形成了追溯链：

```
需求（10_后续优化计划.md）→ 设计（13_x_开发记录）→ 代码改动（文档中记录）→ 问题（问题记录.md）
```

单人开发项目这样已经足够。多人协作时才需要更正式的需求管理工具（如 Jira、GitHub Issues）。

## 三、完整的开发流程模板

后续每个功能迭代建议按这个流程执行：

```
1. 需求明确
   - 要做什么？为什么做？
   - 记录到 10_后续优化计划.md

2. 方案设计
   - 怎么做？改哪些文件？
   - 创建 13_x_开发记录_xxx.md

3. 创建分支（后续执行）
   - git checkout -b feature/xxx

4. 代码实现
   - 按设计文档逐步实现
   - 每步编译验证

5. 测试验证
   - 虚拟机本地测试
   - 开发板实测
   - 记录测试结果到开发文档

6. 问题修复
   - 遇到 bug 记录到问题记录.md
   - 包含现象、原因、修复方案、代码对比

7. 代码提交
   - git commit 描述清楚改动内容
   - 合并到 main（后续用分支时）

8. 文档更新
   - 更新开发记录的状态
   - 更新学习计划的完成状态
   - 更新优化计划的进度
```

## 四、本项目的经验教训

### 做得好的

1. **文档先行**：每个功能先写设计文档再写代码，思路清晰
2. **问题记录详细**：每个 bug 都记录了现象、原因、修复过程和代码对比，可复现可追溯
3. **学习与实践结合**：遇到不懂的语法/概念当场学，学完立刻在项目中应用
4. **渐进式优化**：先做能用的版本，再逐步优化性能和架构

### 需要改进的

1. **缺少自动化测试**：每次改动后只能手动验证，容易遗漏
2. **没有用分支开发**：所有改动直接提交 main，出问题难以回退
3. **部分改动缺少验证**：有些代码改动没有在开发板上验证就提交了
4. **触屏问题排查耗时**：拍照/录像按钮互换的问题花了很多时间，应该更早加调试日志定位
