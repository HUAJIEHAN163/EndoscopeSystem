## 分析方法

### 计算 Overrun 率

从 dmesg 输出中提取：
```
stm32-dcmi 4c006000.dcmi: errors=50 (overrun=50), buffers=2700
```

计算：
- Overrun 率 = errors / buffers = 50 / 2700 = 1.85%

### 判断标准

- Overrun 率 < 1%：性能可接受
- Overrun 率 1-5%：轻微问题
- Overrun 率 > 5%：严重问题，需要优化

## 当前结论

### 测试 1（不开处理）：

✅ **采集线程性能良好**
- 处理时间 37.5ms < 132ms（4 个 buffer 缓冲能力）
- 没有 overrun 错误

### 测试 3（开启 CLAHE）：

❌ **发现严重问题**
- 采集线程耗时增加到 48.4ms（+10.9ms）
- Overrun 率：15.06%（669/4441）
- 平均每秒 overrun 4.5 次

### 关键发现

**Overrun 的真正原因**：

不是 buffer 数量不足，而是**系统整体负载过高导致的线程调度问题**。

**机制分析**：

1. 主线程处理 CLAHE 很慢（~28ms）+ QImage↔Mat 转换（~16ms）= ~44ms
2. 主线程处理不过来，信号队列堆积
3. 采集线程被 `m_pendingFrames >= 2` 限制，频繁跳帧
4. 采集线程跳帧时仍执行 `grabFrame` + `cvtColor`，占用 CPU
5. 系统整体负载高，采集线程响应变慢（37.5ms → 48.4ms）
6. 采集线程来不及归还 buffer → 4 个 buffer 全部被占用 → overrun!

**关键数据**：

- 4 个 buffer 缓冲能力：4 × 33ms = 132ms
- 当前处理时间：48.4ms（采集）+ 44ms（主线程）= 92.4ms
- 理论上不应该 overrun，但实际 overrun 率 15%

**说明**：不是简单的处理时间问题，而是线程调度和队列堆积导致的系统性问题。

### 下一步优化方向

**短期（立即可做）**：

1. ✅ 启用 OV5640 硬件白平衡（省掉软件白平衡的 9ms）
2. 增加 `m_pendingFrames` 限制（从 2 改为 4，允许更多帧在队列中，但可能增加延迟）

**中期（需要重构）**：

1. 多线程流水线：采集线程 → 处理线程 → 主线程（只显示），解耦采集和处理
2. NEON 加速：加速 YUYV→RGB 转换和 CLAHE 等算法

**精准定位方法**：

1. 测试不同 `m_pendingFrames` 限制值（2/4/8）对 overrun 率的影响
2. 测试关闭主线程处理（只采集不处理）时的 overrun 率
3. 添加采集线程的 CPU 占用率监控
4. 测试不同算法组合的 overrun 率（白平衡、CLAHE、去雾等）

下一步：继续测试更多算法组合，观察 overrun 率变化规律
