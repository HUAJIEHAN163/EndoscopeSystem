# M4 开发设计文档

## 一、项目目标

在 STM32MP157 的 M4 核心上实现内窥镜外设控制，通过 UART 与 A7 侧 Qt 程序通信：

1. **LED 补光灯 PWM 调光**：M4 通过 TIM2_CH3 输出 PWM → PB10 直驱 HW-269 LED（已验证不需要 MOSFET）
2. **温度监控**：M4 通过 I2C 读取 SHT30 温湿度，上报给 A7 显示
3. **按键控制**：M4 检测 KEY1/KEY2 按下，通过 UART 发送指令给 A7（冻结/拍照）
4. **A7-M4 通信**：A7 发送亮度指令 → M4 调节 PWM；M4 发送温度/按键事件 → A7 显示/执行

                      开发板
                 ┌──────────────┐

                 │   J1 扩展口   │

                 │  14 SDA  ────┼──→ SHT30 SDA
                 │  19 3V3  ────┼──→ SHT30 VCC
                 │   2 GND  ────┼──→ SHT30 GND
                 │              │
                 │   5 UART1_TX ┼──→ M4 UART RX（A7→M4 指令）
                 │   6 UART1_RX ┼──← M4 UART TX（M4→A7 数据）
                 │              │
                 │   KEY 接口    │
                 │   KEY1 PB13 ─┼──  按键1（冻结/恢复）

                └──────────┘

> **设计变更**：原方案 PB10 → N-MOSFET → HW-269，经实际验证 PB10 GPIO 可直接驱动 HW-269 data 引脚（高电平点亮），无需 MOSFET 中转。接线从 5 根减少到 3 根。

## 三、通信方案设计

> **设计变更**：原方案为物理 UART + DMA 接收，实际开发中改用 **OpenAMP RPMsg 虚拟串口**。优势：不占用物理引脚、不需 DMA 配置、A7 侧用法几乎一样（`/dev/ttyRPMSG0`）。协议层完全不变。详见 `13_11_M4_RPMsg虚拟串口通信.md`。

### 3.1 A7 → M4（指令）

| 指令 | 格式 | 含义 | 示例 |
|---|---|---|---|
| 设置 LED 亮度 | `L<value>\n` | value: 0-100（占空比百分比） | `L75\n` = 75% 亮度 |
| LED 开关 | `L0\n` / `L100\n` | 0=灭，100=最亮 | `L0\n` = 关灯 |

### 3.2 M4 → A7（上报）

| 数据 | 格式 | 含义 | 示例 |
|---|---|---|---|
| 温度上报 | `T<temp>\n` | temp: 温度值（×10，整数） | `T256\n` = 25.6°C |
| 湿度上报 | `H<humi>\n` | humi: 湿度值（×10，整数） | `H654\n` = 65.4% |
| 按键事件 | `K<key>\n` | key: 1=冻结/恢复，2=拍照 | `K1\n` = 冻结按键按下 |

### 3.3 协议说明

- 所有消息以换行符 `\n` 结尾，方便逐行解析
- 第一个字符是消息类型（L/T/H/K），后面是数值
- 纯 ASCII 文本，便于调试（串口终端直接可读）
- 温度/湿度乘以 10 用整数传输，避免浮点解析

## 四、M4 侧软件设计

### 4.1 功能模块

```
M4 main()
  ├── HAL_Init() + SystemClock_Config()
  ├── GPIO_Init()        → KEY1(PB13), KEY2(PH7) 配置为中断输入
  ├── TIM2_Init()        → CH3(PB10) PWM 输出，频率 1kHz
  ├── I2C1_Init()        → 400kHz，连接 SHT30
  ├── UART_Init()        → 与 A7 通信，115200 8N1，DMA 接收 + 空闲中断
  ├── printf 重定向      → printf 输出到 UART（调试用）
  │
  └── while(1) 主循环
        ├── 每 1 秒：I2C 读取 SHT30 → UART 发送温度/湿度
        ├── 温度超限保护：>41°C 自动降低 PWM 到 20% + 上报报警
        ├── 解析 DMA 接收缓冲区（空闲中断触发）→ 调节 PWM 占空比
        └── （按键通过中断处理，不在主循环中）

中断服务：
  HAL_GPIO_EXTI_Callback(GPIO_Pin)
    ├── KEY1 按下 → 消抖 20ms → UART 发送 "K1\n"
    └── KEY2 按下 → 消抖 20ms → UART 发送 "K2\n"

  HAL_UARTEx_RxEventCallback(huart, Size)   ← DMA 空闲中断
    └── 解析接收缓冲区，提取指令
```

### 4.2 UART DMA 接收 + 空闲中断

普通 UART 接收是每收到 1 个字节触发一次中断，收 "L75\n" 需要 4 次中断。DMA + 空闲中断方式：DMA 自动把数据收到缓冲区，一帧数据全部收完后（总线空闲）才触发一次中断：

```c
// 初始化：启动 DMA 接收（只调用一次）
uint8_t rx_buf[32];
HAL_UARTEx_ReceiveToIdle_DMA(&huart1, rx_buf, sizeof(rx_buf));

// 空闲中断回调（一帧数据全部收完时触发）
void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
    // rx_buf[0..Size-1] 就是收到的完整数据，如 "L75\n"
    parse_command(rx_buf, Size);
    // 重新启动 DMA 接收
    HAL_UARTEx_ReceiveToIdle_DMA(&huart1, rx_buf, sizeof(rx_buf));
}
```

优势：
- 不需要在主循环中轮询 UART
- 不会因为主循环忙而丢失数据
- CPU 不参与数据搬运（DMA 硬件完成）

### 4.3 printf 重定向

M4 没有屏幕，printf 默认无输出。重定向到 UART 后可以在 A7 侧串口终端看到 M4 的调试信息：

```c
// 重定向 fputc（所有 printf 最终调用这个函数输出每个字符）
int fputc(int ch, FILE *f) {
    HAL_UART_Transmit(&huart1, (uint8_t*)&ch, 1, 10);
    return ch;
}

// 使用
printf("[M4] 温度: %d.%d C\r\n", temp/10, temp%10);
// A7 侧 cat /dev/ttySTMx 就能看到这行输出
```

### 4.4 温度超限保护逻辑

M4 本地判断温度，不依赖 A7 指令（安全关键功能不能依赖通信链路）：

```c
#define TEMP_WARNING  410   // 41.0°C ×10
#define TEMP_CRITICAL 430   // 43.0°C ×10
#define PWM_SAFE_DUTY 20    // 安全模式下的占空比 20%

void check_temperature(int16_t temp) {
    if (temp > TEMP_CRITICAL) {
        // 紧急关灯
        set_pwm_duty(0);
        send_to_a7("A2\n");    // A=Alarm, 2=严重
        printf("[M4] 紧急关灯! 温度:%d.%d\r\n", temp/10, temp%10);
    } else if (temp > TEMP_WARNING) {
        // 降低亮度
        set_pwm_duty(PWM_SAFE_DUTY);
        send_to_a7("A1\n");    // A=Alarm, 1=警告
        printf("[M4] 温度警告! 降低亮度\r\n");
    }
}
```

为什么保护逻辑在 M4 而不在 A7：
- A7 的 Linux 可能卡顿/崩溃，M4 裸机始终在跑
- UART 通信可能中断，M4 本地判断不依赖通信
- 医疗设备安全功能必须独立于主控系统

### 4.5 FreeRTOS 多任务架构

用 FreeRTOS 把裸机 while(1) 拆分为独立任务，每个任务负责一个功能：

```c
// 任务 1：温度监控（低优先级，每秒执行一次）
void Task_Temperature(void *arg) {
    while(1) {
        int16_t temp = read_sht30_temperature();
        check_temperature(temp);           // 超限保护
        send_temperature_to_a7(temp);      // 上报
        vTaskDelay(pdMS_TO_TICKS(1000));   // 睡眠 1 秒
    }
}

// 任务 2：指令处理（中优先级，等待信号量）
void Task_Command(void *arg) {
    while(1) {
        // 等待 DMA 空闲中断释放信号量（收到数据时才唤醒）
        xSemaphoreTake(xUartRxSemaphore, portMAX_DELAY);
        parse_command(rx_buf, rx_size);
    }
}

// 任务 3：LED 状态指示（最低优先级，空闲时闪烁）
void Task_Heartbeat(void *arg) {
    while(1) {
        HAL_GPIO_TogglePin(LED_GPIO_Port, LED_Pin);
        vTaskDelay(pdMS_TO_TICKS(500));    // 500ms 闪烁
    }
}

// DMA 空闲中断中释放信号量
void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
    rx_size = Size;
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xSemaphoreGiveFromISR(xUartRxSemaphore, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
    HAL_UARTEx_ReceiveToIdle_DMA(&huart1, rx_buf, sizeof(rx_buf));
}

// main 中创建任务并启动调度器
int main(void) {
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_TIM2_Init();
    MX_I2C1_Init();
    MX_USART1_UART_Init();

    xUartRxSemaphore = xSemaphoreCreateBinary();

    xTaskCreate(Task_Temperature, "Temp", 256, NULL, 1, NULL);
    xTaskCreate(Task_Command,     "Cmd",  256, NULL, 2, NULL);
    xTaskCreate(Task_Heartbeat,   "LED",  128, NULL, 0, NULL);

    vTaskStartScheduler();  // 启动调度器，不会返回
    while(1) {}             // 不会执行到这里
}
```

任务优先级设计：

| 任务 | 优先级 | 原因 |
|---|---|---|
| Task_Command | 2（最高） | A7 指令需要快速响应 |
| Task_Temperature | 1 | 每秒执行一次，不紧急 |
| Task_Heartbeat | 0（最低） | 纯指示用，空闲时才跑 |

同步机制：
- 信号量 `xUartRxSemaphore`：DMA 中断中释放，Task_Command 中等待
- 按键中断直接发送 UART（不经过任务，因为只发 4 字节，很快）

FreeRTOS vs uCOS 对照（面试用）：

| 概念 | FreeRTOS API | uCOS API | 功能 |
|---|---|---|---|
| 创建任务 | xTaskCreate() | OSTaskCreate() | 创建一个线程 |
| 延时 | vTaskDelay() | OSTimeDly() | 任务睡眠指定时间 |
| 信号量释放 | xSemaphoreGive() | OSSemPost() | 通知等待的任务 |
| 信号量等待 | xSemaphoreTake() | OSSemPend() | 阻塞等待通知 |
| 消息队列发送 | xQueueSend() | OSQPost() | 发送数据到队列 |
| 消息队列接收 | xQueueReceive() | OSQPend() | 从队列接收数据 |
| 互斥锁 | xSemaphoreCreateMutex() | OSMutexCreate() | 保护共享资源 |

核心概念完全一样，只是 API 名字不同。会 FreeRTOS 就会 uCOS。

## 八、注意事项

1. **PB10 复用冲突**：PB10 默认可能被 UART3 占用，需要在设备树中将其复用为 TIM2_CH3。使用野火提供的 PWM overlay（`stm-fire-pwm-sub-system-overlay.dts`）
2. **I2C1 可能被 A7 占用**：需要确认 I2C1 是否已分配给 M4。如果被 A7 占用，需要在设备树中调整
3. **⭐ A7-M4 通信使用 RPMsg 而非物理 UART**：实际开发中确认 RPMsg 虚拟串口方案更优（不占引脚、不需 DMA），A7 侧设备为 `/dev/ttyRPMSG0`
4. **按键消抖**：机械按键会产生抖动（按一次产生多次中断），需要软件消抖（200ms）
5. **温度报警阈值**：SHT30 读到温度 > 41°C 时，M4 应自动降低 PWM 占空比（安全保护，不依赖 A7 指令）
6. **M4 固件加载**：A7 的 Linux 通过 remoteproc 加载 M4 固件（`/lib/firmware/xxx.elf`）
7. **⭐ STM32MP1 双核 EXTI 限制**：M4 无法直接使用 EXTI GPIO 中断。EXTI 控制器由 A7 侧 Linux 内核管理（通过 hwlock 仲裁），M4 通过 HAL 库配置的 EXTI 寄存器不能真正生效。解决方案：使用 TIM 定时器中断做周期性 GPIO 轮询（详见问题记录 #19）
8. **⭐ 按键电平逻辑**：板载 KEY1/KEY2 实际为**高电平有效**（不按=低，按下=高），与常见的低电平有效相反。GPIO 需配置为下拉+上升沿检测
9. **⭐ A7 设备树抢占 GPIO**：`/boot/uEnv.txt` 中的 `stm-fire-key.dtbo` 会注册 PB13/PH7 为 A7 侧 gpio-keys，必须注释掉才能让 M4 读取这些 GPIO
10. **⭐ GCC 13 链接器 segment alignment 问题**：新版 GCC 默认 `max-page-size=0x1000`，导致野火开发板的 Linux remoteproc 驱动无法识别 ELF 中的 resource table。必须添加 `-Wl,-z,max-page-size=0x10000`（详见问题记录 #20）
11. **⭐ RPMsg service name 必须与内核驱动匹配**：野火开发板的 `rpmsg_tty` 驱动匹配的是 `"rpmsg-tty-channel"`，不是 CubeMX 生成的默认值 `"rpmsg-tty"`（详见问题记录 #20）

### 9.1 PB10 直驱 HW-269 验证

**日期**：2025 年

**目的**：验证 PB10 GPIO 能否直接驱动 HW-269 LED 模块，省去 N-MOSFET

**接线**：

| 开发板引脚 | HW-269 引脚 |
|---|---|
| J1 pin 7 (PB10) | data |
| J1 pin 23 (5V) | VCC |
| J1 pin 1 (GND) | GND |

**测试步骤与结果**：

| 测试 | 命令 | 结果 |
|---|---|---|
| GPIO 导出 | `echo 26 > /sys/class/gpio/export` | ✅ 成功 |
| 设为输出 | `echo out > /sys/class/gpio/gpio26/direction` | ✅ 成功 |
| 高电平点亮 | `echo 1 > /sys/class/gpio/gpio26/value` | ✅ LED 亮 |
| 低电平熄灭 | `echo 0 > /sys/class/gpio/gpio26/value` | ✅ LED 灭 |
| shell 快速切换 | `while true; do echo 1/0 > ...; done` | ✅ 常亮（频率够高） |
| 低占空比调光 | sleep 0.001 亮 / sleep 0.009 灭 | ✅ 可见闪烁，亮度降低 |
| 高占空比调光 | sleep 0.009 亮 / sleep 0.001 灭 | ✅ 明显更亮 |

**结论**：
- PB10 (3.3V GPIO) 可直接驱动 HW-269 data 引脚，**不需要 N-MOSFET**
- PWM 调光方案可行，shell sleep 精度低导致闪烁，硬件 TIM2 PWM（kHz 级）将实现平滑调光
- 5V 引脚上电即输出，LED 亮灭完全由 data 引脚（PB10）的 PWM 信号控制

## 附录 A：命令行控制台 GPIO 控制方法

在 A7 侧 Linux 命令行中，可以通过 sysfs 接口直接控制 GPIO，无需编写任何代码。适用于硬件验证和快速调试。

### A.1 GPIO 编号计算

STM32MP157 的 GPIO 编号 = 组号 × 16 + 引脚号：

| GPIO | 组号 | 引脚号 | 编号 | 计算 |
|---|---|---|---|---|
| PA0 | A=0 | 0 | 0 | 0×16+0 |
| PB10 | B=1 | 10 | 26 | 1×16+10 |
| PB13 | B=1 | 13 | 29 | 1×16+13 |
| PH7 | H=7 | 7 | 119 | 7×16+7 |

### A.2 基本 GPIO 控制

```bash
# 1. 检查引脚是否被占用
cat /sys/kernel/debug/pinctrl/soc:pin-controller@50002000/pinmux-pins | grep PB10

# 2. 导出 GPIO（使其可通过 sysfs 控制）
echo 26 > /sys/class/gpio/export

# 3. 设置方向为输出
echo out > /sys/class/gpio/gpio26/direction

# 4. 控制电平
echo 1 > /sys/class/gpio/gpio26/value   # 高电平（LED 亮）
echo 0 > /sys/class/gpio/gpio26/value   # 低电平（LED 灭）

# 5. 读取当前电平
cat /sys/class/gpio/gpio26/value

# 6. 用完后释放 GPIO
echo 26 > /sys/class/gpio/unexport
```

### A.3 Shell 模拟 PWM 调光

通过调节 sleep 时间的比例模拟不同占空比：

```bash
# 低占空比（约 10%）→ LED 较暗
while true; do
  echo 1 > /sys/class/gpio/gpio26/value
  sleep 0.001
  echo 0 > /sys/class/gpio/gpio26/value
  sleep 0.009
done
# Ctrl+C 停止

# 高占空比（约 90%）→ LED 较亮
while true; do
  echo 1 > /sys/class/gpio/gpio26/value
  sleep 0.009
  echo 0 > /sys/class/gpio/gpio26/value
  sleep 0.001
done
# Ctrl+C 停止
```

> **注意**：shell sleep 精度约 ms 级，频率很低（~100Hz），人眼可见闪烁。仅用于验证调光可行性，实际项目使用 M4 硬件 TIM2 PWM（kHz 级，无闪烁）。

### A.4 GPIO 输入读取（按键）

```bash
# 导出 KEY1 (PB13 = GPIO29)
echo 29 > /sys/class/gpio/export
echo in > /sys/class/gpio/gpio29/direction

# 读取按键状态（低电平有效：0=按下，1=松开）
cat /sys/class/gpio/gpio29/value

# 持续监控按键（每 0.5 秒读一次）
watch -n 0.5 cat /sys/class/gpio/gpio29/value
```

### A.5 常用调试命令

```bash
# 查看所有已导出的 GPIO
ls /sys/class/gpio/

# 查看所有引脚复用状态
cat /sys/kernel/debug/pinctrl/soc:pin-controller@50002000/pinmux-pins

# 查看特定 GPIO 的信息
cat /sys/kernel/debug/gpio
```
