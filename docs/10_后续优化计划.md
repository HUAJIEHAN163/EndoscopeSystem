# 10 后续优化计划

项目基础功能完成后的进阶优化方向，目标：达到产品级嵌入式内窥镜系统的功能与性能要求。

---

## 一、调试模式 / 临床模式切换 + 预设参数管理（P0）

当前界面是开发调参模式，实际产品中医生不会手动调参。需要区分两种工作模式。

### 1.1 调试模式（工程师用）

即当前界面，保留所有参数滑块和复选框，用于：
- 算法调优：用测试视频反复调参
- 参数导出：调好的参数保存为预设文件

### 1.2 临床模式（医生用）

医生只需选择检查部位，一键切换预设参数：

```
┌─────────────────────────────────────────┐
│  [胃镜模式]  [肠镜模式]  [支气管镜模式]    │
│                                         │
│           实时视频画面                    │
│                                         │
│  [拍照]  [录像]  [冻结]     FPS: 30.0   │
└─────────────────────────────────────────┘
```

### 1.3 预设参数管理

用 JSON 或 INI 文件存储预设：

```json
// config/presets/gastroscope.json
{
    "name": "胃镜模式",
    "whiteBalance": true,
    "clahe": true,
    "claheClipLimit": 3.0,
    "dehaze": true,
    "dehazeOmega": 0.85,
    "sharpen": true,
    "sharpenAmount": 1.2,
    "denoise": true
}
```

### 1.4 实施步骤

1. 定义预设文件格式，在 `config/presets/` 下存放各模式参数
2. 新增 `PresetManager` 类，负责加载/保存/切换预设
3. 在 `MainWindow` 中添加模式切换按钮或下拉菜单
4. 调试模式下增加"导出当前参数为预设"功能

### 1.5 进阶：自动参数调整

根据图像特征动态调参（高端产品方向）：
- 根据亮度直方图自动调整 CLAHE 参数
- 根据色温自动调整白平衡
- 暗区自动增强去雾强度

### 1.6 进阶：病灶区域突出显示

行业中称为 CADe（计算机辅助检测）：
- 入门：基于颜色异常检测（红色区域高亮标记），用 OpenCV 即可实现
- 进阶：深度学习模型检测息肉/病变，超出当前项目范围，作为探索方向

**建议时间：3-5 天（基础预设功能）**

## 二、图像传输与存储（P1）

开发板拍摄的图像需要传输到工作电脑查看和归档。

### 2.1 USB 存储（最简单，优先实现）

检测 U 盘挂载，拍照/录像直接保存到 U 盘：

```cpp
// 检测 U 盘挂载点
QDir usbDir("/media/usb");
if (usbDir.exists()) {
    savePath = "/media/usb/endoscope/";
} else {
    savePath = "./captures/";  // 降级到本地
}
```

### 2.2 网络传输（学习价值高）

开发板通过网线连接电脑，提供文件访问服务：

| 方案 | 复杂度 | 说明 |
|------|--------|------|
| FTP 服务 | 低 | 开发板跑 vsftpd，电脑用文件管理器访问 |
| HTTP 服务 | 中 | Qt 内嵌简易 HTTP 服务，浏览器查看图片 |
| RTSP 流 | 高 | 实时视频流推送，电脑用 VLC 查看 |

建议先实现 FTP，再尝试 HTTP。

### 2.3 HDMI 输出

STM32MP157 支持 HDMI 输出，可直接连接外部显示器，无需传输文件。Qt 程序直接在外接屏上显示。

**建议时间：3-5 天**

## 三、开发板按键控制 — M4 裸机开发入口（P2）

通过开发板物理按键控制拍照/冻结/继续，串通 A7-M4 核间通信。

### 3.1 整体架构

```
物理按键 → M4 (GPIO中断检测) → UART/RPMsg → A7 (Qt接收指令) → 执行操作
```

### 3.2 M4 侧（CubeIDE + HAL）

```c
// M4: GPIO 中断回调
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
    switch (GPIO_Pin) {
        case KEY_CAPTURE_Pin:
            send_command_to_a7(CMD_CAPTURE);   // 拍照
            break;
        case KEY_FREEZE_Pin:
            send_command_to_a7(CMD_FREEZE);    // 冻结/继续
            break;
    }
}
```

### 3.3 A7 侧（Qt 接收）

```cpp
// A7: 串口监听线程，收到指令后发信号给主窗口
void SerialListener::onDataReady() {
    char cmd = serial.read(1)[0];
    switch (cmd) {
        case CMD_CAPTURE: emit captureRequested(); break;
        case CMD_FREEZE:  emit freezeRequested(); break;
    }
}
```

### 3.4 学习收获

- M4 侧：GPIO、中断、UART、HAL 库基础
- A7 侧：串口编程、跨核通信
- 系统级：双核协作的完整链路

**建议时间：1-2 周（含 CubeIDE 环境搭建）**

## 四、线程架构优化（P3）

当前采集和处理在同一个 QThread 中串行执行，优化为三级流水线。

```
采集线程 → [环形队列] → 处理线程 → [环形队列] → 渲染线程(主线程)
```

### 4.1 环形 Buffer 队列

```cpp
// src/utils/framebuffer.h
template<int N>
class FrameRingBuffer {
    cv::Mat frames[N];
    std::atomic<int> writeIdx{0};
    std::atomic<int> readIdx{0};
public:
    bool push(const cv::Mat& frame);  // 采集线程写入
    bool pop(cv::Mat& frame);         // 处理线程读取
};
```

- 固定 3-4 个 buffer 槽位
- 无锁设计，采集线程和处理线程各操作一端
- 队列满时丢弃最旧帧，保证实时性

### 4.2 线程优先级设置

```cpp
#include <pthread.h>

struct sched_param param;
param.sched_priority = 80;
pthread_setschedparam(pthread_self(), SCHED_FIFO, &param);
```

**建议时间：3-4 天**

## 六、NEON 指令集加速（P5-P6）

ARM Cortex-A7 支持 NEON SIMD 指令，可对图像处理关键路径进行向量化加速。

### 6.1 优化目标

| 模块 | 当前实现 | 优化方案 | 预期提升 |
|------|---------|---------|---------|
| YUYV → RGB 转换 | OpenCV cvtColor | NEON intrinsic 手写 | 3-5x |
| CLAHE 增强 | OpenCV CLAHE | 直方图统计 NEON 化 | 2-3x |
| 高斯滤波 | OpenCV GaussianBlur | NEON 定点化卷积 | 2-4x |

### 6.2 实施步骤

1. 在 `src/processing/` 下新建 `neon_accel.h/cpp`
2. 先实现 YUYV → RGB 的 NEON 版本（P5），作为入门练习
3. 用 `arm_neon.h` 提供的 intrinsic 函数（vld1q_u8、vmull_u8 等）
4. 编写性能对比测试，验证加速效果
5. 逐步替换其他热点函数（P6）

### 6.3 编译配置


## 八、A7 Linux 用户空间外设通信（P8）

在 Linux 应用层通过设备节点操作 UART/SPI/I2C，不涉及内核驱动开发。

### 8.1 I2C（读写传感器寄存器）

```cpp
#include <linux/i2c-dev.h>
#include <sys/ioctl.h>

int fd = open("/dev/i2c-1", O_RDWR);
ioctl(fd, I2C_SLAVE, 0x3C);  // 设置从机地址
write(fd, data, len);         // 写寄存器
read(fd, buf, len);           // 读寄存器
```

练习方向：挂一个 OLED (SSD1306) 或温湿度传感器 (SHT30)。

### 8.2 SPI（高速外设通信）

```cpp
#include <linux/spi/spidev.h>

int fd = open("/dev/spidev0.0", O_RDWR);
struct spi_ioc_transfer tr = { .tx_buf = tx, .rx_buf = rx, .len = len, .speed_hz = 1000000 };
ioctl(fd, SPI_IOC_MESSAGE(1), &tr);
```

练习方向：驱动一个 SPI 接口的 TFT 小屏或 Flash 芯片。

### 8.3 UART（串口通信）

```cpp
#include <termios.h>

int fd = open("/dev/ttySTM0", O_RDWR | O_NOCTTY);
struct termios tty;
tcgetattr(fd, &tty);
cfsetospeed(&tty, B115200);
tcsetattr(fd, TCSANOW, &tty);
write(fd, "hello", 5);
```

练习方向：A7 与 M4 之间通过 UART 通信，或连接外部 MCU/模块。

**建议时间：3-5 天**

- 按键/脚踏开关的实时响应

### 9.4 LED 补光灯 PWM — 背景与原理

内窥镜光源不是简单的“开灯/关灯”，实际产品采用**频闪照明（strobe illumination）**技术：

```
光源:    ██  　  ██  　  ██  　  ██
         亮  灭  亮  灭  亮  灭  亮
曝光:    [=]     [=]     [=]     [=]
         ↑       ↑       ↑       ↑
       清晰帧  清晰帧  清晰帧  清晰帧
```

**工作原理：**
- 光源只在摄像头曝光瞬间闪亮，持续时间极短（微秒级）
- 相当于每帧都用“闪光灯”拍照，曝光时间极短，运动模糊几乎为零
- 灭灯期间不曝光，所以观看视频时会感觉光在闪动

**为什么需要 M4 做 PWM 控制：**
- **微秒级时序精度**：光源闪烁必须与摄像头曝光精确同步，Linux 的进程调度延迟在毫秒级，无法满足微秒级的实时性要求。曝光时间极短的情况下，可以彻底解决冻结帧的运动模糊问题。
- **硬件定时器**：M4 的 TIM 外设可以输出精确的 PWM 波形，抢占式中断保证响应时间
- **安全红线**：光源常亮会导致探头发热，频闪模式大幅降低平均功耗，减少探头温升（医疗设备对探头温度有严格限制，通常 ≤ 41°C）
- **多模式支持**：可以交替闪烁不同波长的 LED（白光 + 窄带光），实现 NBI（窄带成像）等特殊观察模式

**典型时序示例（胃镜 30fps）：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 帧周期 | 33.3ms | 30fps |
| 光源闪亮时间 | 50-200µs | 曝光窗口 |
| 光源熄灭时间 | ~33ms | 散热期 |
| 占空比 | <1% | 平均功耗极低 |

**建议时间：1-2 周**

## 十一、病灶区域突出显示（P11，探索方向）

行业中称为 CADe（计算机辅助检测），是内窥镜 AI 的热门方向。

- 入门：基于 HSV 颜色空间检测异常红色区域，用半透明遮罩高亮标记
- 进阶：集成轻量深度学习模型（如 MobileNet），检测息肉/病变区域
- 需要标注数据集和模型训练，作为长期探索方向

**建议时间：1-2 周（入门方案）**
