## 三、开发板按键控制 — M4 裸机开发入口（P2）✅ 已完成

通过开发板物理按键控制拍照/冻结/继续，串通 A7-M4 核间通信。

### 3.1 整体架构

```
物理按键 → M4 (TIM3轮询检测) → RPMsg虚拟串口 → A7 (Qt接收指令) → 执行操作
```

### 3.2 M4 侧（CubeIDE + HAL + FreeRTOS）

**原设计方案（GPIO 中断）**：

```c
// M4: GPIO 中断回调
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
    switch (GPIO_Pin) {
        case KEY_CAPTURE_Pin:
            send_command_to_a7(CMD_CAPTURE);   // 拍照
            break;
        case KEY_FREEZE_Pin:
            send_command_to_a7(CMD_FREEZE);    // 冻结/继续
            break;
    }
}
```

**为什么放弃 GPIO 中断？**

STM32MP1 的 EXTI（外部中断控制器）由 A7 侧 Linux 内核管理，通过 hwlock 硬件锁进行双核仲裁。M4 通过 HAL 库配置的 EXTI 寄存器无法真正生效，因为 A7 已经占用了 EXTI 控制权。这是 STM32MP1 双核架构的硬件限制。

**实际实现方案（TIM3 中断轮询）**：

```c
// M4: TIM3 中断回调（10ms 周期轮询按键）
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    if (htim == &htim3) {
        // KEY1 轮询 + 消抖 200ms
        if (HAL_GPIO_ReadPin(KEY1_GPIO_Port, KEY1_Pin) == GPIO_PIN_SET) {
            if (key1_debounce++ > 20) {  // 200ms
                key1_event = 1;
                key1_debounce = 0;
            }
        } else {
            key1_debounce = 0;  // 松开时清零
        }
        // KEY2 同理
    }
}

// Task_Main 中发送 RPMsg
if (key1_event) {
    key1_event = 0;
    rpmsg_send_str("K1\n");  // 冻结/恢复
}
if (key2_event) {
    key2_event = 0;
    rpmsg_send_str("K2\n");  // 拍照
}
```

**轮询方案的优势**：
- 不受 EXTI 双核仲裁限制，M4 完全控制
- 10ms 周期对按键响应足够快（人类感知阈值 ~100ms）
- 软件消抖更灵活（200ms 可调）
- 代码更简单，不需要配置复杂的 EXTI 优先级

### 3.3 A7 侧（Qt 接收）

**原设计方案（物理 UART）**：

```cpp
// A7: 串口监听线程，收到指令后发信号给主窗口
void SerialListener::onDataReady() {
    char cmd = serial.read(1)[0];
    switch (cmd) {
        case CMD_CAPTURE: emit captureRequested(); break;
        case CMD_FREEZE:  emit freezeRequested(); break;
    }
}
```

**为什么改用 RPMsg 虚拟串口？**

物理 UART 需要占用 GPIO 引脚（如 USART1_TX/RX），而开发板的扩展口引脚有限且部分被其他功能占用。RPMsg 虚拟串口通过 IPCC（核间通信控制器）实现，不占用物理引脚，且 A7 侧用法与普通串口几乎一样。

**实际实现方案（RPMsg 虚拟串口）**：

```cpp
// A7: RpmsgListener 线程，收到指令后发信号给主窗口
void RpmsgListener::run() {
    while (!m_stop) {
        QByteArray data = m_serial->readLine();
        if (data.startsWith("K1")) emit freezeRequested();
        if (data.startsWith("K2")) emit captureRequested();
        if (data.startsWith("T")) {  // 温度上报
            int temp = data.mid(1).toInt();
            emit temperatureUpdated(temp / 10.0);
        }
    }
}
```

**RPMsg 方案的优势**：
- 不占用物理引脚，节省硬件资源
- A7 侧设备节点 `/dev/ttyRPMSG0`，用法与普通串口一致
- 通过共享内存传输，效率高于物理 UART
- 野火开发板内核已支持，无需额外驱动

### 3.4 实际实现要点

- **通信方式**：OpenAMP RPMsg 虚拟串口（`/dev/ttyRPMSG0`），不占用物理引脚
- **按键检测**：TIM3 中断轮询（10ms），不用 EXTI（STM32MP1 双核 EXTI 限制）
- **消抖**：软件消抖 200ms，避免误触发
- **多任务**：FreeRTOS 管理 Task_Main（按键+通信）和 Task_Temperature（温度监控）
- **协议**：纯文本协议（K1/K2/T/H/L/A），便于调试

### 3.5 学习收获

- M4 侧：GPIO 轮询、TIM 中断、RPMsg 通信、FreeRTOS 多任务、HAL 库基础
- A7 侧：串口编程（QSerialPort）、跨核通信、信号槽机制
- 系统级：双核协作的完整链路、remoteproc 固件加载机制
- 硬件限制：STM32MP1 双核 EXTI 仲裁、TIM2 与 LCD 背光冲突、I2C1 与触摸屏共享
- 设计权衡：中断 vs 轮询、物理 UART vs RPMsg、硬件资源分配

**实际耗时：约 2 周（含环境搭建、问题排查、协议设计）**

## 四、线程架构优化（P3）

当前采集和处理在同一个 QThread 中串行执行，优化为三级流水线。
目前有两个线程：
采集线程（QThread）              主线程（Qt 事件循环）
├── grabFrame()                 ├── onFrameReady()
├── cvtColor (YUYV→BGR→RGB)     │   ├── processFrame()  ← 算法在这里
├── emit frameReady(image)  ──→ │   ├── update()
└── 循环                        │   └── paintEvent()  ← 显示
                                ├── 按钮点击响应
                                └── 其他 UI 事件

但是算法放错了线程。算法跑在主线程里，和 UI 刷新、按钮响应共用同一个线程，互相阻塞。
主线程在跑 processFrame 的时候，不能做任何其他事——不能刷新画面、不能响应按钮、不能处理下一帧。
采集线程虽然以 30fps 产出帧，但主线程处理不过来，m_pendingFrames >= 2 时采集线程就跳过，帧就丢了。

```
采集线程 → [环形队列] → 处理线程 → [环形队列] → 渲染线程(主线程)

采集线程          处理线程（新增）        主线程
├── grabFrame     ├── processFrame      ├── 显示最新结果
├── cvtColor      ├── CLAHE/去雾        ├── 按钮响应
├── emit ────→    ├── emit ──────→      ├── paintEvent
└── 循环          └── 循环              └── 不再被算法阻塞

这样即使 CLAHE + 去雾要 120ms，主线程仍然可以流畅刷新画面和响应触摸，只是显示的画面会有 120ms 的延迟（显示的是处理线程最近完成的那一帧）。
```

### 4.1 环形 Buffer 队列

```cpp
// src/utils/framebuffer.h
template<int N>
class FrameRingBuffer {
    cv::Mat frames[N];
    std::atomic<int> writeIdx{0};
    std::atomic<int> readIdx{0};
public:
    bool push(const cv::Mat& frame);  // 采集线程写入
    bool pop(cv::Mat& frame);         // 处理线程读取
};
```

- 固定 3-4 个 buffer 槽位
- 无锁设计，采集线程和处理线程各操作一端
- 队列满时丢弃最旧帧，保证实时性

### 4.2 线程优先级设置

```cpp
#include <pthread.h>

struct sched_param param;
param.sched_priority = 80;
pthread_setschedparam(pthread_self(), SCHED_FIFO, &param);
```

- 关联问题：问题 #14：Overrun 的真正机制 — 线程调度问题

## 五、内存池实现（P4）🔻 低优先级

避免每帧动态分配内存，预分配固定数量的 cv::Mat 循环复用。

```cpp
// src/utils/mempool.h
class FrameMemPool {
    std::vector<cv::Mat> pool;
    std::queue<int> freeList;
public:
    void init(int count, int rows, int cols, int type);
    cv::Mat& acquire();   // 获取空闲帧
    void release(int id); // 归还帧
};
```

### 为什么优先级降低？

1. **cv::Mat 已经是引用计数**：
   - cv::Mat 的赋值和拷贝构造只是浅拷贝（共享数据）
   - 只有引用计数归零时才释放内存
   - 已经有一定的内存复用机制

2. **当前架构已经优化**：
   - P16 FrameQueue 零拷贝优化已完成（`std::move` 转移所有权）
   - 三线程管线减少了不必要的内存拷贝
   - 全程 BGR 减少了颜色转换的临时内存

3. **实际测试未发现内存问题**：
   - 开发板运行稳定，无 OOM
   - FPS 瓶颈不在内存分配，而在算法计算

4. **开发成本 vs 收益**：
   - 需要设计复杂的内存管理逻辑（获取/释放/线程安全）
   - 预期性能提升有限（<5%）
   - 代码复杂度增加，维护成本高

### 什么情况下值得做？

- 如果后续增加高帧率支持（60fps+）
- 如果内存分配成为性能瓶颈（通过 profiling 验证）
- 如果需要严格的实时性保证（避免内存分配的不确定性）

**建议时间：2 天（但当前不建议执行）**

## 六、NEON 指令集加速（P5-P6）❌ 已取消

ARM Cortex-A7 支持 NEON SIMD 指令，可对图像处理关键路径进行向量化加速。

### 6.1 为什么取消？

**P5（YUYV→BGR NEON 加速）已关闭**：

原计划手写 NEON intrinsic 代码加速 YUYV 到 BGR 的颜色转换，但实际测试发现：

- OpenCV 的 `cvtColor(YUYV→BGR)` 已经内置 NEON 优化
- 手写版本无法超越 OpenCV 的性能
- 详见：`docs/13_9_开发记录_NEON加速YUYV转BGR.md`

**P6（其他算法 NEON 优化）也已取消**：

- CLAHE、高斯滤波等 OpenCV 算法在编译时已开启 NEON 优化（`-mfpu=neon`）
- 手写 NEON 代码需要深入理解算法内部实现，开发成本高
- 性能瓶颈已通过其他手段解决（降分辨率处理、三线程管线、全程 BGR）

### 6.2 原优化目标（仅供参考）

| 模块 | 当前实现 | 优化方案 | 预期提升 | 状态 |
|------|---------|---------|---------|---------|
| YUYV → RGB 转换 | OpenCV cvtColor | NEON intrinsic 手写 | 3-5x | ❌ 已验证无法超越 OpenCV |
| CLAHE 增强 | OpenCV CLAHE | 直方图统计 NEON 化 | 2-3x | ❌ OpenCV 已内置 NEON |
| 高斯滤波 | OpenCV GaussianBlur | NEON 定点化卷积 | 2-4x | ❌ OpenCV 已内置 NEON |

### 6.3 结论

对于使用 OpenCV 的项目，**手写 NEON 优化的性价比很低**：

- OpenCV 已经做了大量 NEON 优化，且持续维护
- 手写需要深入理解算法内部，开发周期长
- 维护成本高，代码可读性差

**更好的优化方向**：
- 降分辨率处理（640x480 → 320x240，已完成）
- 三线程管线（采集/处理/显示分离，已完成）
- 全程 BGR 减少颜色转换（已完成）
- 实时/离线处理分离（已完成）

#### 7.1 减少颜色空间转换次数 ✅ 已完成

三线程性能基准测试显示，当前每帧的颜色转换链路占用大量时间：

```
采集线程: YUYV → BGR(16-33ms) → RGB(4-12ms) → QImage
```

练习方向：驱动一个 SPI 接口的 TFT 小屏或 Flash 芯片。

### 8.3 UART（串口通信）✅ A7-M4 RPMsg 已实现

A7-M4 通信使用 OpenAMP RPMsg 虚拟串口（`/dev/ttyRPMSG0`），用法与普通串口几乎一样。

- ✅ 已掌握串口编程（QSerialPort）

### 10.4 LED 补光灯 PWM — 背景与原理

内窥镜光源不是简单的“开灯/关灯”，实际产品采用**频闪照明（strobe illumination）**技术：

```
光源:    ██  　  ██  　  ██  　  ██
         亮  灭  亮  灭  亮  灭  亮
曝光:    [=]     [=]     [=]     [=]
         ↑       ↑       ↑       ↑
       清晰帧  清晰帧  清晰帧  清晰帧
```

**工作原理：**
- 光源只在摄像头曝光瞬间闪亮，持续时间极短（微秒级）
- 相当于每帧都用“闪光灯”拍照，曝光时间极短，运动模糊几乎为零
- 灭灯期间不曝光，所以观看视频时会感觉光在闪动

**为什么需要 M4 做 PWM 控制：**
- **微秒级时序精度**：光源闪烁必须与摄像头曝光精确同步，Linux 的进程调度延迟在毫秒级，无法满足微秒级的实时性要求。曝光时间极短的情况下，可以彻底解决冻结帧的运动模糊问题。
- **硬件定时器**：M4 的 TIM 外设可以输出精确的 PWM 波形，抢占式中断保证响应时间
- **安全红线**：光源常亮会导致探头发热，频闪模式大幅降低平均功耗，减少探头温升（医疗设备对探头温度有严格限制，通常 ≤ 41°C）
- **多模式支持**：可以交替闪烁不同波长的 LED（白光 + 窄带光），实现 NBI（窄带成像）等特殊观察模式

**典型时序示例（胃镜 30fps）：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 帧周期 | 33.3ms | 30fps |
| 光源闪亮时间 | 50-200µs | 曝光窗口 |
| 光源熄灭时间 | ~33ms | 散热期 |
| 占空比 | <1% | 平均功耗极低 |

**建议时间：1-2 周**

所有 linuxfb 程序共享同一个 framebuffer 和输入设备
触摸事件会被所有正在监听的程序同时接收

## 十四、算法集成实战：以锐化为例（P14）

本章以**锐化算法**为作业对象，记录一个新算法从设计到集成、调通、调快的完整流程。

当前已有的 USM 锐化（`applySharpen`）作为基准对照，目标是设计并集成一个更适合内窥镜场景的锐化算法。

---

### 14.1 现有实现（基准）

当前 `applySharpen` 是标准 USM（Unsharp Masking）：

```cpp
void ImageProcessor::applySharpen(const cv::Mat &src, cv::Mat &dst,
                                  double sigma, double amount) {
    cv::Mat blurred;
    cv::GaussianBlur(src, blurred, cv::Size(3, 3), 0);
    cv::addWeighted(src, 1.0 + amount, blurred, -amount, 0, dst);
}
```

**基准性能**（320x240，Cortex-A7）：~7ms/帧

**已知问题**：
- 对噪声敏感，噪声区域也会被锐化放大
- 对所有区域一视同仁，平坦区域（黏膜背景）和边缘区域（病灶轮廓）用同样强度
- 内窥镜图像中，我们更希望只锐化边缘，平坦区域保持平滑

---

### 14.2 算法设计

**目标**：设计一个**自适应边缘锐化**算法，只在边缘区域增强，平坦区域不处理。

**设计思路**：

```
1. 计算每个像素的局部梯度强度（用 Sobel 或 Laplacian）
2. 梯度强度 → 锐化权重掩码（梯度大的地方权重高）
3. 只在权重高的区域叠加 USM 锐化结果
```

**伪代码**：

```
mask = Laplacian(gray(src))  // 边缘强度图
mask = normalize(abs(mask))  // 归一化到 [0,1]
usm  = USM(src)              // 标准 USM 结果
dst  = src * (1 - mask) + usm * mask  // 边缘处用 USM，平坦处用原图
```

**与 USM 的对比**：

| | 标准 USM | 自适应边缘锐化 |
|---|---|---|
| 平坦区域 | 轻微增强（可能放大噪声） | 保持原样 |
| 边缘区域 | 均匀增强 | 按梯度强度增强 |
| 噪声敏感性 | 高 | 低（噪声梯度弱，权重低） |
| 计算量 | 低 | 中（多一次 Laplacian + normalize） |

---

### 14.3 集成流程

集成一个新算法需要改动 4 个地方，顺序固定：

#### 步骤 1：在 `imageprocessor.h` 中声明


#### 步骤 2：在 `imageprocessor.cpp` 中实现

```cpp
void ImageProcessor::applyAdaptiveSharpen(const cv::Mat &src, cv::Mat &dst,
                                           double amount) {
    // 1. 计算边缘掩码（在灰度图上做 Laplacian）
    cv::Mat gray, lap;
    cv::cvtColor(src, gray, cv::COLOR_BGR2GRAY);
    cv::Laplacian(gray, lap, CV_32F, 1);
    cv::convertScaleAbs(lap, lap);  // 取绝对值

    // 2. 归一化到 [0,1]，作为锐化权重
    cv::Mat mask;
    lap.convertTo(mask, CV_32F, 1.0 / 255.0);

    // 3. 计算 USM 结果
    cv::Mat blurred, usm;
    cv::GaussianBlur(src, blurred, cv::Size(3, 3), 0);
    cv::addWeighted(src, 1.0 + amount, blurred, -amount, 0, usm);

    // 4. 按掩码混合：边缘处用 USM，平坦处用原图
    cv::Mat srcF, usmF, dstF;
    src.convertTo(srcF, CV_32FC3);
    usm.convertTo(usmF, CV_32FC3);

    std::vector<cv::Mat> maskChannels = {mask, mask, mask};
    cv::Mat mask3;
    cv::merge(maskChannels, mask3);

    dstF = srcF.mul(1.0f - mask3) + usmF.mul(mask3);
    dstF.convertTo(dst, CV_8UC3);
}
```

#### 步骤 3：在 `process()` 管线中插入

```cpp
// imageprocessor.cpp — process() 函数中，在 sharpen 之后插入
if (cfg.sharpen)
    applySharpen(work, work, 3.0, cfg.sharpenAmount);

if (cfg.adaptiveSharpen)                          // 新增
    applyAdaptiveSharpen(work, work, cfg.adaptiveSharpenAmount);
```

#### 步骤 4：在 `mainwindow.cpp` 中连接 UI

```cpp

```

### 14.5 调快流程

**标准流程：先测耗时 → 找瓶颈 → 针对性优化 → 验证提升**

#### 第一步：测耗时

在 `applyAdaptiveSharpen` 内部加计时：

```cpp
void ImageProcessor::applyAdaptiveSharpen(...) {
    QElapsedTimer t; t.start();

    // Laplacian
    cv::cvtColor(src, gray, cv::COLOR_BGR2GRAY);
    cv::Laplacian(gray, lap, CV_32F, 1);
    qDebug() << "[sharpen] laplacian:" << t.restart() << "ms";

    // normalize
    lap.convertTo(mask, CV_32F, 1.0 / 255.0);
    qDebug() << "[sharpen] normalize:" << t.restart() << "ms";

    // USM
    cv::GaussianBlur(src, blurred, cv::Size(3, 3), 0);
    cv::addWeighted(src, 1.0 + amount, blurred, -amount, 0, usm);
    qDebug() << "[sharpen] usm:" << t.restart() << "ms";

    // 混合
    // ...
    qDebug() << "[sharpen] blend:" << t.restart() << "ms";
}
```

**预期瓶颈**（320x240，Cortex-A7）：

| 步骤 | 预估耗时 | 原因 |
|---|---|---|
| Laplacian | ~3ms | 单通道卷积，较快 |
| normalize | ~1ms | 简单乘法 |
| USM（GaussianBlur） | ~5ms | 3x3 核，已优化 |
| 混合（mul + merge） | ~8ms | 浮点乘法 + 3通道，最慢 |
| **合计** | **~17ms** | |

#### 第二步：针对性优化

**优化 A：用整数运算替代浮点混合（最有效）**

浮点混合是最慢的步骤，改用 `cv::addWeighted` 整数路径：

```cpp
// 优化前：浮点 mul + merge（~8ms）
dstF = srcF.mul(1.0f - mask3) + usmF.mul(mask3);

// 优化后：mask 量化为 uint8，用 addWeighted（~2ms）
cv::Mat mask8;
mask.convertTo(mask8, CV_8U, 255.0);
// 对每个通道：dst = src * (255-mask)/255 + usm * mask/255
// 用 cv::blendLinear 或手动 addWeighted
```

**优化 B：mask 复用 Laplacian 结果，跳过 convertScaleAbs**

```cpp
// 直接用 Laplacian 的绝对值做 mask，省一次转换
cv::Laplacian(gray, lap, CV_8U, 1);  // 直接输出 uint8
// 注意：CV_8U 的 Laplacian 会截断负值，需要确认效果可接受
```

**优化 C：降低 mask 精度，用 8x8 块均值代替逐像素**

```cpp
// 将 mask 缩小 4x 再放大，相当于块平均，减少高频噪声影响
cv::resize(mask, mask, cv::Size(), 0.25, 0.25, cv::INTER_LINEAR);
cv::resize(mask, mask, src.size(), 0, 0, cv::INTER_LINEAR);
```

**优化 D：NEON 加速混合步骤（进阶）**

混合步骤是逐像素乘加，天然适合 NEON 向量化：

```cpp
// 用 NEON intrinsic 手写混合，一次处理 8 个像素
// 预期提升：2-3x
// 详见 P5-P6 NEON 加速章节
```

#### 第三步：优化目标

| 阶段 | 目标耗时 | 手段 |
|---|---|---|
| 初始实现 | ~17ms | 直接实现，正确性优先 |
| 优化 A+B | ~8ms | 整数运算替代浮点 |
| 优化 C | ~6ms | 降低 mask 精度 |
| 优化 D（NEON） | ~3ms | 向量化混合 |

**验收标准**：开启自适应锐化后，总处理时间增加 < 10ms，不触发 overrun。

将图像处理算法按实际内窥镜工作流分为两层：

```
实时流（边看边做）              拍照后处理（存下来再做）
├── 硬件白平衡/曝光 (OV5640)  ├── 锐化（精细，不限时间）
├── CLAHE 增强                ├── 降噪（高质量双边滤波）
├── 去雾                      ├── 边缘检测 / 阈值分割
└── 显示 + 拍照键             └── 测量、标注（后续规划）
```

### 为什么要分离

- 实时流中跑多个重算法导致 FPS 严重下降（CLAHE + 锐化 + 降噪 → 个位数 FPS）
- 医生实时观察只需基础增强，精细处理在拍照后做效果更好、不受帧率限制
- 分离后实时流只跑 CLAHE + 去雾，FPS 显著提升

| 界面 | 功能 |
|---|---|
| 实时采集页 | 视频画面 + 硬件参数 + CLAHE/去雾 + 拍照/录像 |

| 图像编辑页 | 对静态图做锐化/降噪/边缘检测/阈值分割，参数滑块实时预览 |

## 附录：M4 开发完成情况总结

基于 `docs/13_11_开发记录_M4开发设计文档/` 的内容，P2 和 P10 已完成以下功能：

### 已实现功能

| 功能模块 | 实现方式 | 关键技术点 |
|---------|---------|-----------|
| LED 补光灯 PWM 调光 | PB10 软件 PWM 500Hz | TIM3 中断 50μs，40 级计数，0-100% 占空比 |
| 温度监控 | SHT30 软件 I2C | PA11/PA12 GPIO 模拟，每 5 秒读取，>41°C 降亮度 |
| 按键检测 | TIM3 中断轮询 | 10ms 周期，200ms 软件消抖，KEY1 冻结/KEY2 拍照 |
| A7-M4 通信 | OpenAMP RPMsg 虚拟串口 | `/dev/ttyRPMSG0`，协议 T/H/K/L/A |
| 多任务管理 | FreeRTOS | Task_Main（优先级 2）+ Task_Temperature（优先级 1） |
| 温度超限保护 | M4 本地判断 | >43°C 关闭 LED，不依赖 A7 指令 |

### 已解决的关键技术问题

| 问题编号 | 问题描述 | 根本原因 | 解决方案 |
|---------|---------|---------|---------|
| #19 | M4 无法使用 EXTI 中断 | STM32MP1 双核 EXTI 仲裁限制 | 改用 TIM3 中断轮询按键 |
| #20 | remoteproc 无法识别 ELF | GCC 13 链接器 segment alignment | 添加 `-Wl,-z,max-page-size=0x10000` |
| #20 | RPMsg 通道建立失败 | service name 不匹配 | 改为 `"rpmsg-tty-channel"` |
| #21 | stop/start 后 RPMsg 损坏 | remoteproc 驱动限制 | 每次更新固件必须重启开发板 |
| #22 | LCD 背光熄灭 | TIM2 与 LCD 背光冲突 | 改用 TIM3 软件 PWM |
| #22 v2 | LED 闪烁 | 软件 PWM 100Hz 太低 | 提高到 500Hz（50μs 中断） |
| #24 | 触摸屏失效 | I2C1 与触摸屏共享 | 改用软件 I2C（PA11/PA12） |
| #26 | RPMsg endpoint 建立 | M4 需要 A7 先发消息 | A7 自动重试定时器 |
| #27 A | DCMI overrun | HAL_Delay 忙等待占用总线 | 改用 vTaskDelay 让出 CPU |
| #27 B | 程序崩溃 | RPMsg 高频信号与 UI 冲突 | 温度读取频率降为 5 秒 |

### CubeMX 生成后必须手动修复的 6 项

每次 CubeMX 重新生成代码后必须修复：

1. `CM4/OPENAMP/openamp_conf.h`：取消注释 `#define VIRTUAL_UART_MODULE_ENABLED`
2. `Middlewares/.../virt_uart.c`：`RPMSG_SERVICE_NAME` 改为 `"rpmsg-tty-channel"`
3. `Middlewares/.../freertos/template/sys.c`：实现 `sys_irq_save_disable`/`sys_irq_restore_enable`
4. `CM4/Core/Src/main.c`：`#include "cmsis_os.h"` → `#include "FreeRTOS.h"` + `#include "task.h"`
5. `CM4/Core/Src/main.c`：`osThreadDef/osKernelStart` → `MX_FREERTOS_Init()` + `vTaskStartScheduler()`
6. `CM4/Core/Src/main.c`：TIM3 `htim3.Init.Period` 从 `99` 改为 `49`

### 硬件接线方案

```
开发板 J1 扩展口：
  Pin 7  (PB10)  → HW-269 LED data（软件 PWM 500Hz）
  Pin 23 (5V)    → HW-269 VCC
  Pin 1  (GND)   → HW-269 GND
  Pin 11 (PA12)  → SHT30 SCL（软件 I2C）
  Pin 12 (PA11)  → SHT30 SDA（软件 I2C）
  Pin 19 (3V3)   → SHT30 VCC
  Pin 2  (GND)   → SHT30 GND

板载按键：
  KEY1 (PB13)    → 冻结/恢复（TIM3 轮询）
  KEY2 (PH7)     → 拍照（TIM3 轮询）
```

### 通信协议

| 方向 | 格式 | 含义 | 示例 |
|-----|------|------|------|
| A7→M4 | `L<value>\n` | 设置 LED 亮度 0-100% | `L75\n` |
| M4→A7 | `T<temp>\n` | 温度上报（×10） | `T256\n` = 25.6°C |
| M4→A7 | `H<humi>\n` | 湿度上报（×10） | `H654\n` = 65.4% |
| M4→A7 | `K1\n` | KEY1 按下（冻结/恢复） | `K1\n` |
| M4→A7 | `K2\n` | KEY2 按下（拍照） | `K2\n` |
| M4→A7 | `A1\n` / `A2\n` | 温度警告/严重 | `A2\n` = >43°C |
