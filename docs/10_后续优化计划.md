## 四、线程架构优化（P3）

当前采集和处理在同一个 QThread 中串行执行，优化为三级流水线。

```
采集线程 → [环形队列] → 处理线程 → [环形队列] → 渲染线程(主线程)
```

### 4.1 环形 Buffer 队列

```cpp
// src/utils/framebuffer.h
template<int N>
class FrameRingBuffer {
    cv::Mat frames[N];
    std::atomic<int> writeIdx{0};
    std::atomic<int> readIdx{0};
public:
    bool push(const cv::Mat& frame);  // 采集线程写入
    bool pop(cv::Mat& frame);         // 处理线程读取
};
```

- 固定 3-4 个 buffer 槽位
- 无锁设计，采集线程和处理线程各操作一端
- 队列满时丢弃最旧帧，保证实时性

### 4.2 线程优先级设置

```cpp
#include <pthread.h>

struct sched_param param;
param.sched_priority = 80;
pthread_setschedparam(pthread_self(), SCHED_FIFO, &param);
```

- 关联问题：问题 #14：Overrun 的真正机制 — 线程调度问题

**建议时间：3-4 天**

### 后续优化计划

#### 7.1 减少颜色空间转换次数

当前每帧的颜色转换链路：
```
采集线程: YUYV → BGR → RGB → QImage
processFrame: QImage → Mat(BGR) → 算法处理 → Mat → QImage
```
共 4~5 次转换。可以考虑全程使用 BGR，只在最终显示时转 RGB，减少 2~3 次转换（预计省 10~15ms/帧）。

#### 7.2 图像处理线程化

当前图像处理在主线程执行，阻塞 UI 刷新。可以将处理移到独立线程：
```
采集线程 → 处理线程 → 主线程（只做显示）
```
UI 不再被处理耗时阻塞，按钮响应更流畅。

#### 7.3 NEON 加速

Cortex-A7 支持 NEON SIMD 指令集。OpenCV 编译时开启 NEON 可以加速矩阵运算。当前交叉编译已开启 `-mfpu=neon`，但需确认 OpenCV 是否编译时启用了 NEON 优化。

#### 7.4 特定算法全分辨率处理

当前所有算法统一降到 320x240 处理。如果后续对画质有更高要求，可以对特定算法（锐化、去雾、边缘检测）改回全分辨率，以性能换画质。

#### 7.5 端到端延迟监控

添加采集时间戳，计算从采集到显示的端到端延迟，便于评估实时性。

**建议时间：2 天**（后续优化部分）

```cpp
struct spi_ioc_transfer tr = { .tx_buf = tx, .rx_buf = rx, .len = len, .speed_hz = 1000000 };
ioctl(fd, SPI_IOC_MESSAGE(1), &tr);
```

练习方向：驱动一个 SPI 接口的 TFT 小屏或 Flash 芯片。

### 8.3 UART（串口通信）

```cpp

```

练习方向：A7 与 M4 之间通过 UART 通信，或连接外部 MCU/模块。

### 10.4 LED 补光灯 PWM — 背景与原理

内窥镜光源不是简单的“开灯/关灯”，实际产品采用**频闪照明（strobe illumination）**技术：

```
光源:    ██  　  ██  　  ██  　  ██
         亮  灭  亮  灭  亮  灭  亮
曝光:    [=]     [=]     [=]     [=]
         ↑       ↑       ↑       ↑
       清晰帧  清晰帧  清晰帧  清晰帧
```

**工作原理：**
- 光源只在摄像头曝光瞬间闪亮，持续时间极短（微秒级）
- 相当于每帧都用“闪光灯”拍照，曝光时间极短，运动模糊几乎为零
- 灭灯期间不曝光，所以观看视频时会感觉光在闪动

**为什么需要 M4 做 PWM 控制：**
- **微秒级时序精度**：光源闪烁必须与摄像头曝光精确同步，Linux 的进程调度延迟在毫秒级，无法满足微秒级的实时性要求。曝光时间极短的情况下，可以彻底解决冻结帧的运动模糊问题。
- **硬件定时器**：M4 的 TIM 外设可以输出精确的 PWM 波形，抢占式中断保证响应时间
- **安全红线**：光源常亮会导致探头发热，频闪模式大幅降低平均功耗，减少探头温升（医疗设备对探头温度有严格限制，通常 ≤ 41°C）
- **多模式支持**：可以交替闪烁不同波长的 LED（白光 + 窄带光），实现 NBI（窄带成像）等特殊观察模式

**典型时序示例（胃镜 30fps）：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 帧周期 | 33.3ms | 30fps |
| 光源闪亮时间 | 50-200µs | 曝光窗口 |
| 光源熄灭时间 | ~33ms | 散热期 |
| 占空比 | <1% | 平均功耗极低 |

**建议时间：1-2 周**

## 十三、开发板启动项优化（P13，产品优化）

点击内窥镜程序界面时会发生穿透 - 这是因为 linuxfb 平台没有窗口管理器，你的程序和桌面程序共享同一个 framebuffer 和输入设备，触摸事件会同时被两边接收。

这是 linuxfb 平台的特性：
linuxfb 没有窗口管理器，不像 X11/Wayland 能管理窗口层级和输入焦点
所有 linuxfb 程序共享同一个 framebuffer 和输入设备
触摸事件会被所有正在监听的程序同时接收
所以在 linuxfb 环境下，同时只能运行一个 GUI 程序，这是正常的。以后可以在启动脚本里先杀掉 App 再启动内窥镜：
# 每次测试时手动执行
kill $(pidof App) 2>/dev/null
cp /mnt/nfs/endoscope /tmp/
/tmp/endoscope -platform linuxfb

**建议时间：1天**
脚本内容后我不建议直接禁用它。这个脚本做了很多重要的环境配置：
触摸屏设备检测和校准
屏幕旋转设置
tslib 环境变量配置
字体路径设置
如果直接 chmod -x 禁用，你的触摸屏可能就不工作了。
更好的做法是：修改脚本，保留环境配置，只把最后一行启动 App 改成启动内窥镜。但这个等项目稳定后再做，现在开发阶段手动操作更灵活：

# 每次测试时手动执行
kill $(pidof App) 2>/dev/null
cp /mnt/nfs/endoscope /tmp/
/tmp/endoscope -platform linuxfb

等项目最终部署时，再把启动脚本改成启动内窥镜程序。

## 十四、算法集成实战：以锐化为例（P14）

本章以**锐化算法**为作业对象，记录一个新算法从设计到集成、调通、调快的完整流程。

当前已有的 USM 锐化（`applySharpen`）作为基准对照，目标是设计并集成一个更适合内窥镜场景的锐化算法。

---

### 14.1 现有实现（基准）

当前 `applySharpen` 是标准 USM（Unsharp Masking）：

```cpp
void ImageProcessor::applySharpen(const cv::Mat &src, cv::Mat &dst,
                                  double sigma, double amount) {
    cv::Mat blurred;
    cv::GaussianBlur(src, blurred, cv::Size(3, 3), 0);
    cv::addWeighted(src, 1.0 + amount, blurred, -amount, 0, dst);
}
```

**基准性能**（320x240，Cortex-A7）：~7ms/帧

**已知问题**：
- 对噪声敏感，噪声区域也会被锐化放大
- 对所有区域一视同仁，平坦区域（黏膜背景）和边缘区域（病灶轮廓）用同样强度
- 内窥镜图像中，我们更希望只锐化边缘，平坦区域保持平滑

---

### 14.2 算法设计

**目标**：设计一个**自适应边缘锐化**算法，只在边缘区域增强，平坦区域不处理。

**设计思路**：

```
1. 计算每个像素的局部梯度强度（用 Sobel 或 Laplacian）
2. 梯度强度 → 锐化权重掩码（梯度大的地方权重高）
3. 只在权重高的区域叠加 USM 锐化结果
```

**伪代码**：

```
mask = Laplacian(gray(src))  // 边缘强度图
mask = normalize(abs(mask))  // 归一化到 [0,1]
usm  = USM(src)              // 标准 USM 结果
dst  = src * (1 - mask) + usm * mask  // 边缘处用 USM，平坦处用原图
```

**与 USM 的对比**：

| | 标准 USM | 自适应边缘锐化 |
|---|---|---|
| 平坦区域 | 轻微增强（可能放大噪声） | 保持原样 |
| 边缘区域 | 均匀增强 | 按梯度强度增强 |
| 噪声敏感性 | 高 | 低（噪声梯度弱，权重低） |
| 计算量 | 低 | 中（多一次 Laplacian + normalize） |

### 14.3 集成流程

集成一个新算法需要改动 4 个地方，顺序固定：

#### 步骤 1：在 `imageprocessor.h` 中声明

```cpp
// 在 Config 结构体中添加开关和参数
struct Config {
    // ... 现有字段 ...
    bool adaptiveSharpen = false;       // 新增
    double adaptiveSharpenAmount = 1.5; // 新增
};

// 在类中声明静态方法
static void applyAdaptiveSharpen(const cv::Mat &src, cv::Mat &dst,
                                  double amount = 1.5);
```

#### 步骤 2：在 `imageprocessor.cpp` 中实现

```cpp
void ImageProcessor::applyAdaptiveSharpen(const cv::Mat &src, cv::Mat &dst,
                                           double amount) {
    // 1. 计算边缘掩码（在灰度图上做 Laplacian）
    cv::Mat gray, lap;
    cv::cvtColor(src, gray, cv::COLOR_BGR2GRAY);
    cv::Laplacian(gray, lap, CV_32F, 1);
    cv::convertScaleAbs(lap, lap);  // 取绝对值

    // 2. 归一化到 [0,1]，作为锐化权重
    cv::Mat mask;
    lap.convertTo(mask, CV_32F, 1.0 / 255.0);

    // 3. 计算 USM 结果
    cv::Mat blurred, usm;
    cv::GaussianBlur(src, blurred, cv::Size(3, 3), 0);
    cv::addWeighted(src, 1.0 + amount, blurred, -amount, 0, usm);

    // 4. 按掩码混合：边缘处用 USM，平坦处用原图
    cv::Mat srcF, usmF, dstF;
    src.convertTo(srcF, CV_32FC3);
    usm.convertTo(usmF, CV_32FC3);

    std::vector<cv::Mat> maskChannels = {mask, mask, mask};
    cv::Mat mask3;
    cv::merge(maskChannels, mask3);

    dstF = srcF.mul(1.0f - mask3) + usmF.mul(mask3);
    dstF.convertTo(dst, CV_8UC3);
}
```

#### 步骤 3：在 `process()` 管线中插入

```cpp
// imageprocessor.cpp — process() 函数中，在 sharpen 之后插入
if (cfg.sharpen)
    applySharpen(work, work, 3.0, cfg.sharpenAmount);

if (cfg.adaptiveSharpen)                          // 新增
    applyAdaptiveSharpen(work, work, cfg.adaptiveSharpenAmount);
```

#### 步骤 4：在 `mainwindow.cpp` 中连接 UI

```cpp
// 在 buildConfig() 中读取 UI 状态
cfg.adaptiveSharpen = m_chkAdaptiveSharpen->isChecked();
cfg.adaptiveSharpenAmount = m_sliderAdaptiveSharpen->value() / 10.0;
```

### 14.4 调通流程

**在虚拟机上先跑通，再上开发板。**

#### 阶段 1：单元测试（不依赖摄像头）

写一个独立的测试程序，用静态图片验证算法效果：

```cpp
// test_sharpen.cpp
int main() {
    cv::Mat src = cv::imread("test.jpg");

    cv::Mat dst_usm, dst_adaptive;
    ImageProcessor::applySharpen(src, dst_usm, 3.0, 1.5);
    ImageProcessor::applyAdaptiveSharpen(src, dst_adaptive, 1.5);

    cv::imwrite("out_usm.jpg", dst_usm);
    cv::imwrite("out_adaptive.jpg", dst_adaptive);

    // 清晰度对比
    printf("原图清晰度:    %.1f\n", ImageProcessor::calcSharpness(src));
    printf("USM 清晰度:    %.1f\n", ImageProcessor::calcSharpness(dst_usm));
    printf("自适应清晰度:  %.1f\n", ImageProcessor::calcSharpness(dst_adaptive));
    return 0;
}
```

**验证要点**：
- 输出图像不崩溃、不花屏
- 边缘区域有锐化效果
- 平坦区域（纯色背景）与原图无明显差异
- 清晰度分数高于原图

#### 阶段 2：接入视频流（虚拟机）

用 `filecapture` 接测试视频，在 UI 上勾选新算法，观察：
- 实时效果是否符合预期
- FPS 是否在可接受范围（虚拟机上 > 15fps 即可）
- 参数滑块调节是否有响应

#### 阶段 3：上开发板验证

交叉编译后部署到开发板，验证：
- 效果与虚拟机一致
- FPS 是否满足要求（目标 > 20fps）
- 是否触发 overrun（观察 dmesg）

**常见问题排查**：

| 现象 | 可能原因 | 排查方法 |
|---|---|---|
| 输出全黑 | convertTo 溢出或类型错误 | 打印 src/dst 的 type()、min/max 值 |
| 边缘过度增强 | mask 归一化范围不对 | 打印 mask 的 min/max，检查是否在 [0,1] |
| 平坦区域也变化 | mask 阈值太低 | 对 mask 加阈值截断：`mask = max(mask - 0.1, 0)` |
| 开发板效果与虚拟机不同 | 浮点精度差异 | 检查是否有 CV_64F，改为 CV_32F |

### 14.5 调快流程

**标准流程：先测耗时 → 找瓶颈 → 针对性优化 → 验证提升**

#### 第一步：测耗时

在 `applyAdaptiveSharpen` 内部加计时：

```cpp
void ImageProcessor::applyAdaptiveSharpen(...) {
    QElapsedTimer t; t.start();

    // Laplacian
    cv::cvtColor(src, gray, cv::COLOR_BGR2GRAY);
    cv::Laplacian(gray, lap, CV_32F, 1);
    qDebug() << "[sharpen] laplacian:" << t.restart() << "ms";

    // normalize
    lap.convertTo(mask, CV_32F, 1.0 / 255.0);
    qDebug() << "[sharpen] normalize:" << t.restart() << "ms";

    // USM
    cv::GaussianBlur(src, blurred, cv::Size(3, 3), 0);
    cv::addWeighted(src, 1.0 + amount, blurred, -amount, 0, usm);
    qDebug() << "[sharpen] usm:" << t.restart() << "ms";

    // 混合
    // ...
    qDebug() << "[sharpen] blend:" << t.restart() << "ms";
}
```

**预期瓶颈**（320x240，Cortex-A7）：

| 步骤 | 预估耗时 | 原因 |
|---|---|---|
| Laplacian | ~3ms | 单通道卷积，较快 |
| normalize | ~1ms | 简单乘法 |
| USM（GaussianBlur） | ~5ms | 3x3 核，已优化 |
| 混合（mul + merge） | ~8ms | 浮点乘法 + 3通道，最慢 |
| **合计** | **~17ms** | |

#### 第二步：针对性优化

**优化 A：用整数运算替代浮点混合（最有效）**

浮点混合是最慢的步骤，改用 `cv::addWeighted` 整数路径：

```cpp
// 优化前：浮点 mul + merge（~8ms）
dstF = srcF.mul(1.0f - mask3) + usmF.mul(mask3);

// 优化后：mask 量化为 uint8，用 addWeighted（~2ms）
cv::Mat mask8;
mask.convertTo(mask8, CV_8U, 255.0);
// 对每个通道：dst = src * (255-mask)/255 + usm * mask/255
// 用 cv::blendLinear 或手动 addWeighted
```

**优化 B：mask 复用 Laplacian 结果，跳过 convertScaleAbs**

```cpp
// 直接用 Laplacian 的绝对值做 mask，省一次转换
cv::Laplacian(gray, lap, CV_8U, 1);  // 直接输出 uint8
// 注意：CV_8U 的 Laplacian 会截断负值，需要确认效果可接受
```

**优化 C：降低 mask 精度，用 8x8 块均值代替逐像素**

```cpp
// 将 mask 缩小 4x 再放大，相当于块平均，减少高频噪声影响
cv::resize(mask, mask, cv::Size(), 0.25, 0.25, cv::INTER_LINEAR);
cv::resize(mask, mask, src.size(), 0, 0, cv::INTER_LINEAR);
```

**优化 D：NEON 加速混合步骤（进阶）**

混合步骤是逐像素乘加，天然适合 NEON 向量化：

```cpp
// 用 NEON intrinsic 手写混合，一次处理 8 个像素
// 预期提升：2-3x
// 详见 P5-P6 NEON 加速章节
```

#### 第三步：优化目标

| 阶段 | 目标耗时 | 手段 |
|---|---|---|
| 初始实现 | ~17ms | 直接实现，正确性优先 |
| 优化 A+B | ~8ms | 整数运算替代浮点 |
| 优化 C | ~6ms | 降低 mask 精度 |
| 优化 D（NEON） | ~3ms | 向量化混合 |

**验收标准**：开启自适应锐化后，总处理时间增加 < 10ms，不触发 overrun。

### 14.7 通用方法论总结

这套流程适用于任何新算法的集成：

```
设计阶段：明确目标 → 选择算法思路 → 与现有实现对比分析

集成阶段：
  1. imageprocessor.h  — 添加 Config 字段 + 方法声明
  2. imageprocessor.cpp — 实现算法
  3. imageprocessor.cpp — 插入 process() 管线
  4. mainwindow.cpp    — 连接 UI 控件

调通阶段：
  静态图片单测 → 虚拟机视频流 → 开发板实测
  每步验证：不崩溃 → 效果正确 → 参数可调

调快阶段：
  加计时 → 找最慢步骤 → 整数化/降精度/降分辨率 → NEON（进阶）
  每步验证：效果不退化 → 耗时下降
```

**建议时间：3-5 天（含设计、集成、调通、基础优化）**
