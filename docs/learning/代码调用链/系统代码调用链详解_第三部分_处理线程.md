# 系统代码调用链详解 — 第三部分：处理线程

## 一、处理线程的职责

处理线程是管线的中间环节，负责：
1. 从 captureQueue 取帧（BGR 640×480）
2. 执行图像处理算法（如果启用）
3. resize 到显示尺寸（460×360）
4. 送入 displayQueue 供主线程显示

```
captureQueue → pop → process/resize → push → displayQueue
```

---

## 二、处理主循环（run 函数）

### 2.1 整体流程

```cpp
void ProcessThread::run() {
    m_running = true;

    while (!m_quit) {
        if (!m_running) { msleep(10); continue; }  // 暂停

        cv::Mat frame;
        if (!m_captureQueue->pop(frame)) {          // 队列空
            msleep(1); continue;
        }

        cv::Mat result;
        if (!anyEnabled) {
            // 不开算法：直接 resize
            cv::resize(frame, result, Size(displayWidth, displayHeight));
        } else {
            // 开算法：process 管线 + resize
            cv::Mat dst;
            ImageProcessor::process(frame, dst, config, map1, map2);
            cv::resize(dst, result, Size(displayWidth, displayHeight));
        }

        m_displayQueue->push(std::move(result));    // 零拷贝送出
    }
}
```

### 2.2 config 参数同步

处理线程的 `config` 成员由主线程每 16ms 同步一次：

```
主线程 displayTimer 回调：
  m_processThread->config.clahe = m_chkClahe->isChecked();
  m_processThread->config.dehaze = m_chkDehaze->isChecked();
  // ... 其他参数 ...

处理线程 run() 中：
  bool anyEnabled = config.clahe || config.undistort || config.dehaze || ...;
```

这里没有加锁——config 是简单的 POD 结构体（bool + double + int），主线程写、处理线程读，在 ARM 上单个字段的读写是原子的，不会读到半写状态。最坏情况是某一帧用了旧参数，下一帧就会更新，对实时视频无影响。

## 三、ImageProcessor::process 管线详解

### 3.1 调用链

```cpp
void ImageProcessor::process(const cv::Mat &src, cv::Mat &dst,
                             const Config &cfg,
                             const cv::Mat &map1, const cv::Mat &map2) {
    // 降分辨率处理
    cv::Mat work;
    cv::resize(src, work, Size(), 0.5, 0.5);   // 640×480 → 320×240

    // 按固定顺序执行启用的算法
    if (cfg.whiteBalance) applyWhiteBalance(work, work);   // 不会执行（始终 false）
    if (cfg.clahe)        applyCLAHE(work, work, cfg.claheClipLimit);
    if (cfg.undistort)    cv::remap(work, work, map1, map2, INTER_LINEAR);
    if (cfg.dehaze)       applyDehaze(work, work, cfg.dehazeOmega, cfg.dehazeRadius);
    if (cfg.sharpen)      applySharpen(work, work, 3.0, cfg.sharpenAmount);
    if (cfg.denoise)      applyDenoise(work, work, cfg.denoiseD);
    if (cfg.edgeDetect)   applyEdgeDetect(work, work);
    if (cfg.threshold)    applyThreshold(work, work, cfg.thresholdValue);

    dst = work;  // 输出 320×240（由调用方 resize 到显示尺寸）
}
```

### 3.2 为什么先降分辨率

640×480 上处理太慢（Cortex-A7），降到 320×240 后像素量减少 75%，耗时降 4-5 倍：

```
640×480 = 307,200 像素    320×240 = 76,800 像素
CLAHE:  ~115ms            CLAHE:  ~28ms
去雾:   ~390ms            去雾:   ~87ms
```

最终显示尺寸是 460×360，320×240 放大 1.5 倍，画质损失在 800×480 小屏上不明显。

### 3.3 算法顺序的设计理由

```
白平衡 → CLAHE → 畸变校正 → 去雾 → 锐化 → 降噪 → 边缘检测 → 阈值分割
```

| 顺序 | 理由 |
|---|---|
| 白平衡最先 | 先校正色偏，后续算法基于正确颜色工作 |
| CLAHE 在畸变校正前 | 畸变校正会引入黑边，黑边会影响 CLAHE 的直方图统计 |
| 去雾在锐化前 | 先恢复清晰度再增强细节，否则锐化会放大雾的噪声 |
| 降噪在锐化后 | 锐化会放大噪声，降噪在后面清理 |
| 边缘检测/阈值分割最后 | 这是分析类算法，输出是特殊图像（边缘图/二值图），放最后不影响前面 |

## 四、各算法的数据变换过程

### 4.1 CLAHE（applyCLAHE）

输入 BGR → 输出 BGR，内部经过 Lab 色彩空间：

```
输入: BGR 320×240
  │
  ▼ cv::cvtColor(BGR → Lab)
Lab 320×240
  │ L 通道 = 感知均匀的亮度
  │ a 通道 = 绿-红色度
  │ b 通道 = 蓝-黄色度
  │
  ▼ cv::split(Lab → [L, a, b])
L 通道 320×240 (单通道 uint8)
  │
  ▼ clahe->apply(L, L)    ← 只对 L 通道做自适应均衡化
  │   将图像分成 8×8 块
  │   每块独立做直方图均衡化
  │   clipLimit 限制对比度放大倍数
  │   块边界做双线性插值消除接缝
  │
  ▼ cv::merge([L, a, b] → Lab)
Lab 320×240（L 已增强，a/b 不变 → 色彩保留）
  │
  ▼ cv::cvtColor(Lab → BGR)
输出: BGR 320×240

耗时：~28ms（其中 BGR↔Lab 转换各 ~5ms，CLAHE 本身 ~18ms）
```

### 4.2 去雾（applyDehaze）

输入 BGR → 输出 BGR，内部用浮点运算：

```
输入: BGR 320×240 (uint8)
  │
  ▼ src.convertTo(srcFloat, CV_32FC3, 1.0/255.0)
BGR float 320×240（像素值归一化到 0.0-1.0）
  │
  ▼ cv::split → [B, G, R] 三个单通道 float
  │
  ▼ 暗通道计算
  │   darkChannel = min(B, G, R)     每像素取三通道最小值
  │   cv::erode(darkChannel, kernel) 最小值滤波（7×7 窗口）
  │
  ▼ 大气光估计
  │   A = mean(srcFloat, darkChannel > maxVal * 0.9)
  │   取暗通道最亮区域（最可能是雾的区域）的均值作为大气光
  │
  ▼ 透射率估计
  │   对每个通道除以大气光 A，再取最小值
  │   transmission = 1 - omega × 归一化暗通道
  │   max(transmission, 0.1)  下限 0.1 防止除零
  │
  ▼ 场景恢复
  │   channels[i] = (channels[i] - A[i]) / transmission + A[i]
  │
  ▼ cv::merge → BGR float
  │
  ▼ dst.convertTo(dst, CV_8UC3, 255.0)
输出: BGR 320×240 (uint8)

耗时：~87ms（最慢的算法，主要耗时在 erode 和 divide）
```

### 4.3 USM 锐化（applySharpen）

```
输入: BGR 320×240
  │
  ▼ cv::GaussianBlur(src, blurred, Size(3,3), 0)
  │   3×3 高斯模糊，得到低频分量
  │
  ▼ cv::addWeighted(src, 1+amount, blurred, -amount, 0, dst)
  │   dst = src × (1+amount) - blurred × amount
  │       = src + amount × (src - blurred)
  │                         ↑ 高频细节（边缘+噪声）
  │
输出: BGR 320×240（边缘被增强）

耗时：~7ms
```

### 4.4 双边滤波降噪（applyDenoise）

```
输入: BGR 320×240
  │
  ▼ cv::resize(src, small, 0.5, 0.5)
BGR 160×120（再缩小一半，加速双边滤波）
  │
  ▼ cv::bilateralFilter(small, tmp, d=5, sigmaColor=50, sigmaSpace=50)
  │   空间权重：距离近的邻居权重大（和高斯模糊一样）
  │   像素值权重：值接近的邻居权重大（边缘两侧值差异大，不会被模糊）
  │   → 平滑噪声但保留边缘
  │
  ▼ cv::resize(tmp, dst, src.size())
输出: BGR 320×240（放大回原尺寸）

耗时：~55ms（双边滤波本身很慢，缩小后处理是关键优化）
```

### 4.5 边缘检测（applyEdgeDetect）

```
输入: BGR 320×240
  │
  ▼ cv::cvtColor(BGR → Gray)
Gray 320×240
  │
  ▼ cv::blur(gray, edge, Size(3,3))    轻微模糊去噪
  │
  ▼ cv::Canny(edge, edge, 3, 9, 3)    Canny 边缘检测
  │   低阈值 3，高阈值 9，核大小 3
  │   输出二值图：边缘像素=255，非边缘=0
  │
  ▼ cv::cvtColor(Gray → BGR)           转回 BGR 保持管线格式一致
输出: BGR 320×240（灰度边缘图，三通道值相同）

耗时：~19ms
```

### 4.6 阈值分割（applyThreshold）

```
输入: BGR 320×240
  │
  ▼ cv::cvtColor(BGR → Gray)
  │
  ▼ cv::threshold(gray, gray, threshValue, 255, THRESH_BINARY)
  │   > threshValue → 255（白）
  │   ≤ threshValue → 0（黑）
  │
  ▼ cv::cvtColor(Gray → BGR)
输出: BGR 320×240（二值图）

耗时：~1ms
```

## 五、处理线程的两条路径对比

### 5.1 不开算法

```
captureQueue.pop(frame)          BGR 640×480
  │
  ▼ cv::resize → result         BGR 460×360
  │   耗时：22-59ms（CPU 调度波动）
  │
  ▼ displayQueue.push(move)
```

### 5.2 开 CLAHE

```
captureQueue.pop(frame)          BGR 640×480
  │
  ▼ ImageProcessor::process
  │   resize(640→320)            BGR 320×240     ~2ms
  │   applyCLAHE                 BGR 320×240     ~28ms
  │   dst = work                 BGR 320×240
  │
  ▼ cv::resize(320→460×360)     BGR 460×360     ~12ms
  │
  ▼ displayQueue.push(move)

合计：~42ms → 理论 ~24fps（实际受 CPU 调度影响为 ~13fps）
```

### 5.3 开 CLAHE + 去雾

```
captureQueue.pop(frame)          BGR 640×480
  │
  ▼ ImageProcessor::process
  │   resize(640→320)            BGR 320×240     ~2ms
  │   applyCLAHE                 BGR 320×240     ~28ms
  │   applyDehaze                BGR 320×240     ~87ms
  │   dst = work                 BGR 320×240
  │
  ▼ cv::resize(320→460×360)     BGR 460×360     ~12ms
  │
  ▼ displayQueue.push(move)

合计：~129ms → 理论 ~8fps（实际 ~5fps）
```

## 六、处理线程耗时分析

| 场景 | 各环节耗时 | 合计 | 实际 FPS |
|---|---|---|---|
| 不开算法 | resize 22-59ms | 22-59ms | 17-22fps |
| 开 CLAHE | resize 2ms + CLAHE 28ms + resize 12ms | ~42ms | ~13fps |
| 开 CLAHE+去雾 | resize 2ms + CLAHE 28ms + 去雾 87ms + resize 12ms | ~129ms | ~5fps |

理论 FPS = 1000 / 合计耗时，但实际 FPS 更低，因为：
1. CPU 调度波动（其他线程争抢 CPU）
2. pop 等待时间（captureQueue 可能为空）
3. 主线程 displayTimer 的 16ms 轮询间隔
