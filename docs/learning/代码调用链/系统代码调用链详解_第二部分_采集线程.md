# 系统代码调用链详解 — 第二部分：采集线程

## 一、采集线程的职责

采集线程是整个管线的源头，负责：
1. 从 V4L2 驱动获取摄像头原始数据（YUYV 格式）
2. 颜色转换（YUYV → BGR）
3. 将 BGR Mat 送入 captureQueue 供处理线程消费

```
V4L2 驱动 → grabFrame → cvtColor → push → captureQueue
```

---

## 二、V4L2 初始化流程（open 函数）

程序启动时 `setupVideoSource()` 调用 `V4l2Capture::open()`，按固定顺序执行 5 步：

### 2.1 第 1 步：打开设备

```cpp
m_fd = ::open("/dev/video0", O_RDWR | O_NONBLOCK);
```

| 标志 | 含义 |
|---|---|
| O_RDWR | 可读可写（采集需要读，控制参数需要写） |
| O_NONBLOCK | 非阻塞模式（配合 select 使用，避免 read 卡死） |

`m_fd` 是文件描述符，后续所有 V4L2 操作都通过它进行。Linux 中一切皆文件，摄像头也是一个文件。

### 2.2 第 2 步：查询设备能力（queryCapability）

```cpp
struct v4l2_capability cap;
ioctl(m_fd, VIDIOC_QUERYCAP, &cap);
```

检查两个关键能力：
- `V4L2_CAP_VIDEO_CAPTURE` — 支持视频采集
- `V4L2_CAP_STREAMING` — 支持 streaming I/O（mmap 方式需要）

如果不支持，直接报错退出。

### 2.3 第 3 步：设置像素格式和分辨率（tryAndSetFormat）

```cpp
struct v4l2_format fmt;
fmt.fmt.pix.width = 640;
fmt.fmt.pix.height = 480;
fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_YUYV;
fmt.fmt.pix.field = V4L2_FIELD_NONE;        // 逐行扫描

ioctl(m_fd, VIDIOC_TRY_FMT, &fmt);  // 试探：驱动调整到最接近的支持值
ioctl(m_fd, VIDIOC_S_FMT, &fmt);    // 正式设置
```

TRY_FMT 不会真正生效，只是让驱动告诉你它能支持什么。如果请求 640×480 但驱动只支持 320×240，`fmt.fmt.pix.width/height` 会被驱动修改。

### 2.4 第 4 步：设置帧率（setFrameRate）

```cpp
struct v4l2_streamparm parm;
parm.parm.capture.timeperframe.numerator = 1;
parm.parm.capture.timeperframe.denominator = 30;  // 30fps

ioctl(m_fd, VIDIOC_S_PARM, &parm);
```

帧率 = denominator / numerator = 30/1 = 30fps。设置失败不致命，使用驱动默认帧率。

### 2.5 第 5 步：申请缓冲区并 mmap 映射（requestBuffers）

这是 V4L2 最核心的部分，分三个子步骤：

**子步骤 A：申请缓冲区**

```cpp
struct v4l2_requestbuffers req;
req.count = 4;                          // 请求 4 个
req.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
req.memory = V4L2_MEMORY_MMAP;          // 使用 mmap 方式

ioctl(m_fd, VIDIOC_REQBUFS, &req);
// req.count 可能被驱动修改为实际分配的数量
```

**子步骤 B：查询并映射每个缓冲区**

```cpp
for (int i = 0; i < req.count; i++) {
    struct v4l2_buffer buf;
    buf.index = i;
    ioctl(m_fd, VIDIOC_QUERYBUF, &buf);  // 查询偏移量和大小

    m_buffers[i].start = mmap(NULL, buf.length,
                              PROT_READ | PROT_WRITE,
                              MAP_SHARED, m_fd, buf.m.offset);
    m_buffers[i].length = buf.length;
}
```

mmap 后 `m_buffers[i].start` 直接指向内核缓冲区，摄像头数据写入后用户空间立刻能读到，不需要 copy。

**子步骤 C：将所有缓冲区放入采集队列**

```cpp
for (int i = 0; i < req.count; i++) {
    struct v4l2_buffer buf;
    buf.index = i;
    ioctl(m_fd, VIDIOC_QBUF, &buf);  // 入队，等待驱动填充
}
```

初始化完成后的状态：

```
内核采集队列：[buf0] [buf1] [buf2] [buf3]   ← 全部在队列中等待填充
用户空间：     mmap 映射已建立，可以直接读取
```

## 三、采集主循环（run 函数）

### 3.1 整体流程

```cpp
void V4l2Capture::run() {
    startStreaming();    // VIDIOC_STREAMON，驱动开始往 buffer 填数据
    m_running = true;

    while (!m_quit) {
        // 1. 暂停检查
        if (!m_running) { returnBuffer(bufIndex); continue; }

        // 2. 获取一帧
        grabFrame(&data, &size, &bufIndex);

        // 3. 颜色转换
        cv::cvtColor(yuyv, bgr, COLOR_YUV2BGR_YUYV);

        // 4. 归还 buffer
        returnBuffer(bufIndex);

        // 5. 送入队列
        m_captureQueue->push(std::move(bgr));
    }

    stopStreaming();     // VIDIOC_STREAMOFF
}
```

### 3.2 grabFrame 详解

```cpp
bool V4l2Capture::grabFrame(void **data, int *size, int *bufIndex) {
    // select 等待数据就绪（最多 2 秒超时）
    fd_set fds;
    FD_SET(m_fd, &fds);
    struct timeval tv = {2, 0};
    select(m_fd + 1, &fds, NULL, NULL, &tv);

    // DQBUF：取出已填充数据的缓冲区
    struct v4l2_buffer buf;
    ioctl(m_fd, VIDIOC_DQBUF, &buf);

    *data = m_buffers[buf.index].start;  // mmap 地址，直接指向 YUYV 数据
    *size = buf.bytesused;               // 实际数据大小
    *bufIndex = buf.index;               // 记录索引，用完后归还
    return true;
}
```

**select 的作用**：阻塞等待摄像头数据就绪。摄像头 30fps 意味着每 33ms 产出一帧，select 在数据就绪时立即返回，不浪费 CPU。

**DQBUF 的含义**：Dequeue Buffer，从内核采集队列中取出一个已填充数据的缓冲区。取出后该 buffer 不在队列中，驱动不会再往里写数据，用户可以安全读取。

### 3.3 Buffer 轮转机制

```
时间线（4 个 buffer）：

T0:  驱动填充 buf0
T1:  DQBUF 取出 buf0 → 用户读取（cvtColor）
     驱动同时填充 buf1
T2:  QBUF 归还 buf0 → 驱动可以再次使用 buf0
     DQBUF 取出 buf1 → 用户读取
     驱动同时填充 buf2
T3:  QBUF 归还 buf1
     DQBUF 取出 buf2
     驱动同时填充 buf3
...

关键：用户处理 buf0 的同时，驱动在填充 buf1/buf2/buf3
      这就是"流水线"——采集和处理并行进行
```

**为什么 returnBuffer 必须在 cvtColor 之后**（问题 #7）：

```
错误顺序：DQBUF → QBUF（归还）→ cvtColor（读取）
  → 归还后驱动可能立刻往 buffer 写新数据
  → cvtColor 读到的是新旧混合数据 → 花屏

正确顺序：DQBUF → cvtColor（读取完成）→ QBUF（归还）
  → 数据已拷贝到 bgr，归还 buffer 安全
```

### 3.4 颜色转换

```cpp
cv::Mat yuyv(m_height, m_width, CV_8UC2, data);
cv::Mat bgr;
cv::cvtColor(yuyv, bgr, cv::COLOR_YUV2BGR_YUYV);
```

| 步骤 | 说明 |
|---|---|
| `cv::Mat yuyv(...)` | 用 mmap 地址包装为 Mat，不拷贝数据。CV_8UC2 = 每像素 2 字节（YUYV） |
| `cv::cvtColor(...)` | 内部分配新的 bgr Mat（640×480×3），执行 YUYV→BGR 转换 |

cvtColor 内部对每个像素做 BT.601 转换：

```
R = Y + 1.402 × (V - 128)
G = Y - 0.344 × (U - 128) - 0.714 × (V - 128)
B = Y + 1.772 × (U - 128)

然后按 BGR 顺序存储（OpenCV 默认通道顺序）
```

OpenCV 的 cvtColor 已内置 NEON 优化（CV_NEON=1），耗时 17-55ms（平均 ~33ms）。

### 3.5 送入队列

```cpp
m_captureQueue->push(std::move(bgr));
```

`std::move(bgr)` 将 bgr 的数据指针转移给队列槽位：
- 转移前：bgr 持有 921KB 数据，队列槽位为空
- 转移后：bgr 变为空 Mat，队列槽位持有 921KB 数据
- 耗时：~0ms（只交换几个指针，不拷贝像素数据）

转移后 bgr 为空，下一帧循环中 cvtColor 会分配新的 bgr，不会冲突。

### 3.6 暂停/恢复机制

```cpp
while (!m_quit) {                    // m_quit 控制线程退出
    grabFrame(&data, &size, &bufIndex);

    if (!m_running) {                // m_running 控制暂停
        returnBuffer(bufIndex);      // 暂停时仍然归还 buffer
        continue;                    // 跳过处理，但保持 V4L2 流活跃
    }
    // ... 正常处理 ...
}
```

**为什么暂停时仍然 grabFrame + returnBuffer**（问题 #16）：

V4L2 的 STREAMOFF 会取消所有 buffer 的排队。如果用 stop/start 暂停恢复，STREAMOFF 后所有 buffer 被取消，STREAMON 时没有 buffer 在队列中，采集失败。

所以用 pause/resume 代替 stop/start：线程始终运行，暂停时只是不处理帧，但仍然 grabFrame + returnBuffer 保持 V4L2 流活跃。

## 四、硬件参数控制

### 4.1 调用链

```
UI 滑块拖动
  → Qt 信号 valueChanged(int)
    → lambda: v4l2->setSaturation(value)
      → setControl(V4L2_CID_SATURATION, value)
        → ioctl(m_fd, VIDIOC_S_CTRL, &ctrl)
          → Linux 内核 V4L2 子系统
            → OV5640 驱动 (drivers/media/i2c/ov5640.c)
              → I2C 总线写入 OV5640 寄存器
                → Sensor 内部 ISP 更新参数
                  → 下一帧图像生效
```

### 4.2 所有控制参数

| 方法 | V4L2 CID | OV5640 功能 | 范围 | 说明 |
|---|---|---|---|---|
| setAutoWhiteBalance | V4L2_CID_AUTO_WHITE_BALANCE | AWB 开关 | 0/1 | 默认开启，替代软件白平衡 |
| setAutoExposure | V4L2_CID_EXPOSURE_AUTO | AE 开关 | 0(手动)/1(自动) | 默认自动 |
| setExposure | V4L2_CID_EXPOSURE | 手动曝光值 | 0-65535 | 需先关闭 AE |
| setGain | V4L2_CID_GAIN | 手动增益 | 0-1023 | 需先关闭 AGC |
| setAutoGain | V4L2_CID_AUTOGAIN | AGC 开关 | 0/1 | |
| setContrast | V4L2_CID_CONTRAST | 对比度 | 0-255 | 硬件 ISP 处理，零 CPU 开销 |
| setSaturation | V4L2_CID_SATURATION | 饱和度 | 0-255 | 硬件 ISP 处理，零 CPU 开销 |
| setHue | V4L2_CID_HUE | 色调 | 0-359 | 一般不调 |
| setHFlip | V4L2_CID_HFLIP | 水平翻转 | 0/1 | 根据摄像头安装方向 |
| setVFlip | V4L2_CID_VFLIP | 垂直翻转 | 0/1 | 根据摄像头安装方向 |
| setRedBalance | V4L2_CID_RED_BALANCE | 红色增益 | 0-4095 | 需先关闭 AWB |
| setBlueBalance | V4L2_CID_BLUE_BALANCE | 蓝色增益 | 0-4095 | 需先关闭 AWB |

### 4.3 硬件参数 vs 软件算法

| | 硬件参数（ioctl） | 软件算法（ImageProcessor） |
|---|---|---|
| 执行位置 | OV5640 Sensor 内部 ISP | Cortex-A7 CPU |
| CPU 开销 | 零 | 7-87ms/帧 |
| 处理阶段 | RAW 数据阶段（cvtColor 之前） | BGR 数据阶段（cvtColor 之后） |
| 精度 | 高（在原始数据上处理） | 较低（在已转换的数据上处理） |
| 灵活性 | 受 Sensor 支持的参数限制 | 可以实现任意算法 |
| 例子 | 对比度、饱和度、白平衡 | CLAHE、去雾、锐化 |

## 五、FileCapture（虚拟机测试用）

### 5.1 与 V4l2Capture 的关系

```
VideoSource（抽象基类）
  ├── V4l2Capture   — 开发板，从摄像头采集
  └── FileCapture   — 虚拟机，从视频文件读取
```

MainWindow 只持有 `VideoSource*` 指针，不关心具体是哪个子类。切换视频源只需 new 不同的子类，其他代码不用改（多态）。

### 5.2 FileCapture 的 run 循环

```cpp
void FileCapture::run() {
    m_running = true;
    int delayMs = 1000 / m_fps;  // 30fps → 33ms

    while (!m_quit) {
        if (!m_running) { QThread::msleep(50); continue; }

        cv::Mat frame;
        m_cap.read(frame);           // OpenCV VideoCapture 读一帧（BGR 格式）

        if (m_captureQueue) {
            m_captureQueue->push(frame);  // 直接 push BGR Mat
        }

        QThread::msleep(delayMs);    // 模拟摄像头帧率
    }
}
```

与 V4l2Capture 的区别：
- 不需要 V4L2 初始化、mmap、DQBUF/QBUF
- `VideoCapture::read()` 输出的直接就是 BGR Mat，不需要 cvtColor
- 用 msleep 模拟帧率（真实摄像头由硬件控制帧率）

## 六、采集线程耗时分析

### 6.1 各环节耗时

```
select 等待:     ~0ms（数据就绪时立即返回，等待时间不算 CPU 耗时）
DQBUF:           ~0ms
cvtColor:        17-55ms（平均 ~33ms，OpenCV NEON 优化）
returnBuffer:    ~0ms
push(move):      0-1ms（P16 零拷贝）
─────────────────────────
合计:            ~33ms/帧 → 理论上限 ~30fps
```

### 6.2 为什么 cvtColor 波动大（17-55ms）

不是算法本身不稳定，而是 CPU 调度波动：
- 处理线程和主线程也在争抢 CPU
- 当其他线程占用 CPU 时，cvtColor 被中断，实际执行时间拉长
- 这是单核 Cortex-A7 的固有限制（见 16_FPS性能问题总跟踪 第五章）
