# V4L2 摄像头采集完整流程

## 一、整体流程概览

```
应用程序                          内核驱动                        硬件
                                                              
1. open("/dev/video0")  ────→  打开设备文件              
2. ioctl(QUERYCAP)      ────→  查询设备能力              
3. ioctl(S_FMT)         ────→  设置格式/分辨率           
4. ioctl(S_PARM)        ────→  设置帧率                  
5. ioctl(REQBUFS)       ────→  申请内核缓冲区            
6. ioctl(QUERYBUF)      ────→  查询缓冲区地址            
7. mmap()               ────→  映射到用户空间            
8. ioctl(QBUF)          ────→  缓冲区入队                
9. ioctl(STREAMON)      ────→  开始采集          ────→  摄像头开始输出数据
                                                              
   ┌─ 采集循环 ──────────────────────────────────────────────┐
   │ 10. select()        ────→  等待数据就绪               │
   │ 11. ioctl(DQBUF)   ────→  取出已填充的缓冲区         │
   │ 12. 处理数据（YUYV→RGB→显示）                          │
   │ 13. ioctl(QBUF)    ────→  归还缓冲区                 │
   │ 14. 回到 10                                           │
   └───────────────────────────────────────────────────────┘
                                                              
15. ioctl(STREAMOFF)    ────→  停止采集          ────→  摄像头停止输出
16. munmap()            ────→  解除映射                  
17. close()             ────→  关闭设备                  
```

## 二、每一步详解

### 2.1 打开设备

```c
int fd = open("/dev/video0", O_RDWR | O_NONBLOCK);
```

- `/dev/video0` — Linux 把摄像头抽象为文件，读写文件就是读写摄像头
- `O_RDWR` — 可读可写（采集需要读，控制参数需要写）
- `O_NONBLOCK` — 非阻塞模式，配合后面的 select 使用
- 返回值 `fd` — 文件描述符，后续所有操作都通过它

### 2.2 查询设备能力

```c
struct v4l2_capability cap;
ioctl(fd, VIDIOC_QUERYCAP, &cap);

// 检查是否支持视频采集
if (!(cap.capabilities & V4L2_CAP_VIDEO_CAPTURE))
    // 不是摄像头，退出

// 检查是否支持 streaming（mmap 方式需要）
if (!(cap.capabilities & V4L2_CAP_STREAMING))
    // 不支持 mmap，退出
```

这一步是确认设备确实是摄像头，而且支持我们要用的采集方式。

### 2.3 设置像素格式和分辨率

```c
struct v4l2_format fmt;
memset(&fmt, 0, sizeof(fmt));
fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
fmt.fmt.pix.width = 640;
fmt.fmt.pix.height = 480;
fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_YUYV;  // YUYV 格式
fmt.fmt.pix.field = V4L2_FIELD_NONE;            // 逐行扫描

ioctl(fd, VIDIOC_S_FMT, &fmt);

// 注意：驱动可能调整参数，实际值以 fmt 返回的为准
int actual_width = fmt.fmt.pix.width;
int actual_height = fmt.fmt.pix.height;
```

### 2.4 设置帧率

```c
struct v4l2_streamparm parm;
memset(&parm, 0, sizeof(parm));
parm.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
parm.parm.capture.timeperframe.numerator = 1;    // 分子
parm.parm.capture.timeperframe.denominator = 30;  // 分母 → 1/30 = 30fps

ioctl(fd, VIDIOC_S_PARM, &parm);
```

### 2.5 申请缓冲区

```c
struct v4l2_requestbuffers req;
memset(&req, 0, sizeof(req));
req.count = 4;                            // 请求 4 个缓冲区
req.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
req.memory = V4L2_MEMORY_MMAP;            // 使用 mmap 方式

ioctl(fd, VIDIOC_REQBUFS, &req);

// 注意：驱动实际分配的数量可能少于请求值
int actual_count = req.count;
```

这一步只是让内核分配缓冲区，用户空间还不能访问。

### 2.6 查询缓冲区并 mmap 映射

这是最关键的一步——把内核缓冲区映射到用户空间：

```c
struct FrameBuffer {
    void *start;    // 用户空间地址
    size_t length;  // 缓冲区大小
};

struct FrameBuffer buffers[4];

for (int i = 0; i < actual_count; i++) {
    // 查询第 i 个缓冲区的信息
    struct v4l2_buffer buf;
    memset(&buf, 0, sizeof(buf));
    buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    buf.memory = V4L2_MEMORY_MMAP;
    buf.index = i;
    ioctl(fd, VIDIOC_QUERYBUF, &buf);

    // mmap：把内核缓冲区映射到用户空间
    buffers[i].length = buf.length;
    buffers[i].start = mmap(
        NULL,                   // 让系统选择映射地址
        buf.length,             // 映射大小
        PROT_READ | PROT_WRITE, // 可读可写
        MAP_SHARED,             // 与内核共享（驱动写入的数据用户立刻能看到）
        fd,                     // 设备文件描述符
        buf.m.offset            // 内核缓冲区的偏移量
    );

    // 把缓冲区放入采集队列，等待驱动填充
    ioctl(fd, VIDIOC_QBUF, &buf);
}
```

### 2.7 mmap 的工作原理

```
                    物理内存
                 ┌──────────┐
                 │ buffer 0 │ ← 摄像头 DMA 直接写入
                 │ buffer 1 │
                 │ buffer 2 │
                 │ buffer 3 │
                 └──────────┘
                   ↑      ↑
                   │      │
          内核空间  │      │  用户空间
          (驱动)   │      │  (应用程序)
                   │      │
              驱动写入   mmap 映射后
              像素数据   buffers[i].start
                         直接指向同一块物理内存
```

关键点：
- **零拷贝**：mmap 后用户空间和内核空间看到的是同一块物理内存，不需要 copy
- **驱动写入后用户立刻可见**：摄像头通过 DMA 把数据写入 buffer，用户空间直接读取
- `buffers[i].start` 就是像素数据的起始地址，可以直接当数组用

### 2.8 开始采集

```c
enum v4l2_buf_type type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
ioctl(fd, VIDIOC_STREAMON, &type);
```

此时摄像头开始输出数据，驱动从队列中取出空缓冲区，让 DMA 填充。

### 2.9 采集循环

```c
while (running) {
    // 1. 等待数据就绪（select 阻塞，直到有缓冲区被填充）
    fd_set fds;
    FD_ZERO(&fds);
    FD_SET(fd, &fds);
    struct timeval tv = {2, 0};  // 超时 2 秒
    select(fd + 1, &fds, NULL, NULL, &tv);

    // 2. 取出已填充的缓冲区（DQBUF）
    struct v4l2_buffer buf;
    memset(&buf, 0, sizeof(buf));
    buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    buf.memory = V4L2_MEMORY_MMAP;
    ioctl(fd, VIDIOC_DQBUF, &buf);

    // 3. 此时 buffers[buf.index].start 指向本帧的像素数据
    //    数据格式是 YUYV，需要转换后才能显示
    void *data = buffers[buf.index].start;
    int size = buf.bytesused;

    // 4. 处理数据（转换格式、显示等）
    process_frame(data, size);

    // 5. 归还缓冲区（QBUF），驱动可以继续往里写数据
    ioctl(fd, VIDIOC_QBUF, &buf);
}
```

### 2.10 缓冲区轮转机制

```
时间线（4 个 buffer）：

T0: 队列 [buf0, buf1, buf2, buf3]
    驱动取出 buf0，DMA 开始填充

T1: 驱动填充完 buf0
    用户 DQBUF 取出 buf0 → 处理数据
    驱动取出 buf1，DMA 开始填充

T2: 用户处理完 buf0，QBUF 归还
    队列 [buf2, buf3, buf0]
    驱动填充完 buf1
    用户 DQBUF 取出 buf1 → 处理数据

T3: ...循环往复

如果用户处理太慢，所有 buffer 都被占用：
    驱动没有空 buffer → overrun 错误！
```

### 2.11 停止和清理

```c
// 停止采集
enum v4l2_buf_type type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
ioctl(fd, VIDIOC_STREAMOFF, &type);

// 解除 mmap 映射
for (int i = 0; i < actual_count; i++)
    munmap(buffers[i].start, buffers[i].length);

// 关闭设备
close(fd);
```

## 三、YUYV 到 RGB 的转换

### 3.1 为什么要转换

摄像头输出的是 **YUYV**（也叫 YUV422），但屏幕显示需要 **RGB**。两者是不同的颜色表示方式：

| 格式 | 含义 | 每像素字节 | 用途 |
|---|---|---|---|
| YUYV | Y=亮度, U/V=色度 | 2 字节 | 摄像头输出（节省带宽） |
| BGR | 蓝绿红三通道 | 3 字节 | OpenCV 默认格式 |
| RGB | 红绿蓝三通道 | 3 字节 | Qt QImage 显示格式 |

**为什么摄像头用 YUYV 而不是 RGB？**

YUYV 每像素只需 2 字节（两个像素共享 U、V），RGB 需要 3 字节。640×480 一帧：
- YUYV：640 × 480 × 2 = 614,400 字节
- RGB：640 × 480 × 3 = 921,600 字节

YUYV 节省 33% 的带宽，对嵌入式设备的 DCMI 接口和内存都很重要。

### 3.2 YUYV 的数据排列

```
YUYV 格式中，每 4 个字节描述 2 个像素：

字节:  [Y0] [U0] [Y1] [V0] [Y2] [U1] [Y3] [V1] ...
像素:   px0        px1        px2        px3

像素 0 = (Y0, U0, V0)
像素 1 = (Y1, U0, V0)  ← 和像素 0 共享 U0、V0
像素 2 = (Y2, U1, V1)
像素 3 = (Y3, U1, V1)  ← 和像素 2 共享 U1、V1
```

### 3.3 YUV 到 RGB 的数学公式

```
R = Y + 1.402 × (V - 128)
G = Y - 0.344 × (U - 128) - 0.714 × (V - 128)
B = Y + 1.772 × (U - 128)

结果需要 clamp 到 [0, 255]
```

### 3.4 OpenCV 转换方法（项目中使用的方式）

```cpp
// YUYV 原始数据在 mmap buffer 中
void *data = buffers[buf.index].start;

// 用 cv::Mat 包装原始数据（不拷贝，只是创建一个指向 data 的头）
// YUYV 每像素 2 字节，所以类型是 CV_8UC2（8位无符号，2通道）
cv::Mat yuyv(480, 640, CV_8UC2, data);

// OpenCV 一行代码完成转换
cv::Mat bgr;
cv::cvtColor(yuyv, bgr, cv::COLOR_YUV2BGR_YUYV);  // YUYV → BGR

cv::Mat rgb;
cv::cvtColor(bgr, rgb, cv::COLOR_BGR2RGB);          // BGR → RGB（Qt 需要）
```

为什么要转两次：
```
YUYV → BGR：OpenCV 所有算法基于 BGR 格式
BGR → RGB：Qt 的 QImage 使用 RGB 格式

如果不做图像处理，可以直接：
YUYV → RGB：cv::cvtColor(yuyv, rgb, cv::COLOR_YUV2RGB_YUYV);
```

### 3.5 不用 OpenCV 的手写转换

```c
// 纯 C 实现 YUYV → RGB
void yuyv_to_rgb(unsigned char *yuyv, unsigned char *rgb, int width, int height) {
    int size = width * height * 2;  // YUYV 总字节数

    for (int i = 0, j = 0; i < size; i += 4, j += 6) {
        int y0 = yuyv[i + 0];
        int u  = yuyv[i + 1];
        int y1 = yuyv[i + 2];
        int v  = yuyv[i + 3];

        // 像素 0
        int r0 = y0 + 1.402 * (v - 128);
        int g0 = y0 - 0.344 * (u - 128) - 0.714 * (v - 128);
        int b0 = y0 + 1.772 * (u - 128);

        // 像素 1
        int r1 = y1 + 1.402 * (v - 128);
        int g1 = y1 - 0.344 * (u - 128) - 0.714 * (v - 128);
        int b1 = y1 + 1.772 * (u - 128);

        // clamp 到 [0, 255]
        rgb[j + 0] = r0 < 0 ? 0 : (r0 > 255 ? 255 : r0);
        rgb[j + 1] = g0 < 0 ? 0 : (g0 > 255 ? 255 : g0);
        rgb[j + 2] = b0 < 0 ? 0 : (b0 > 255 ? 255 : b0);
        rgb[j + 3] = r1 < 0 ? 0 : (r1 > 255 ? 255 : r1);
        rgb[j + 4] = g1 < 0 ? 0 : (g1 > 255 ? 255 : g1);
        rgb[j + 5] = b1 < 0 ? 0 : (b1 > 255 ? 255 : b1);
    }
}
```

### 3.6 NEON 加速转换（ARM 优化，后续 P5）

手写版本每次处理 2 个像素，NEON 可以一次处理 8 个像素：

## 四、从 mmap 到屏幕显示的完整数据流

### 4.1 我们项目中的完整链路

```
摄像头硬件
    ↓ DMA 写入
内核 buffer（物理内存）
    ↓ mmap 映射（零拷贝）
用户空间 buffers[i].start（指向同一块物理内存）
    ↓ cv::Mat yuyv(height, width, CV_8UC2, data)  — 包装，不拷贝
OpenCV Mat（YUYV 格式）
    ↓ cv::cvtColor(yuyv, bgr, COLOR_YUV2BGR_YUYV) — 第一次拷贝
OpenCV Mat（BGR 格式）
    ↓ returnBuffer() — 此时才归还 buffer 给驱动
    ↓ cv::cvtColor(bgr, rgb, COLOR_BGR2RGB)        — 第二次拷贝
OpenCV Mat（RGB 格式）
    ↓ QImage image(rgb.data, ...)                   — 包装，不拷贝
QImage（引用 rgb 的内存）
    ↓ image.copy()                                  — 第三次拷贝（深拷贝）
QImage（独立内存）
    ↓ emit frameReady(image)                        — 跨线程传递到主线程
    ↓ processFrame() — CLAHE/去雾等算法处理
    ↓ update() → paintEvent()
QPainter::drawImage()
    ↓ 写入 framebuffer
屏幕显示
```

### 4.2 为什么 returnBuffer 要在 cvtColor 之后

```
错误顺序（问题 #7 的 bug）：
  DQBUF → returnBuffer → cvtColor
  归还后驱动可能立刻往 buffer 写新数据
  cvtColor 读到的是新旧混合的数据 → 花屏

正确顺序：
  DQBUF → cvtColor（数据已拷贝到 bgr）→ returnBuffer
  拷贝完成后再归还，buffer 中的数据已经不需要了
```

### 4.3 为什么需要 image.copy()

```cpp
// rgb 是局部变量，函数返回后内存释放
cv::Mat rgb;
cv::cvtColor(bgr, rgb, cv::COLOR_BGR2RGB);

// QImage 只是包装 rgb.data 的指针，不拥有内存
QImage image(rgb.data, rgb.cols, rgb.rows, rgb.step, QImage::Format_RGB888);

// 如果直接 emit image，主线程收到时 rgb 已经释放 → 野指针 → 崩溃
// 所以必须 copy() 创建独立副本
emit frameReady(image.copy());
```

## 六、采集 → 处理 → 显示管线

### 6.1 当前架构：两线程管线

```
采集线程（QThread）                          主线程（Qt 事件循环）
                                            
┌─────────────────────────┐                ┌─────────────────────────┐
│ while (!m_quit) {       │                │                         │
│   select() 等待数据      │                │  onFrameReady(image)    │
│   DQBUF 取出 buffer     │                │    ├── processFrame()   │
│   cvtColor YUYV→BGR     │                │    │   ├── QImage→Mat    │
│   returnBuffer 归还      │                │    │   ├── resize 缩小   │
│   cvtColor BGR→RGB      │   frameReady   │    │   ├── CLAHE/去雾    │
│   QImage 包装           │ ──────────→    │    │   ├── resize 到显示  │
│   emit frameReady(copy) │   (信号槽)     │    │   └── Mat→QImage    │
│ }                       │                │    ├── 缓存帧            │
└─────────────────────────┘                │    ├── 旋转（如果需要）   │
                                           │    ├── 录像（如果需要）   │
                                           │    ├── m_displayImage =  │
                                           │    └── update()          │
                                           │                         │
                                           │  paintEvent()           │
                                           │    ├── fillRect 黑色背景 │
                                           │    ├── drawImage 画视频帧│
                                           │    └── drawText 画 OSD  │
                                           │                         │
                                           │  其他事件处理            │
                                           │    ├── 按钮点击          │
                                           │    ├── 滑块拖动          │
                                           │    └── 定时器（FPS）     │
                                           └─────────────────────────┘
```

### 6.2 一帧数据的完整生命周期

```
时间   采集线程                              主线程
─────────────────────────────────────────────────────────────
T0     select() 等待...                     处理上一帧 / 空闲
       
T1     DQBUF 取出 buf[2]                   
       data = buf[2].start                  
       ↓                                    
T2     cvtColor(YUYV→BGR)  ~15ms           
       数据从 mmap buffer 拷贝到 bgr        
       ↓                                    
T3     returnBuffer(2)                      
       buf[2] 归还给驱动                     
       ↓                                    
T4     检查 m_pendingFrames < 2?            
       ↓ 是                                 
T5     cvtColor(BGR→RGB)   ~5ms            
       QImage 包装 + copy   ~5ms            
       m_pendingFrames++                    
       emit frameReady(image)               
       ↓                                    
T6     回到 select() 等待下一帧     ──→     onFrameReady(image) 被调用
                                            ↓
T7                                          processFrame()
                                            ├── QImage→Mat        ~5ms
                                            ├── resize 640→320    ~2ms
                                            ├── CLAHE             ~28ms
                                            ├── 去雾              ~87ms
                                            ├── resize 320→显示   ~2ms
                                            └── Mat→QImage        ~5ms
                                            ↓
T8                                          m_displayImage = processed
                                            update()
                                            m_pendingFrames--
                                            ↓
T9                                          paintEvent() 被 Qt 调度
                                            drawImage() 画到屏幕
                                            ↓
                                            一帧完成，等待下一个信号
```

### 6.3 各环节耗时（Cortex-A7 实测）

**采集线程：**

| 环节 | 耗时 | 说明 |
|---|---|---|
| select 等待 | ~33ms | 摄像头 30fps，每 33ms 产出一帧 |
| DQBUF | <1ms | ioctl 调用 |
| cvtColor YUYV→BGR | ~15ms | 640×480 像素转换 |
| returnBuffer | <1ms | ioctl 调用 |
| cvtColor BGR→RGB | ~5ms | 640×480 像素转换 |
| QImage copy | ~5ms | 深拷贝 900KB 图像数据 |
| **合计** | **~37ms** | 略大于 33ms，偶尔丢帧 |

**主线程（不开算法）：**

| 环节 | 耗时 | 说明 |
|---|---|---|
| scaled 缩放 | ~5ms | 缩放到显示尺寸 |
| paintEvent | ~2ms | drawImage 画到 framebuffer |
| **合计** | **~7ms** | |

**主线程（开 CLAHE）：**

| 环节 | 耗时 | 说明 |
|---|---|---|
| QImage→Mat | ~5ms | 格式转换 |
| resize 缩小 | ~2ms | 640→320 |
| CLAHE | ~28ms | 算法处理 |
| resize 到显示 | ~2ms | 320→显示尺寸 |
| Mat→QImage | ~5ms | 格式转换 |
| paintEvent | ~2ms | |
| **合计** | **~44ms** | |

### 6.4 FPS 瓶颈分析

FPS 取决于**最慢的环节**：

```
不开算法：
  采集线程 ~37ms，主线程 ~7ms
  瓶颈在采集线程 → FPS ≈ 1000/37 ≈ 27fps

开 CLAHE：
  采集线程 ~37ms，主线程 ~44ms
  瓶颈在主线程 → FPS ≈ 1000/44 ≈ 22fps
  但 m_pendingFrames 限制导致实际更低 → ~13fps

开 CLAHE + 去雾：
  采集线程 ~37ms，主线程 ~130ms
  瓶颈在主线程 → FPS ≈ 1000/130 ≈ 8fps
```

**为什么实际 FPS 比理论值更低？**

因为采集线程和主线程是**串行依赖**的：

```
采集线程：  [采集帧1]  [采集帧2]  [采集帧3]  [采集帧4]
                ↓          ↓ 丢弃    ↓ 丢弃     ↓
主线程：    [处理帧1（44ms）]      [处理帧4（44ms）]

帧2、帧3 到达时主线程还在处理帧1，m_pendingFrames >= 2，被丢弃
```

### 6.5 m_pendingFrames 机制

```cpp
// 采集线程：发送前检查
if (m_pendingFrames.load() >= 2) continue;  // 队列满，跳过这帧
m_pendingFrames.fetch_add(1);               // 计数 +1
emit frameReady(image.copy());              // 发送

// 主线程：处理完后递减
void MainWindow::onFrameReady(const QImage &image) {
    // ... 处理帧 ...
    m_source->m_pendingFrames.fetch_sub(1);  // 计数 -1
}
```

作用：限制 Qt 信号队列中最多 2 帧，防止主线程处理慢时队列无限堆积导致 OOM（见问题记录 #11）。

代价：主线程处理慢时，大量帧被丢弃，实际 FPS 远低于采集帧率。

### 6.6 理想架构：三线程管线（P3 规划）

```
采集线程              处理线程（新增）          主线程
                                              
[grabFrame]          [processFrame]          [paintEvent]
[cvtColor]           [CLAHE/去雾]            [drawImage]
[emit] ──→ 环形队列 ──→ [emit] ──→ 环形队列 ──→ [显示最新结果]
  ~37ms                ~44ms                   ~2ms

三个线程并行，互不阻塞：
- 采集线程：全速采集，不受处理速度影响
- 处理线程：独立跑算法，处理完一帧就送给主线程
- 主线程：只负责显示，UI 始终流畅

FPS = 采集帧率（~27fps），不受算法耗时影响
延迟 = 采集 + 处理 + 显示 ≈ 37 + 44 + 2 = 83ms
```

### 6.7 当前管线的已知问题汇总

| 问题 | 原因 | 现状 | 解决方案 |
|---|---|---|---|
| 开算法后 FPS 下降 | 算法在主线程阻塞 | 已通过降分辨率缓解 | P3 三线程管线 |
| Overrun 错误 | 系统负载高，采集线程响应慢 | 15% overrun 率 | P3 + P5 NEON |
| 颜色转换次数多 | YUYV→BGR→RGB→QImage→Mat→BGR→处理→Mat→QImage | 4-5 次转换 | P7.1 全程 BGR |
| UI 卡顿 | 主线程被算法阻塞 | Phase 1 精简后缓解 | P3 处理线程分离 |
| 帧丢弃严重 | m_pendingFrames 限制 | 开算法时丢 60-70% 帧 | P3 环形队列 |
