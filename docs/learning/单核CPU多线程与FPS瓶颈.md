# 单核 CPU 上的多线程与 FPS 瓶颈——从项目实践中学到的

## 一、为什么 FPS 上不去

### 1.1 不是单纯的"算法太慢"

FPS 瓶颈是两层叠加：

```
第一层：硬件算力不足（不可改变）
  Cortex-A7 单核 650MHz，三个线程共享
  → 即使不开算法，简单的 resize 也要 22-59ms（理论 5-10ms）

第二层：算法计算量大（可优化但空间有限）
  CLAHE 28ms + 去雾 87ms
  → OpenCV 已内置 NEON 优化，手写也快不了
```

### 1.2 当前 FPS 已接近硬件极限

| 场景 | FPS | 说明 |
|---|---|---|
| 不开算法 | 21-22fps | 接近采集帧率上限 |
| 开 CLAHE | ~13fps | 算法 + CPU 调度 |
| 开 CLAHE+去雾 | ~5fps | 算法计算量的物理限制 |

实际内窥镜产品用更强的芯片（Cortex-A53 四核 + 硬件 ISP）不存在这个问题。

## 二、单核 CPU 怎么跑多线程

### 2.1 轮流执行，不是同时执行

单核 CPU 同一时刻只能执行一个线程。三个线程"同时运行"其实是快速轮流切换：

```
CPU: [采集线程][处理线程][主线程][采集线程][处理线程][主线程]...
      2ms      3ms     1ms    2ms      3ms     1ms

切换速度很快，看起来像同时在跑，实际是轮流的
```

Linux 调度器（CFS）负责决定每个线程跑多久然后切换。每次切换叫上下文切换，需要保存/恢复寄存器状态。

### 2.2 时间片分配

一个核的时间是 100%，三个线程分：

```
三个线程都很忙时：
  采集线程：~33%
  处理线程：~33%
  主线程：  ~33%
```

时间片分配由 Linux CFS 调度器自动决定，默认公平分配。我们可以通过 SCHED_FIFO 干预，但在单核上设高优先级会饿死其他线程，不建议做。

### 2.3 墙钟时间 vs CPU 时间

```
采集线程的 cvtColor 需要 15ms 的 CPU 时间来完成

两线程时：
  CPU: [===cvtColor 7ms===][主线程 3ms][===cvtColor 8ms===]
  墙钟时间：~18ms（被打断一次）

三线程时：
  CPU: [cvtColor 5ms][处理3ms][主1ms][cvtColor 5ms][处理3ms][主1ms][cvtColor 5ms]
  墙钟时间：~23ms（被打断多次）
```

日志里 QElapsedTimer 测的是墙钟时间（从开始到结束），不是 CPU 时间。所以看到的是 40-50ms 而不是 15ms。

## 三、为什么三线程后 FPS 反而降了

### 3.1 数据对比

| 阶段 | 不开算法 FPS | 原因 |
|---|---|---|
| 两线程 | 22-23fps | 2 个线程分 CPU |
| 三线程 | ~20fps | 3 个线程分 CPU，多了调度开销 |
| 三线程 + 全程 BGR | 17-20fps | clone 每帧 2×900KB 内存拷贝 |
| 三线程 + 零拷贝 | 21-22fps | std::move 消除 clone，恢复 |

### 3.2 为什么不撤回三线程

三线程解决的是 UI 卡顿问题，不是 FPS 问题：

```
两线程时开算法：
  主线程：[===processFrame 130ms===][响应按钮][刷新画面]
  → 130ms 内画面不刷新、按钮点不动、完全卡死

三线程时开算法：
  主线程：[取帧5ms][空闲11ms][取帧5ms][空闲11ms]...
  → 每 16ms 刷新一次，按钮随时响应
  → FPS 没变（处理线程是瓶颈），但用户体验完全不同
```

2-3fps 的损失换来 UI 流畅性，这个权衡是值得的。

## 四、单核上的关键规律

### 4.1 省下的时间会被其他线程吃掉

这是单核 CPU 最反直觉的特性。在多核上，一个核空闲了就是真空闲。在单核上，一个线程省下的时间立刻被调度器分给其他线程。

**实验验证：处理线程 pop 改 latest 跳帧**

理论：队列积压 3 帧，跳过 2 帧只处理最新帧，应该快 3 倍。

实际：FPS 无变化（不开算法 19fps，CLAHE+去雾 14fps）。

原因：

```
用 pop（逐帧处理）：
  处理线程一直在忙 → CPU 调度器给它稳定的时间片
  → 每帧墙钟时间稳定

用 latest（跳帧）：
  处理线程跳帧后队列空了 → msleep 等待 → CPU 给了其他线程
  → 新帧到来后要重新竞争时间片 → 墙钟时间反而更长

类比：
  食堂窗口（单核），三人排队（三线程）
  pop：你一直站在窗口前，一碗接一碗打 → 效率稳定
  latest：你说"跳过前两碗"然后离开 → 等新菜时要重新排队 → 总时间更长
```

### 4.2 真正有效的优化是减少 CPU 总负载

| 优化类型 | 例子 | 效果 | 原因 |
|---|---|---|---|
| 减少单环节工作量 | 跳帧、省颜色转换 | 效果被掩盖 | 省下的时间被其他线程分走 |
| **减少 CPU 总负载** | **P16 零拷贝（std::move）** | **FPS 14→22** | **三个线程都少了 memcpy 的负担** |
| 减少 CPU 总负载 | 降分辨率 640→320 | 算法耗时降 4-5x | 所有线程处理的数据量都变小 |

P16 零拷贝为什么有效：clone 是在队列的 push 方法里做的，采集线程 push captureQueue 时 clone 一次，处理线程 push displayQueue 时 clone 一次。改成 std::move 后，**两个线程都省了 memcpy**，CPU 总负载降低，所有线程都受益。

### 4.3 频率决定算力上限

```
cvtColor 处理 640×480 YUYV→BGR 需要约 5000 万条指令

650MHz Cortex-A7：
  每秒 6.5 亿条指令
  三线程各分 33% → 有效 2.2 亿条/秒
  5000万 ÷ 2.2亿 ≈ 23ms

1.5GHz Cortex-A53 四核：
  单核每秒 15 亿条指令 × 乱序执行 1.5x = 22.5 亿条/秒
  每个线程独占一个核，不需要分时
  5000万 ÷ 22.5亿 ≈ 2ms
```

同样的代码，A53 四核比 A7 单核快 10 倍以上。这不是软件能弥补的差距。

## 五、我们项目中的设计权衡

### 5.1 每个决策都有代价

| 决策 | 得到了什么 | 付出了什么 |
|---|---|---|
| 两线程→三线程 | UI 流畅、算法设计自由 | 多一个线程争 CPU，FPS 降 2-3 |
| QImage→cv::Mat 全程 BGR | 消除冗余颜色转换 | clone 开销导致 FPS 降 2-3（后被零拷贝修复） |
| 降分辨率 640→320 | 算法耗时降 4-5 倍 | 画质略有损失 |
| 实时流只跑 CLAHE+去雾 | FPS 可接受 | 锐化/降噪移到离线 |

### 5.2 队列设计的选择

| 位置 | 用 pop 还是 latest | 原因 |
|---|---|---|
| 处理线程从 captureQueue 取 | **pop**（逐帧） | 保持稳定的 CPU 时间片分配，跳帧在单核上无收益 |
| 主线程从 displayQueue 取 | **latest**（跳帧） | 减少显示延迟，让用户看到最新画面（不是为了省 CPU） |

### 5.3 尝试后撤回的优化

| 尝试 | 预期 | 实际 | 教训 |
|---|---|---|---|
| 手写 NEON 替换 cvtColor | 快 3-5 倍 | 反而慢 43-63% | OpenCV 已内置 NEON，不要重复造轮子 |
| 处理线程 pop→latest 跳帧 | 积压时快 3 倍 | FPS 无变化 | 单核上省下的时间被其他线程分走 |
| QImage::scaled→cv::resize | 更快 | 反而慢 | 格式转换开销 > 算法本身 |

**共同教训**：理论上合理的优化，在单核 CPU 上不一定有效。必须实测验证。

## 六、Cortex-A7 vs 实际产品芯片

| 参数 | Cortex-A7 (STM32MP157) | Cortex-A53 (i.MX8M) |
|---|---|---|
| 核心数 | 1 | 4 |
| 主频 | 650MHz | 1.5GHz |
| 流水线 | 顺序执行 | 乱序执行 |
| NEON | 单发射 | 双发射 |
| 内存带宽 | DDR3 ~2GB/s | DDR4 ~6GB/s |
| 等效算力差距 | 基准 | **约 10 倍** |

实际内窥镜产品的做法：
- 颜色转换由硬件 ISP 完成，零 CPU 开销
- 图像增强由专用硬件加速器或 FPGA 完成
- CPU 只负责 UI 和控制逻辑
- 每个线程独占一个核，不存在调度争抢

## 七、单核 vs 多核：哪些设计是通用的，哪些是单核特有的

### 7.1 多核上也需要的设计（通用）

以下设计与核心数无关，在任何平台上都需要：

| 设计 | 为什么多核也需要 | 我们项目中的体现 |
|---|---|---|
| 三线程流水线架构 | 采集、处理、显示职责分离是架构设计原则，不是性能补丁。即使 CPU 够快，算法和 UI 混在一个线程也会导致代码耦合、难维护 | 采集线程 + 处理线程 + 主线程 |
| 无锁环形队列 | 多核上线程是真并行，数据竞争更严重，无锁队列比加锁队列性能更好 | FrameQueue 用 atomic 保证线程安全 |
| 生产者-消费者模型 | 线程间解耦的标准模式，与核心数无关 | 采集 push、处理 pop/push、主线程 latest |
| 队列满时丢弃最旧帧 | 实时视频系统的通用策略，保证实时性比完整性重要 | captureQueue 满时覆盖最旧帧 |
| 主线程用 latest 取最新帧 | 实时预览场景永远只关心最新画面，与硬件无关 | displayQueue.latest() |
| 全程 BGR 数据格式 | 减少格式转换是架构设计原则，多核上同样有效（转换仍然耗 CPU） | 队列存 cv::Mat BGR，只在显示时转 RGB |
| 零拷贝（std::move） | 内存拷贝在任何平台都是开销，多核上内存带宽同样是瓶颈 | push(std::move(bgr)) |
| 降分辨率处理 | 减少计算量是通用优化手段 | 640→320 处理，算法耗时降 4-5x |
| 实时/离线分离 | 产品架构设计，与硬件无关 | 实时流 CLAHE+去雾，离线做锐化/降噪 |
| 硬件参数卸载给 Sensor | 能用硬件做的不用软件，这是嵌入式的基本原则 | 白平衡/曝光/对比度由 OV5640 硬件完成 |
| V4L2 采集框架 | Linux 摄像头采集的标准接口，与 CPU 架构无关 | mmap 零拷贝、DQBUF/QBUF 轮转 |
| 性能分析方法论 | 测量→定位瓶颈→优化→验证，任何项目都通用 | 各环节加计时、FPS 统计、日志分析 |

### 7.2 单核特有的问题（多核上不存在或大幅缓解）

| 单核问题 | 为什么单核有这个问题 | 多核怎么解决 |
|---|---|---|
| CPU 调度争抢导致墙钟时间膨胀 | 三个线程轮流用一个核，每个线程只分到 33% CPU | 每个线程绑定一个核（`pthread_setaffinity`），独享 100% CPU，不被打断 |
| 优化效果被调度波动掩盖 | 省下 10ms，但被其他线程分走 | 省下 10ms 就是真省 10ms，其他线程在其他核上，不会抢 |
| 跳帧优化无效 | 处理线程省下的时间被采集/主线程分走 | 处理线程省下的时间就是空闲，跳帧能直接提升 FPS |
| 三线程比两线程 FPS 更低 | 多一个线程争抢同一个核 | 多一个线程只是多用一个核，不影响其他线程 |
| Overrun（V4L2 buffer 归还不及时） | 采集线程被处理线程抢占 CPU，cvtColor 被打断 | 采集线程独占一个核，cvtColor 不会被打断，buffer 及时归还 |

### 7.3 多核上的典型做法

**默认行为（不绑核）**：

Linux 调度器自动把线程分配到空闲的核上，不需要改代码。我们的程序直接拿到四核平台上跑就能用。

但调度器可能会迁移线程（把线程从一个核挪到另一个核），迁移时 L1 cache 失效，导致性能波动。绑核可以避免这个问题。

```
不绑核时调度器可能的行为：

情况 A（还行）：
  核0: [采集线程]          ← 一个线程一个核
  核1: [处理线程]
  核2: [主线程]
  核3: [空闲]

情况 B（不理想）：
  核0: [采集线程][处理线程]  ← 两个线程挤在同一个核
  核1: [主线程]
  核2: [空闲]
  核3: [空闲]

情况 C（迁移）：
  核0: [采集线程]...[迁移到核2]...  ← cache 失效
  核1: [处理线程]
  核2: [采集线程]...[迁移回核0]...
  核3: [主线程]
```

**绑核方法**：

只需要一个工具函数 + 每个线程加一行调用：

```cpp
// 新增文件：utils/threadutil.h
#ifndef THREADUTIL_H
#define THREADUTIL_H

#include <pthread.h>
#include <sched.h>

inline void bindToCore(int coreId) {
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);          // 清空集合
    CPU_SET(coreId, &cpuset);   // 添加指定核
    pthread_setaffinity_np(pthread_self(), sizeof(cpuset), &cpuset);
}

#endif
```

```cpp
// v4l2capture.cpp — run() 开头加一行
#include "utils/threadutil.h"
void V4l2Capture::run() {
    bindToCore(0);    // 采集线程绑定到核 0
    if (!startStreaming()) return;
    // ... 其余代码完全不变 ...
}

// processthread.h — run() 开头加一行
#include "utils/threadutil.h"
void run() override {
    bindToCore(1);    // 处理线程绑定到核 1
    m_running = true;
    // ... 其余代码完全不变 ...
}

// main.cpp — 主线程绑定到核 2
#include "utils/threadutil.h"
int main(int argc, char *argv[]) {
    bindToCore(2);
    QApplication app(argc, argv);
    // ... 其余代码完全不变 ...
}
```

绑核后的效果：

```
核 0: [采集线程 独占]     ← cvtColor 不被打断，稳定 15ms
核 1: [处理线程 独占]     ← CLAHE+去雾 不被打断，稳定 115ms
核 2: [主线程 独占]       ← UI 始终流畅
核 3: [系统服务]           ← WiFi、内核线程等
```

架构、队列、数据格式、算法管线全部复用，零修改。只加了一个工具函数 + 三行调用。

### 7.4 多核上的预期 FPS

| 场景 | 单核 A7 650MHz | 四核 A53 1.5GHz（预估） |
|---|---|---|
| 不开算法 | 21-22fps | **30fps**（采集帧率上限） |
| 开 CLAHE | ~13fps | **28-30fps**（CLAHE ~3ms） |
| 开 CLAHE+去雾 | ~5fps | **20-25fps**（去雾 ~10ms） |

提升来源：
- 频率高 2.3x（650MHz → 1.5GHz）
- 乱序执行效率 ~1.5x
- 线程独占核心，无调度争抢（最大的差异）
- NEON 双发射（A7 单发射）

```
我们项目中学到的能力：

  ✔ 三线程流水线架构

  ✔ V4L2 采集框架
  ✔ 全程 BGR 数据格式
  ✔ 零拷贝优化
  ✔ 实时/离线分离架构
  ✔ 性能分析方法论

单核特有，多核上不需要操心：
  ✘ CPU 调度波动分析
  ✘ “省下的时间被其他线程分走”的问题
  ✘ 为了省 CPU 而做的妃协（如降分辨率、简化算法）

多核上新增的工作：
  ○ 线程绑核（pthread_setaffinity）
  ○ cache 一致性问题（多核共享内存时）
  ○ 更复杂的内存序问题（我们用的 memory_order 在多核上更重要）
```
