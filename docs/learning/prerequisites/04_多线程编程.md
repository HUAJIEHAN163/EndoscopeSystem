# 04 多线程编程

## 一、为什么需要多线程

### 当前问题

```
采集线程          主线程
[采集帧]          [processFrame 算法处理 44ms]  ← 阻塞
[采集帧]          [还在处理...]                  ← UI 卡死
[采集帧] 丢弃     [还在处理...]                  ← 帧丢失
[采集帧] 丢弃     [处理完了，paintEvent 显示]
```

算法在主线程跑，阻塞了 UI 刷新和按钮响应。

### 目标架构

```
采集线程          处理线程          主线程
[采集帧] ──→     [算法处理] ──→   [显示]
[采集帧] ──→     [算法处理] ──→   [显示]
[采集帧] ──→     [算法处理] ──→   [显示]
三个线程并行，互不阻塞
```

## 二、线程基础

### 2.1 QThread 的使用方式

```cpp
// 方式 1：继承 QThread，重写 run()（我们项目的方式）
class V4l2Capture : public QThread {
protected:
    void run() override {
        while (!m_quit) {
            // 采集循环
        }
    }
};

// 启动
V4l2Capture *capture = new V4l2Capture();
capture->start();  // 创建新线程，在新线程中执行 run()

// 停止
capture->stop();   // 设置退出标志
capture->wait();   // 等待线程退出
```

### 2.2 线程间通信

**信号槽（当前方式）：**

```cpp
// 采集线程 emit → Qt 事件队列 → 主线程处理
connect(source, &VideoSource::frameReady,
        this, &MainWindow::onFrameReady,
        Qt::QueuedConnection);  // 跨线程必须用 QueuedConnection
```

优点：安全，Qt 自动处理线程切换
缺点：信号队列可能堆积（OOM 问题 #11）

**共享变量（P3 目标方式）：**

```cpp
// 采集线程写入环形队列，处理线程读取
// 不经过 Qt 信号槽，直接共享内存
ringBuffer.push(frame);   // 采集线程
ringBuffer.pop(frame);    // 处理线程
```

优点：无队列堆积，延迟低
缺点：需要自己处理线程安全

## 三、线程安全

### 3.1 竞态条件

多个线程同时读写同一个变量，结果不确定：

```cpp
// 危险：两个线程同时操作 count
int count = 0;

// 线程 A          // 线程 B
count++;           count++;
// 期望 count = 2，实际可能 = 1

// count++ 不是原子操作，实际是三步：
// 1. 读取 count 的值
// 2. 加 1
// 3. 写回 count
// 两个线程可能同时读到 0，各自加 1 写回 1
```

### 3.2 atomic — 原子变量

```cpp
#include <atomic>

std::atomic<int> count{0};

// 线程 A              // 线程 B
count.fetch_add(1);    count.fetch_add(1);
// 保证 count = 2，原子操作不会被打断
```

项目中的使用：
- `std::atomic<bool> m_running` — 控制线程暂停/恢复
- `std::atomic<bool> m_quit` — 控制线程退出
- `std::atomic<int> m_pendingFrames` — 信号队列计数

适用场景：简单的计数器、标志位。复杂的数据结构不能只靠 atomic。

### 3.3 mutex — 互斥锁

```cpp
#include <mutex>

std::mutex mtx;
std::vector<int> data;

// 线程 A
{
    std::lock_guard<std::mutex> lock(mtx);  // 加锁
    data.push_back(1);
}  // 离开作用域自动解锁

// 线程 B
{
    std::lock_guard<std::mutex> lock(mtx);  // 等线程 A 解锁后才能进入
    data.push_back(2);
}
```

`lock_guard` 是 RAII 风格：构造时加锁，析构时解锁，不会忘记解锁。

缺点：加锁有开销（~1μs），高频操作（每帧 30 次）会影响性能。

### 3.4 无锁设计 — 环形队列

对于单生产者单消费者的场景，可以用无锁环形队列：

```cpp
template<int N>
class RingBuffer {
    cv::Mat frames[N];           // 固定大小数组
    std::atomic<int> writeIdx{0}; // 写索引（只有生产者修改）
    std::atomic<int> readIdx{0};  // 读索引（只有消费者修改）

public:
    bool push(const cv::Mat& frame) {
        int next = (writeIdx.load() + 1) % N;
        if (next == readIdx.load()) return false;  // 队列满
        frames[writeIdx.load()] = frame.clone();
        writeIdx.store(next);
        return true;
    }

    bool pop(cv::Mat& frame) {
        if (readIdx.load() == writeIdx.load()) return false;  // 队列空
        frame = frames[readIdx.load()];
        readIdx.store((readIdx.load() + 1) % N);
        return true;
    }
};
```

为什么无锁：
- writeIdx 只有生产者写，消费者只读
- readIdx 只有消费者写，生产者只读
- 不存在两个线程同时写同一个变量的情况
- 用 atomic 保证可见性即可

## 四、生产者-消费者模型

### 4.1 概念

```
生产者（采集线程）→ [缓冲区] → 消费者（处理线程）

生产者不断往缓冲区放数据
消费者不断从缓冲区取数据
缓冲区满时：生产者等待或丢弃最旧数据
缓冲区空时：消费者等待
```

### 4.2 P3 的三级流水线

```
采集线程（生产者）→ [环形队列 A] → 处理线程（消费者/生产者）→ [环形队列 B] → 主线程（消费者）

采集线程：grabFrame → cvtColor → push 到队列 A
处理线程：pop 从队列 A → CLAHE/去雾 → push 到队列 B
主线程：  pop 从队列 B → paintEvent 显示
```

### 4.3 队列满时的策略

```
策略 1：阻塞等待（不适合实时系统）
  生产者等消费者取走数据再放入
  → 采集线程被阻塞，可能导致 V4L2 overrun

策略 2：丢弃最新帧（当前 m_pendingFrames 的方式）
  队列满时不放入，跳过这帧
  → 简单，但丢弃的是最新数据

策略 3：丢弃最旧帧（推荐）
  队列满时覆盖最旧的帧
  → 保证显示的始终是最新数据，延迟最低
```

## 五、线程优先级

### 5.1 为什么需要

Cortex-A7 只有 1 个核心（单核），三个线程共享 CPU 时间。默认调度器可能让处理线程抢占采集线程，导致 overrun。

### 5.2 设置方法

```cpp
#include <pthread.h>

// 在采集线程的 run() 开头设置高优先级
struct sched_param param;
param.sched_priority = 80;  // 0-99，越大优先级越高
pthread_setschedparam(pthread_self(), SCHED_FIFO, &param);
```

`SCHED_FIFO`：实时调度策略，高优先级线程不会被低优先级线程抢占。

### 5.3 优先级分配

```
采集线程：最高（80）— 必须及时归还 V4L2 buffer，否则 overrun
处理线程：中等（50）— 算法处理，可以慢一点
主线程：  默认（0） — UI 刷新，最不紧急
```

## 六、常见陷阱

### 6.1 死锁

```cpp
// 线程 A              // 线程 B
lock(mutexA);          lock(mutexB);
lock(mutexB);  // 等B  lock(mutexA);  // 等A
// 互相等待，永远不会结束
```

避免：始终按相同顺序加锁，或用 `std::lock` 同时锁多个。

### 6.2 数据竞争

```cpp
// 危险：一个线程写，另一个线程读，没有同步
std::string msg;
// 线程 A: msg = "hello";
// 线程 B: std::cout << msg;  // 可能读到半写入的数据
```

避免：用 atomic、mutex 或无锁队列。

### 6.3 线程退出时机

```cpp
// 我们遇到的问题 #16：
stop();    // 设置退出标志
start();   // 线程还没退出就重启 → 失败

// 正确做法：
stop();
wait();    // 等待线程完全退出
start();   // 再重启

// 更好的做法（当前方案）：
pause();   // 不杀线程，只暂停处理
resume();  // 恢复处理，无需重启
```

## 七、标准库的线程写法
用 C++ 标准库的 std::thread，不依赖任何框架：

#include <thread>
#include <atomic>

class MyThread {
public:
    void start() {
        m_quit = false;
        m_thread = std::thread(&MyThread::run, this);  // 创建线程，调用 run()
    }

    void stop() {
        m_quit = true;
        if (m_thread.joinable())
            m_thread.join();    // 等待线程结束（等价于 QThread::wait）
    }

protected:
    virtual void run() {
        while (!m_quit) {
            // 线程主循环，子类重写这里
        }
    }

private:
    std::thread m_thread;
    std::atomic<bool> m_quit{false};
};


---
使用方式和 QThread 几乎一样：

// QThread 方式（我们项目）
class V4l2Capture : public VideoSource {  // VideoSource 继承 QThread
    void run() override { /* 采集循环 */ }
};
V4l2Capture cap;
cap.start();   // QThread::start → 内部创建线程 → 调用 run()
cap.stop();
cap.wait();    // 等待线程结束

// std::thread 方式（无 Qt）
class V4l2Capture : public MyThread {
    void run() override { /* 采集循环 */ }
};
V4l2Capture cap;
cap.start();   // std::thread 创建线程 → 调用 run()
cap.stop();    // 内部 join 等待线程结束
---

对比：

QThread	std::thread
头文件	<QThread>	<thread>
启动	start()	std::thread(&run, this)
等待结束	wait()	join()
线程安全变量	同样用 std::atomic	同样用 std::atomic
信号槽	有（跨线程通信方便）	没有（用队列或回调替代）
依赖	Qt 库	C++11 标准库，无依赖
核心逻辑完全一样——创建线程、跑 run 循环、用 atomic 控制退出。QThread 只是多了信号槽的便利，底层都是 pthread。
