# 双核资源共享与调试前置知识

## 一、STM32MP157 双核资源共享

### 1.1 AHB 总线架构

STM32MP157 的 A7 和 M4 共享 AHB 总线访问外设和内存。当两个核心同时访问总线时会产生竞争：

```
A7 (Cortex-A7)                    M4 (Cortex-M4)
     │                                  │
     ├── DCMI DMA（摄像头数据传输）       ├── GPIO 寄存器（软件 I2C / PWM）
     ├── 内存访问                        ├── SysTick 寄存器（HAL_Delay）
     ├── SDIO（WiFi/SD卡）              ├── SRAM3（OpenAMP 共享内存）
     │                                  │
     └──────────── AHB 总线 ────────────┘
                  （共享，有仲裁器）
```

**实际遇到的问题**：M4 的 `HAL_Delay(20ms)` 忙等待期间持续读取 SysTick 寄存器，加上软件 I2C 的 GPIO bit-bang，密集占用总线 20ms，导致 A7 的 DCMI DMA 传输被延迟，产生 overrun。

**解决方法**：在 FreeRTOS 环境下用 `vTaskDelay()` 替代 `HAL_Delay()`，让 M4 在等待期间进入空闲（WFI 指令），不访问总线。

### 1.2 HAL_Delay vs vTaskDelay

| | HAL_Delay | vTaskDelay |
|---|---|---|
| 实现 | while 循环读 SysTick | 将任务挂起，调度器切换 |
| CPU 占用 | 100%（忙等待） | 0%（进入空闲任务） |
| 总线访问 | 持续访问 SysTick 寄存器 | WFI 指令，不访问总线 |
| 精度 | 1ms（SysTick 精度） | 取决于 configTICK_RATE_HZ |
| 适用场景 | 调度器启动前、中断中 | FreeRTOS 任务中 |

### 1.3 外设冲突

STM32MP157 的外设由 A7 和 M4 共享，需要注意：

| 冲突类型 | 示例 | 后果 | 解决方法 |
|---|---|---|---|
| 定时器冲突 | TIM2 被 LCD 背光占用，M4 初始化 TIM2 | LCD 熄灭 | M4 不使用 TIM2 |
| I2C 冲突 | I2C1 被触摸屏占用，M4 初始化 I2C1 | 触摸屏失效 | M4 用软件 I2C |
| GPIO 冲突 | GPIO 被 A7 设备树占用 | M4 无法控制 | 禁用 A7 侧 overlay |
| 总线竞争 | M4 忙等待占用总线 | DCMI DMA overrun | vTaskDelay 让出总线 |

确认外设是否被 A7 占用的方法见「Linux 调试命令速查」文档。

## 二、CubeMX 代码生成与手动修复

### 2.1 CubeMX 的 USER CODE 机制

CubeMX 生成代码时会保留 `USER CODE BEGIN` 和 `USER CODE END` 之间的用户代码，但会覆盖其他区域。

```c
/* USER CODE BEGIN 2 */
// 这里的代码不会被覆盖
HAL_TIM_Base_Start_IT(&htim3);
/* USER CODE END 2 */
```

### 2.2 每次生成后必须手动修复的项目

本项目中 CubeMX 每次生成代码后需要修复 6 个地方（详见总览文档）：

| # | 文件 | 修改内容 | 原因 |
|---|---|---|---|
| 1 | openamp_conf.h | 取消注释 VIRTUAL_UART_MODULE_ENABLED | CubeMX 默认注释掉 |
| 2 | virt_uart.c | service name 改为 "rpmsg-tty-channel" | CubeMX 生成 "rpmsg-tty"，与野火内核不匹配 |
| 3 | sys.c | 实现 sys_irq_save_disable（PRIMASK） | CubeMX 生成空实现 |
| 4 | main.c | cmsis_os.h → FreeRTOS.h + task.h | CMSIS_V1 不生成 cmsis_os.h |
| 5 | main.c | osThreadDef → xTaskCreate + vTaskStartScheduler | 同上 |
| 6 | main.c | TIM3 ARR 从 99 改为 49 | CubeMX 中未修改此参数 |

### 2.3 .cproject 中的关键配置

CubeMX 通常不会覆盖 .cproject，但每次生成后应确认：
- 链接器参数：`-Wl,-z,max-page-size=0x10000`（GCC 13 兼容性）
- 编译宏：`METAL_FREERTOS`（libmetal FreeRTOS 后端）

## 三、软件 I2C（GPIO Bit-Bang）

### 3.1 为什么用软件 I2C

当硬件 I2C 引脚被其他外设占用时（如本项目 I2C1 被触摸屏占用），可以用任意 GPIO 模拟 I2C 协议。

### 3.2 实现要点

```c
// GPIO 配置：Open Drain + Pull-up
GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_OD;
GPIO_InitStruct.Pull = GPIO_PULLUP;

// 时序控制：延时函数
static void soft_i2c_delay(void) {
    for (volatile int i = 0; i < 50; i++);  // ~5us @64MHz
}

// I2C 起始条件：SDA 先拉低，SCL 再拉低
static void soft_i2c_start(void) {
    SDA_HIGH(); SCL_HIGH(); soft_i2c_delay();
    SDA_LOW(); soft_i2c_delay();
    SCL_LOW(); soft_i2c_delay();
}
```

### 3.3 注意事项

- Open Drain 模式：I2C 要求线与（wired-AND），必须用开漏输出 + 上拉
- 延时精度：软件延时受 CPU 频率影响，换平台需要重新校准
- 总线占用：bit-bang 期间 CPU 持续访问 GPIO 寄存器，会占用总线（见 1.1）

## 四、软件 PWM vs 硬件 PWM

### 4.1 对比

| | 硬件 PWM | 软件 PWM |
|---|---|---|
| 实现 | 定时器自动输出 | 定时器中断中 GPIO 翻转 |
| CPU 占用 | 0%（硬件自动） | 每次中断消耗少量 CPU |
| 频率上限 | MHz 级 | 受中断频率限制 |
| 引脚限制 | 必须是定时器通道引脚 | 任意 GPIO |
| 精度 | 硬件精确 | 受中断延迟影响 |

### 4.2 本项目的选择

LED 补光灯原计划用 TIM2_CH3 硬件 PWM（PB10），但 TIM2 与 LCD 背光冲突。尝试 TIM4_CH1（PB6）但引脚物理位置不可用。最终采用 PB10 + TIM3 中断软件 PWM（500Hz，40 级调光）。

## 五、remoteproc 固件加载

### 5.1 加载流程

```bash
# 1. 复制 ELF 到 firmware 目录
cp firmware.elf /lib/firmware/

# 2. 指定固件文件名
echo firmware.elf > /sys/class/remoteproc/remoteproc0/firmware

# 3. 启动 M4
echo start > /sys/class/remoteproc/remoteproc0/state

# 4. 确认状态
cat /sys/class/remoteproc/remoteproc0/state  # 输出: running
```

### 5.2 注意事项

- **每次更新固件必须重启开发板**：stop/start 会破坏 RPMsg 数据通道
- **ELF segment alignment**：GCC 13 需要 `-Wl,-z,max-page-size=0x10000`，否则 Linux 找不到 resource table
- **RPMsg service name**：必须与内核驱动匹配（野火内核用 "rpmsg-tty-channel"）
