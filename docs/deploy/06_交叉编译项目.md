# Step 6：交叉编译项目 [✅ 已完成]

## 这一步在做什么

用交叉编译器在虚拟机上编译内窥镜项目，生成 ARM 可执行文件，然后传到开发板运行。

```
虚拟机上编译：
  源代码 + 交叉编译工具链 + OpenCV ARM + Qt5 ARM
      ↓ cmake + make
  build_arm/endoscope（ELF 32-bit ARM 可执行文件）
      ↓ NFS
  开发板运行
```

---

## 正常情况 vs 我们的特殊情况

### 正常情况（厂商提供配套 SDK）

芯片厂商（ST、瑞芯微、海思等）通常提供完整的 SDK，包含：
- 配套的交叉编译工具链（glibc 版本与开发板一致）
- 配套的 sysroot（头文件 + 库文件，路径正确）
- 预编译的 Qt5 / OpenCV ARM 版（与工具链 glibc 版本匹配）

正常的编译命令只需要 3 个参数：

```bash
# 正常情况：工具链和库配套，不需要额外的 sysroot 和 rpath-link
mkdir build_arm && cd build_arm
cmake .. -DCMAKE_TOOLCHAIN_FILE=toolchain.cmake \
         -DQt5_DIR=/path/to/qt5_arm/lib/cmake/Qt5 \
         -DOpenCV_DIR=/path/to/opencv_arm/lib/cmake/opencv4
make -j$(nproc)
# 一次通过，不会有任何 glibc 兼容问题
```

### 我们的特殊情况（工具链和库不配套）

我们的环境是拼凑的，glibc 版本不一致：

```
Linaro gcc 7.5.0 → sysroot glibc 2.25
野火预编译 Qt5   → 依赖 glibc 2.28
自编译 OpenCV    → 依赖 glibc 2.25
开发板 Debian 10 → glibc 2.28
```

Qt5 需要 glibc 2.28 的符号（renameat2、statx、fcntl），但工具链的 sysroot 只有 glibc 2.25，链接时找不到这些符号。

**解决方案**：从开发板复制完整的 sysroot（glibc 2.28），替代工具链自带的 sysroot。

### cmake 命令在做什么

cmake 本身不编译，只是**配置编译环境**，告诉后续的 `make` 用什么工具、去哪里找库。运行完后会在当前目录生成 `Makefile`。

逐行解释：

- `cmake /mnt/hgfs/VM_Share/project/EndoscopeSystem` — 指定源代码在哪里（CMakeLists.txt 的位置）
- `-DCMAKE_TOOLCHAIN_FILE=...arm_toolchain.cmake` — 告诉 cmake 不要用本机 gcc，改用 Linaro 交叉编译器
- `-DQt5_DIR=...` — 告诉 cmake Qt5 的 ARM 版在哪里，否则会找到 x86 版本
- `-DOpenCV_DIR=...` — 同上，指定 OpenCV ARM 版位置
- `-DCMAKE_SYSROOT=...board_sysroot` — 系统库（glibc）去这个目录找，用开发板的 glibc 2.28 替代工具链自带的 2.25
- `-DCMAKE_EXE_LINKER_FLAGS=...` — 额外告诉链接器再去这几个目录搜索库文件，防止找不到 Qt5 和 glibc 的依赖

### make 怎么知道要编译哪些文件

cmake 读取项目里的 `CMakeLists.txt`，生成 `Makefile`，`make` 再读 `Makefile` 执行编译。

```
CMakeLists.txt         cmake 读取它
    ↓
cmake 运行             生成 Makefile
    ↓
Makefile               make 读取它
    ↓
make 运行              按顺序编译所有文件，生成 endoscope
```

`CMakeLists.txt` 里的 `SOURCES` 列出了所有要编译的文件：

```cmake
set(SOURCES
    src/main.cpp
    src/ui/mainwindow.cpp
    src/capture/filecapture.cpp
    src/processing/imageprocessor.cpp
    ...)
```

**新增 .cpp 文件时必须同步更新 CMakeLists.txt**，否则 cmake/make 不知道要编译它。

### make -j$(nproc) 是什么意思

- `make` — 执行编译
- `-j` — 允许并行编译，同时跑多个编译任务
- `$(nproc)` — shell 命令，自动获取当前机器的 CPU 核心数

比如虚拟机分配了 4 个核心，等价于 `make -j4`，同时编译 4 个文件，速度约是单核的 4 倍。不加 `-j` 默认单核串行，编译大项目会非常慢。

### CMakeLists.txt 是手写的吗

是手写的，在**项目创建之初**就要写好，是整个项目的构建配置文件。内容根据项目实际情况填写：用了哪些库、有哪些源文件、编译出什么程序、需要链接哪些库。后续每次新增 .cpp 文件都需要手动更新。

### 参数说明

| 参数 | 含义 | 通用/特殊 |
|------|------|----------|
| `-DCMAKE_TOOLCHAIN_FILE` | 指定交叉编译工具链文件 | 通用（任何交叉编译都需要） |
| `-DQt5_DIR` | Qt5 ARM 版的 CMake 配置路径 | 通用 |
| `-DOpenCV_DIR` | OpenCV ARM 版的 CMake 配置路径 | 通用 |
| `-DCMAKE_SYSROOT` | 开发板的 sysroot（glibc 2.28） | ⚠️ 特殊（工具链配套时不需要） |
| `-DCMAKE_EXE_LINKER_FLAGS` | 链接器搜索路径 | ⚠️ 特殊（工具链配套时不需要） |
| `-j$(nproc)` | 使用所有 CPU 核心并行编译 | 通用 |

## 6.2 [特殊] 开发板 sysroot 搭建过程

> 这一整节是因为 glibc 版本不匹配才需要的。如果厂商提供配套 SDK，跳过此节。

### 为什么需要 sysroot

```
链接时报错：
  libQt5Core.so: undefined reference to `renameat2@GLIBC_2.28'
  libQt5Core.so: undefined reference to `statx@GLIBC_2.28'
  libQt5Core.so: undefined reference to `fcntl@GLIBC_2.28'

原因：
  Qt5 编译时用的 glibc 2.28，里面调用了 2.28 的新函数
  Linaro 工具链的 sysroot 只有 glibc 2.25，没有这些函数
  链接器在 sysroot 里找不到，报 undefined reference

解决：
  用开发板的 glibc 2.28 替代工具链的 glibc 2.25
```

### 搭建步骤

```bash
# 1. 创建 sysroot 目录结构
mkdir -p /home/chris/board_sysroot/lib/arm-linux-gnueabihf
mkdir -p /home/chris/board_sysroot/usr/lib/arm-linux-gnueabihf
mkdir -p /home/chris/board_sysroot/usr/include

# 2. 开发板上安装开发包（提供 crt1.o 等链接文件）
# 开发板 root 终端执行：
apt install libc6-dev linux-libc-dev -y

# 3. 开发板上打包
cd /lib/arm-linux-gnueabihf && tar czf /tmp/lib_arm.tar.gz *.so* ld-*
cd /usr/lib/arm-linux-gnueabihf && tar czf /tmp/usr_lib.tar.gz *.a *.so *.o
tar czf /tmp/usr_include_all.tar.gz -C /usr include

# 4. 虚拟机上拉取并解压
scp debian@192.168.0.43:/tmp/lib_arm.tar.gz /home/chris/
scp debian@192.168.0.43:/tmp/usr_lib.tar.gz /home/chris/
scp debian@192.168.0.43:/tmp/usr_include_all.tar.gz /home/chris/

tar xzf /home/chris/lib_arm.tar.gz -C /home/chris/board_sysroot/lib/arm-linux-gnueabihf/
tar xzf /home/chris/usr_lib.tar.gz -C /home/chris/board_sysroot/usr/lib/arm-linux-gnueabihf/
tar xzf /home/chris/usr_include_all.tar.gz -C /home/chris/board_sysroot/usr/

# 5. 修复链接脚本和符号链接中的绝对路径
cd /home/chris/board_sysroot/usr/lib/arm-linux-gnueabihf

# libc.so 是链接脚本，绝对路径改相对路径
cat > libc.so << 'EOF'
OUTPUT_FORMAT(elf32-littlearm)
GROUP ( ../../../lib/arm-linux-gnueabihf/libc.so.6 libc_nonshared.a  AS_NEEDED ( ../../../lib/arm-linux-gnueabihf/ld-linux-armhf.so.3 ) )
EOF

# 修复断开的符号链接
rm -f libdl.so librt.so libm.so
ln -s ../../../lib/arm-linux-gnueabihf/libdl.so.2 libdl.so
ln -s ../../../lib/arm-linux-gnueabihf/librt.so.1 librt.so
ln -s ../../../lib/arm-linux-gnueabihf/libm.so.6 libm.so
```

### 为什么要修复绝对路径

sysroot 里的 `libc.so` 是一个链接脚本（文本文件），内容是：

```
# 开发板上的原始内容（绝对路径）
GROUP ( /lib/arm-linux-gnueabihf/libc.so.6 ... )
```

当使用 `--sysroot=/home/chris/board_sysroot` 时，链接器会在 sysroot 内部查找 `/lib/arm-linux-gnueabihf/libc.so.6`，实际路径变成 `/home/chris/board_sysroot/lib/arm-linux-gnueabihf/libc.so.6`。但链接脚本里的绝对路径不会自动加 sysroot 前缀，所以找不到。改成相对路径就能正确解析。

符号链接同理：`libm.so → /lib/arm-linux-gnueabihf/libm.so.6` 是绝对路径，在 sysroot 环境下断开了。

## 6.4 踩过的坑

### 通用问题（任何项目都可能遇到）

| 问题 | 原因 | 解决 |
|------|------|------|
| 文件被截断 | 在 VMware 共享目录下编译，I/O 不稳定 | 改在虚拟机本地目录编译 |
| Qt5/OpenCV 找不到 | DIR 路径不对 | 确认 cmake 配置路径存在 |

### 特殊问题（因为工具链和库不配套）

| 问题 | 原因 | 解决 | 正常 SDK 会遇到吗 |
|------|------|------|------------------|
| GLIBC_2.28 undefined | Qt5 需要 glibc 2.28，工具链只有 2.25 | 使用开发板 sysroot | ❌ 不会 |
| crt1.o 找不到 | 开发板没装 libc6-dev | apt install libc6-dev | ❌ 不会（SDK 自带） |
| features.h 找不到 | sysroot 缺少头文件 | 复制开发板 /usr/include | ❌ 不会（SDK 自带） |
| libc_nonshared.a 找不到 | sysroot 缺少静态库 | 复制开发板 /usr/lib 下 .a 文件 | ❌ 不会（SDK 自带） |
| libpthread.a 链接报错 | libc.so 链接脚本里是绝对路径 | 改为相对路径 | ❌ 不会（SDK 路径正确） |
| libm.so 找不到 | 符号链接指向绝对路径，断开了 | 重建为相对路径符号链接 | ❌ 不会（SDK 路径正确） |

## 6.6 经验总结

这次交叉编译折腾的根本原因是**工具链和库不配套**。实际工作中的建议：

1. **优先使用厂商提供的完整 SDK**，工具链、sysroot、库全部配套，一次编译通过
2. **如果必须拼凑**，确保所有组件的 glibc 版本一致
3. **搭建 sysroot 时**，从开发板复制完整的 /lib、/usr/lib、/usr/include，不要一个一个文件补
4. **注意链接脚本和符号链接中的绝对路径**，在 sysroot 环境下需要改成相对路径
