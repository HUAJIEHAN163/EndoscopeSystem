## 问题 #1：bilateralFilter 崩溃

**现象**
```
error: (-215:Assertion failed) src.data != dst.data in function 'bilateralFilter_8u'
terminate called after throwing an instance of 'cv::Exception'
已中止 (核心已转储)
```

**原因**

`ImageProcessor::process` 中所有算法都以 `dst, dst` 方式调用（输入输出同一个 Mat）。OpenCV 的 `bilateralFilter` 不允许原地操作（src 和 dst 不能指向同一块内存）。

**修复**

`src/processing/imageprocessor.cpp` 中 `applyDenoise` 加临时变量中转：

```cpp
// 修复前
cv::bilateralFilter(src, dst, d, sigmaColor, sigmaSpace);

// 修复后
cv::Mat tmp;
cv::bilateralFilter(src, tmp, d, sigmaColor, sigmaSpace);
dst = tmp;
```

**文件**: `src/processing/imageprocessor.cpp`

## 问题 #3：临床模式切换后画面严重卡顿

**来源**：`11_开发记录_调试临床模式切换` 开发过程中

**现象**

切换到临床模式后，预设参数同时开启白平衡+CLAHE+去雾+锐化+降噪，帧率从 30fps 骤降到个位数，画面明显卡顿。

**原因分析**

两个性能瓶颈：

1. **去雾算法（applyDehaze）**：使用 `CV_64FC3`（双精度浮点），每帧需要多次 split/merge/erode，计算量极大
2. **降噪算法（applyDenoise）**：`bilateralFilter` 是 OpenCV 中最慢的滤波器之一，768×576 分辨率下每帧耗时约 30-50ms

**修复**

文件：`src/processing/imageprocessor.cpp`

### 修复 1：去雾算法 — 双精度改单精度 + 减少冗余操作

```cpp
// ===== 修复前 =====
void ImageProcessor::applyDehaze(const cv::Mat &src, cv::Mat &dst,
                                 double omega, int radius) {
    cv::Mat srcFloat;
    src.convertTo(srcFloat, CV_64FC3, 1.0 / 255.0);  // 双精度，慢

    std::vector<cv::Mat> channels;
    cv::split(srcFloat, channels);
    cv::Mat darkChannel = cv::min(cv::min(channels[0], channels[1]), channels[2]);
    // ... 省略中间 ...

    // 透射率计算：多余的 split/merge/split
    std::vector<cv::Mat> normChannels;
    cv::split(srcFloat, normChannels);
    for (int i = 0; i < 3; i++)
        normChannels[i] /= (A[i] + 1e-6);

    cv::Mat normImg;
    cv::merge(normChannels, normImg);     // 多余的 merge
    cv::split(normImg, normChannels);     // 又 split 回来

    cv::Mat transmission = cv::min(cv::min(normChannels[0], normChannels[1]),
                                   normChannels[2]);
    // ...
}

// ===== 修复后 =====
void ImageProcessor::applyDehaze(const cv::Mat &src, cv::Mat &dst,
                                 double omega, int radius) {
    cv::Mat srcFloat;
    src.convertTo(srcFloat, CV_32FC3, 1.0f / 255.0f);  // 单精度，速度翻倍

    std::vector<cv::Mat> channels;
    cv::split(srcFloat, channels);
    cv::Mat darkChannel;
    cv::min(channels[0], channels[1], darkChannel);     // 避免临时对象
    cv::min(darkChannel, channels[2], darkChannel);
    // ... 省略中间 ...

    // 透射率计算：直接 divide，省去多余的 merge/split
    cv::Mat normMin;
    cv::Mat c0, c1, c2;
    cv::divide(channels[0], A[0] + 1e-6, c0);
    cv::divide(channels[1], A[1] + 1e-6, c1);
    cv::divide(channels[2], A[2] + 1e-6, c2);
    cv::min(c0, c1, normMin);
    cv::min(normMin, c2, normMin);
    // ...
}
```

优化点：
- `CV_64FC3` → `CV_32FC3`：单精度浮点运算速度约为双精度的 2 倍
- 去掉透射率计算中多余的 `merge` + `split`，直接用 `cv::divide` 逐通道计算
- `cv::min` 用两参数版本避免创建临时 Mat 对象

### 修复 2：降噪算法 — 缩小后处理再放大

```cpp
// ===== 修复前 =====
void ImageProcessor::applyDenoise(const cv::Mat &src, cv::Mat &dst,
                                  int d, double sigmaColor,
                                  double sigmaSpace) {
    cv::Mat tmp;
    cv::bilateralFilter(src, tmp, d, sigmaColor, sigmaSpace);  // 全分辨率处理，非常慢
    dst = tmp;
}

// ===== 修复后 =====
void ImageProcessor::applyDenoise(const cv::Mat &src, cv::Mat &dst,
                                  int d, double sigmaColor,
                                  double sigmaSpace) {
    cv::Mat small, tmp;
    cv::resize(src, small, cv::Size(), 0.5, 0.5, cv::INTER_LINEAR);  // 缩小到 50%
    cv::bilateralFilter(small, tmp, d, sigmaColor, sigmaSpace);       // 在小图上滤波
    cv::resize(tmp, dst, src.size(), 0, 0, cv::INTER_LINEAR);        // 放大回原尺寸
}
```

优化点：
- 像素量减少 75%（768×576 → 384×288），双边滤波速度提升约 4 倍
- 降噪本身就是模糊操作，缩小再放大的视觉差异很小

**效果**

临床模式下帧率从个位数恢复到可用水平，画面流畅度明显改善。

> → FPS 问题的完整跟踪见 `16_FPS性能问题总跟踪.md`

## 问题 #5：冻结画面不清晰（运动模糊）

**来源**：日常测试中发现

**现象**

按下冻结后，显示的帧呈现运动模糊，不够清晰。

**原因**

原实现直接停在最后一帧，而视频流中每帧都带有不同程度的运动模糊（曝光时间内画面移动导致）。最后一帧不一定是最清晰的。

**修复**

缓存最近 8 帧，冻结时用拉普拉斯方差评估每帧清晰度，选得分最高的帧显示。

详见：`docs/12_开发记录_冻结帧清晰度优化.md`

**文件**：`src/processing/imageprocessor.h/cpp`、`src/ui/mainwindow.h/cpp`

## 问题 #7：开发板上移动摄像头时局部花屏

**来源**：开发板实际运行测试

**现象**

移动摄像头时，画面局部出现花屏（擕裂），感觉是刷新没跟上。

**初始猜想与排查**

| 猜想 | 排查结果 |
|---|---|
| 画面擕裂（tearing） | ✘ 不是，tearing 是水平分界线，实际是随机亮线条 |
| V4L2 buffer 覆盖 | ✔ **属于这个范畴**，但具体机制是归还时机错误，不是 buffer 数量不足 |
| YUYV 转换错误 | ✘ 静止时正常，排除 |
| 带宽不足 | ✘ 本地运行也花，排除 |
| NFS 运行导致 | ✘ 本地运行也花，排除 |

**最终原因**

V4L2 buffer 读写冲突：`grabFrame()` 中 DQBUF 取出缓冲区后立刻 QBUF 归还，但 `run()` 中的 `cvtColor` 还在读取该 buffer 的数据。驱动归还后可能立刻往 buffer 写入新帧，导致一帧中混合新旧数据，新数据区域亮度不同产生亮线条。

**排查方法：**

**花屏可能的原因有几个：**
可能原因 现象特征 排查方法
画面撕裂（tearing） 水平分界线，上半部分和下半部分不一致 冻结画面后看是否正常
V4L2 buffer 覆盖 随机位置出现色块或乱码 减少帧率看是否消失
YUYV 转换错误 固定位置出现颜色异常 静止不动时是否也有
带宽不足 整帧或大面积花屏 降低分辨率看是否消失
NFS 运行导致 程序在 NFS 上运行，I/O 抢占 CPU 复制到本地运行对比

**你在开发板上做几个测试：**

测试 1：静止时是否花屏

摄像头对着不动的物体，观察画面是否正常
→ 正常：说明是运动时才出现，倾向于 tearing 或 buffer 覆盖
→ 也花：说明是转换错误或带宽问题
测试结果：静止时不花屏

测试 2：冻结后是否正常

按空格冻结画面，看冻结的那一帧是否完整清晰
→ 完整：说明单帧数据没问题，是显示刷新时的撕裂
→ 也花：说明采集到的数据本身就有问题
测试结果：不花

测试 3：花屏的形状

观察花屏的样子：
→ 水平一条线分界，上下内容不一致 → tearing（撕裂）
→ 随机位置出现绿色/粉色色块 → YUYV 数据不完整或 buffer 被覆盖
→ 整帧闪烁 → 带宽或帧率问题
测试结果：有新的物品进入画面时，新物品的区域内随机出现一些宽的线条，线条的亮度比其他区域更亮

测试 4：复制到本地运行

开发板上
cp /mnt/nfs/endoscope /root/
cp -r /mnt/nfs/config /root/
cd /root
./endoscope -platform linuxfb

对比 NFS 运行和本地运行是否有区别
→ 本地不花：NFS I/O 抢占导致
→ 都花：和 NFS 无关
测试结果：都花

**解决方案**

将 QBUF 归还操作从 `grabFrame()` 移到 `run()` 中，确保 `cvtColor` 完成数据拷贝后再归还 buffer：

**修复前** (`grabFrame` 中立刻归还 buffer)：

```cpp
// grabFrame — DQBUF 后立刻 QBUF
bool V4l2Capture::grabFrame(void **data, int *size) {
    // ... select 等待 ...
    if (xioctl(m_fd, VIDIOC_DQBUF, &buf) < 0) return false;
    *data = m_buffers[buf.index].start;
    *size = buf.bytesused;
    if (xioctl(m_fd, VIDIOC_QBUF, &buf) < 0) return false;  // 立刻归还！
    return true;
}

// run — 读取时 buffer 可能已被驱动覆写
void V4l2Capture::run() {
    while (m_running) {
        void *data; int size;
        grabFrame(&data, &size);
        cv::Mat yuyv(m_height, m_width, CV_8UC2, data);  // data 指向已归还的 buffer
        cv::cvtColor(yuyv, bgr, cv::COLOR_YUV2BGR_YUYV); // 读写冲突！
        // ...
    }
}
```

**修复后** (拷贝完成后才归还 buffer)：

```cpp
// grabFrame — 只 DQBUF，不归还，返回 bufIndex
bool V4l2Capture::grabFrame(void **data, int *size, int *bufIndex) {
    // ... select 等待 ...
    if (xioctl(m_fd, VIDIOC_DQBUF, &buf) < 0) return false;
    *data = m_buffers[buf.index].start;
    *size = buf.bytesused;
    *bufIndex = buf.index;  // 记录索引，用完再归还
    return true;
}

// returnBuffer — 新增方法，单独归还
bool V4l2Capture::returnBuffer(int bufIndex) {
    struct v4l2_buffer buf;
    memset(&buf, 0, sizeof(buf));
    buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    buf.memory = V4L2_MEMORY_MMAP;
    buf.index = bufIndex;
    return xioctl(m_fd, VIDIOC_QBUF, &buf) >= 0;
}

// run — cvtColor 完成后才归还
void V4l2Capture::run() {
    while (m_running) {
        void *data; int size; int bufIndex;
        grabFrame(&data, &size, &bufIndex);
        cv::Mat yuyv(m_height, m_width, CV_8UC2, data);
        cv::cvtColor(yuyv, bgr, cv::COLOR_YUV2BGR_YUYV); // 数据已拷贝到 bgr
        returnBuffer(bufIndex);                            // 现在才归还
        // ...
    }
}
```

**文件**：`src/capture/v4l2capture.h`、`src/capture/v4l2capture.cpp`

**状态**：已修复，亮线条花屏已消除，剩余轻微拖影属于刷新率/延迟的正常现象

## 问题 #9：触摸事件穿透内窥镜界面到桌面程序

**日期**：2025-07

**现象**：运行内窥镜程序后，触摸操作会同时被内窥镜和开发板自带的 `/usr/local/qt-app/App` 程序接收，点击穿透到桌面图标。

```bash
kill $(pidof App) 2>/dev/null
sleep 0.5
/tmp/endoscope -platform linuxfb
```

**状态**：已解决

## 问题 #11：开启白平衡/CLAHE后程序被OOM Killer杀掉

**日期**：2025-07

**现象**

在开发板上开启白平衡和CLAHE后，程序卡住，内核日志显示：

```
endoscope invoked oom-killer
total-vm:974928kB, anon-rss:864212kB
Out of memory: Kill process 2542 (endoscope) score 907
```

程序占用 864MB 内存（开发板总共约 960MB RAM），被系统强制杀掉。

**原因分析**

信号队列堆积导致内存耗尽。根本原因：

1. 采集线程以 30fps 持续 `emit frameReady(image.copy())`，每个信号携带一个 QImage 拷贝（~900KB）
2. `onFrameReady` 通过 `Qt::QueuedConnection` 跨线程投递到主线程事件队列
3. 开启图像处理后，`processFrame` 耗时增加（CLAHE + 白平衡 + 颜色空间转换），主线程处理速度跟不上采集速度
4. 信号在队列中持续堆积，每个信号都持有一个 QImage 拷贝，内存持续增长
5. 开发板无 swap 分区，内存耗尽后 OOM Killer 直接杀掉进程

```
采集线程 (30fps)          主线程事件队列              主线程处理
emit frame1 ──────→  [frame1]                    处理 frame1 (慢)
emit frame2 ──────→  [frame1, frame2]            还在处理 frame1...
emit frame3 ──────→  [frame1, frame2, frame3]    还在处理 frame1...
...                   队列越来越长，内存持续增长     ...
emit frameN ──────→  [frame1...frameN]           OOM!
```

**解决方案 — 版本 1：主线程丢帧标志**

思路：在主线程 `onFrameReady` 中加 `m_processing` 标志，处理中时丢弃新帧。

```cpp
// mainwindow.h
bool m_processing = false;

// mainwindow.cpp — onFrameReady
if (m_processing) return;  // 上一帧还没处理完，丢弃新帧
m_processing = true;
// ... 处理帧 ...
m_processing = false;
```

结果：**无效，仍然 OOM**。原因：信号通过 `Qt::QueuedConnection` 入队时，QImage 拷贝已经分配内存，`m_processing` 检查是在信号出队后才执行的，此时内存已经占用。

**解决方案 — 版本 2：采集线程端丢帧**

思路：将控制移到采集线程端，`emit` 之前检查标志，信号根本不发出，内存不会增长。

```cpp
// videosource.h — 新增 public 成员
std::atomic<bool> m_frameProcessing{false};

// v4l2capture.cpp — run() 中 emit 之前检查
if (m_frameProcessing.load()) continue;  // 主线程还没处理完，不发信号
m_frameProcessing.store(true);
emit frameReady(image.copy());

// mainwindow.cpp — onFrameReady 末尾清除标志
m_source->m_frameProcessing.store(false);
```

结果：**OOM 已解决，但 FPS 从 24 降到 13~15**。原因：丢帧时采集线程 `continue` 回到循环顶部，仍然执行 `grabFrame`（select 等待 ~33ms）+ `cvtColor`（YUYV→BGR），然后才检查标志并丢弃，每次丢帧浪费约 40~50ms。将检查移到 `grabFrame` 之前后，FPS 仅恢复到 14~15，仍不理想。撤回版本 2 后 FPS 恢复到 24~28，确认是此方案导致性能下降。

**解决方案 — 版本 3：共享缓存 + 无参信号（当前版本）**

思路：不阻止采集，不通过信号传递 QImage。采集线程全速运行，只更新共享缓存（永远只保存 1 帧），发无参信号通知主线程。

```cpp
// videosource.h — 新增共享缓存
QImage m_latestFrame;                    // 最新帧（采集线程写，主线程读）
QMutex m_frameMutex;                     // 保护共享缓存
std::atomic<bool> m_frameReady{false};   // 新帧就绪标志

// 信号改为无参
void frameReady();  // 不再携带 QImage

// v4l2capture.cpp — 采集线程写入共享缓存
{
    QMutexLocker lock(&m_frameMutex);
    m_latestFrame = image.copy();
}
if (!m_frameReady.exchange(true))  // 只在上一帧被取走后才发信号
    emit frameReady();

// mainwindow.cpp — 主线程从共享缓存取帧
void MainWindow::onFrameReady() {
    if (!m_source->m_frameReady.exchange(false)) return;  // 多余信号直接跳过
    QImage image;
    {
        QMutexLocker lock(&m_source->m_frameMutex);
        image.swap(m_source->m_latestFrame);
    }
    // ... 处理 image ...
}
```

与版本 2 的对比：

| | 版本 2（采集端丢帧） | 版本 3（共享缓存） |
|---|---|---|
| 信号携带 | QImage 拷贝（~900KB） | 无参数（0字节） |
| 队列堆积 | 阻止采集来避免 | 最多 1 个空信号，不占内存 |
| 采集线程 | 被阻塞，影响 FPS | 全速运行，不受影响 |
| 帧数据内存 | 队列中多份 | 共享缓存永远只有 1 帧 |

**文件**：`src/capture/videosource.h`、`src/capture/v4l2capture.cpp`、`src/capture/filecapture.cpp`、`src/ui/mainwindow.h`、`src/ui/mainwindow.cpp`

**状态**：版本 3 结果：OOM 已解决，但 FPS 从 24~28 降到 15~20（QMutex + image.copy 开销），开图像处理后 UI 卡死（空信号堆积）。版本 3b（加 exchange 控制）后 UI 不再卡死，但 FPS 仍低。

**解决方案 — 版本 4：atomic 计数器限制队列深度（当前版本）**

思路：回到最简单的直接 emit QImage 方式，去掉 QMutex 和共享缓存，只用一个 atomic 计数器限制队列中最多 2 帧。

```cpp
// videosource.h — 新增 public 成员
std::atomic<int> m_pendingFrames{0};

// v4l2capture.cpp — run() 中，归还 buffer 后检查队列深度
returnBuffer(bufIndex);
if (m_pendingFrames.load() >= 2) continue;  // 队列满，跳过 BGR→RGB 和 image.copy
cv::cvtColor(bgr, rgb, cv::COLOR_BGR2RGB);
m_pendingFrames.fetch_add(1);
emit frameReady(image.copy());

// mainwindow.cpp — onFrameReady 末尾
m_source->m_pendingFrames.fetch_sub(1);
```

与前三个版本的对比：

| | 版本 1 | 版本 2 | 版本 3 | 版本 4 |
|---|---|---|---|---|
| 方案 | 主线程丢帧标志 | 采集端丢帧标志 | 共享缓存+无参信号 | atomic 计数器 |
| OOM | 未解决 | 已解决 | 已解决 | 已解决 |
| 基础 FPS | - | 13~15 | 15~20 | ~22 |
| 额外开销 | 无 | 阻塞采集线程 | QMutex+共享拷贝 | 1个 atomic int |

**文件**：`src/capture/videosource.h`、`src/capture/v4l2capture.cpp`、`src/capture/filecapture.cpp`、`src/ui/mainwindow.h`、`src/ui/mainwindow.cpp`

**版本 4 结果**：基础 FPS ~22（接近原始 24~28），开白平衡后 FPS 降到 ~8，未 OOM，未卡死。开处理后 FPS 下降属于算法耗时问题，需单独优化。

## 问题 #12：开启图像处理算法后 FPS 大幅下降

**日期**：2025-07

**现象**

- 基础 FPS（不开处理）：~22
- 开白平衡：FPS 降到 ~8
- 开多个算法：FPS 预计更低

**原因分析**

图像处理算法在主线程执行，每帧处理耗时过长。640x480 图像在 Cortex-A7 上各算法耗时待测量。

**排查结果**

各算法在 Cortex-A7 上的实测耗时（640x480，单帧）：

| 环节 | 平均耗时 |
|---|---|
| QImage→Mat | ~5ms |
| Mat→QImage | ~11ms |
| 白平衡 | ~40ms |
| CLAHE | ~115ms（首次 3380ms） |
| 去雾 | ~390ms |
| 锐化 | ~210ms |
| 降噪 | ~215ms |
| 边缘检测 | ~90ms |
| 阈值分割 | ~4ms |

每帧固定开销：QImage↔Mat 转换 ~16ms + src.clone() + scaled() + image.copy() 等。

**瓶颈分析**

1. 去雾最慢（390ms）：多次 split/merge/erode/divide，计算量大
2. 锐化（210ms）：GaussianBlur sigma=3.0 在 640x480 上开销大
3. 降噪（215ms）：双边滤波即使半分辨率仍然慢
4. CLAHE（115ms）：BGR→Lab→均衡→Lab→BGR 两次颜色空间转换
5. 白平衡（40ms）：split + mean + 乘法 + merge

**状态**：已优化

### 优化结果

| 算法 | 优化前 | 优化后 | 提升 |
|---|---|---|---|
| 白平衡 | ~40ms | ~9ms | **4.4x** |
| CLAHE | ~115ms（首次 3380ms） | ~28ms（首次 3308ms） | **4.1x** |
| 去雾 | ~390ms | ~87ms | **4.5x** |
| 锐化 | ~210ms | ~7ms | **30x** |
| 降噪 | ~215ms | ~55ms | **3.9x** |
| 边缘检测 | ~90ms | ~19ms | **4.7x** |
| 阈值分割 | ~4ms | ~1ms | **4x** |

降分辨率处理平均提升 4~5 倍，锐化因同时改用 3x3 固定核提升 30 倍。

CLAHE 首次调用仍然耗时 3308ms（OpenCV 内部初始化），复用对象后后续帧稳定在 ~28ms。

### 优化方案

#### 优化 1：降分辨率处理

将 640x480 缩小到 320x240 后再执行算法，处理完放大回原始尺寸。像素量减少 75%，耗时约降为 1/3~1/4。白平衡、CLAHE、去雾这类全局/区域算法在低分辨率下效果差别不大。

修改文件：`src/processing/imageprocessor.cpp` — `process()` 函数

修改前：

修改后：

#### 优化 3：锐化改用固定小核


### 降分辨率处理的画质风险

优化 1 将处理分辨率从 640x480 降到 320x240，处理完后放大到显示尺寸（480x360），放大倍率 1.5x。对不同算法的画质影响：

| 算法 | 影响程度 | 原因 |
|---|---|---|
| 白平衡 | 小 | 全局通道均值计算，不依赖分辨率 |
| CLAHE | 小 | 按区块做直方图均衡，320x240 区块数足够 |
| 阈值分割 | 小 | 像素级操作，放大后边缘稍模糊但可接受 |
| 降噪 | 小 | 本身就是模糊操作，低分辨率影响不大 |
| 锐化 | 中 | 低分辨率上锐化再放大，细节增强效果打折 |
| 去雾 | 中 | 暗通道先验依赖局部最小值，低分辨率会丢失细节 |
| 边缘检测 | 中 | 低分辨率会丢失细边缘 |

当前显示屏 800x480，视频区域 480x360，320x240 放大 1.5 倍到 480x360，画质损失不明显。如果后续换更高分辨率屏幕或对画质有更高要求，可以针对特定算法（锐化、去雾）改回全分辨率处理。

### 5. overrun 错误的根本原因

**overrun 发生机制**：

```
时间线（只有 2 个 buffer）：
T0: DQBUF buffer0 → CPU 开始处理（cvtColor + 图像处理）
T1: 摄像头写入 buffer1
T2: 摄像头写入完 buffer1，但 buffer0 还在 CPU 处理中
T3: 摄像头没有空闲 buffer，只能覆盖 buffer0 → overrun!
    （驱动报错：errors=3721 (overrun=3724)）
```

**为什么 2 个 buffer 不够？**

- 摄像头帧间隔：33ms（30fps）
- CPU 处理时间：
  - 不开处理：~20ms（勉强够用）
  - 开白平衡：~45ms（超时，发生 overrun）
  - 开多个算法：~100ms+（严重 overrun）

**如果有 4 个 buffer：**

```
T0: DQBUF buffer0 → CPU 处理
T1: 摄像头写入 buffer1
T2: 摄像头写入 buffer2
T3: 摄像头写入 buffer3
T4: buffer0 处理完，归还 → 摄像头有空闲 buffer，不会 overrun
```

## 问题 #14：Overrun 的真正机制 — 线程调度问题

**日期**：2025-01

**关键测试数据**（开启 CLAHE 后）：

| 指标 | 数值 |
|---|---|
| 采集线程平均耗时 | 48.4 ms/帧 |
| 总帧数 | 4441 帧 |
| Overrun 次数 | 669 次 |
| Overrun 率 | 15.06% |
| 运行时长 | ~148 秒 |
| 平均 overrun 频率 | 4.5 次/秒 |

**对比分析**：

| 场景 | 采集线程耗时 | Overrun 次数 | Overrun 率 | 结论 |
|---|---|---|---|---|
| 不开处理 | 37.5 ms | 0 | 0% | ✅ 正常 |
| 开启 CLAHE | 48.4 ms | 669 | 15.06% | ❌ 严重 |

**关键发现**：

采集线程耗时增加了 10.9ms（从 37.5ms → 48.4ms），但采集线程只做 YUYV→BGR→RGB 转换，不应该受 CLAHE 影响！

**Overrun 的真正原因**：

❌ **之前的假设错误**：不是 buffer 数量不足（实际有 4 个），也不是简单的处理时间过长

✅ **真正的原因**：**系统整体负载过高导致的线程调度问题**

**机制分析**：

```
1. 主线程处理 CLAHE 很慢（~28ms）+ QImage↔Mat 转换（~16ms）= ~44ms
2. 主线程处理不过来，信号队列堆积
3. 采集线程被 m_pendingFrames >= 2 限制，频繁跳帧
4. 采集线程跳帧时仍执行 grabFrame（select 等待）+ cvtColor，占用 CPU
5. 系统整体负载高，采集线程的调度优先级受影响
6. 采集线程响应变慢（37.5ms → 48.4ms）
7. 采集线程来不及归还 buffer → 4 个 buffer 全部被占用 → overrun!
```

**关键数据**：

- 4 个 buffer 缓冲能力：4 × 33ms = 132ms
- 当前处理时间：48.4ms（采集）+ 44ms（主线程）= 92.4ms
- 理论上不应该 overrun，但实际 overrun 率 15%

**说明**：不是简单的处理时间问题，而是线程调度和队列堆积导致的系统性问题。几乎每 4-5 帧就有 1 次 overrun，问题非常严重。

**优化方向**：

**短期（立即可做）**：

1. ✅ 启用 OV5640 硬件白平衡（省掉软件白平衡的 9ms）
2. 增加 `m_pendingFrames` 限制（从 2 改为 4，允许更多帧在队列中，但可能增加延迟）

**中期（需要重构）**：

1. 多线程流水线：采集线程 → 处理线程 → 主线程（只显示），解耦采集和处理
2. NEON 加速：加速 YUYV→RGB 转换和 CLAHE 等算法

**精准定位方法**：

1. 测试不同 `m_pendingFrames` 限制值（2/4/8）对 overrun 率的影响
2. 测试关闭主线程处理（只采集不处理）时的 overrun 率
3. 添加采集线程的 CPU 占用率监控
4. 测试不同算法组合的 overrun 率（白平衡、CLAHE、去雾等）

## 问题 #16：从图像编辑模式返回后视频采集无法恢复

**日期**：2025-01

**现象**

进入图像编辑模式后返回调试或临床模式，视频画面不恢复，终端持续输出"采集帧超时"。

**原因分析**

进入编辑模式时调用 `m_source->stop()` 设置 `m_running = false`，采集线程的 `run()` 循环检测到后退出，并在末尾调用 `stopStreaming()` 关闭视频流。

返回时调用 `m_source->start()` 尝试重启线程，但此时旧线程可能还没完全退出（`run()` 末尾的 `stopStreaming()` 还在执行），QThread 发现线程仍在运行，忽略了 `start()` 调用。等旧线程真正退出后，没有人再调用 `start()`，采集就永久停止了。

```
进入编辑模式：
  stop()          → m_running = false
  （线程还在执行 stopStreaming...）

返回调试模式：
  start()         → QThread 发现线程还在运行，忽略！
  （线程退出后无人再 start）
  → 采集永久停止，"采集帧超时"
```

**修复**

在 `stop()` 之后加 `wait()` 等待线程完全退出，在 `start()` 之前也加 `wait()` 确保线程已退出：

**改动前：**

```cpp
// 进入编辑模式
case 2:
    if (m_source && m_source->isRunning())
        m_source->stop();
    break;

// 离开编辑模式
if (m_currentMode == 2 && mode != 2) {
    if (m_source && !m_source->isRunning())
        m_source->start();
}
```

**改动后：**

```cpp
// 进入编辑模式：stop + wait 确保线程完全退出
case 2:
    if (m_source) {
        m_source->stop();
        m_source->wait();  // 等待线程完全退出
    }
    break;

// 离开编辑模式：wait + start 确保线程已退出再重启
if (m_currentMode == 2 && mode != 2) {
    if (m_source) {
        m_source->wait();  // 确保线程完全退出
        m_source->start(); // 重新启动采集线程
    }
}
```

**关键点**：QThread 的 `start()` 只能在线程未运行时调用，如果线程还在退出过程中（`run()` 末尾清理），`start()` 会被忽略。`wait()` 阻塞等待线程真正结束后再 `start()`，保证重启成功。

**文件**：`src/ui/mainwindow.cpp` — `switchToMode()` 函数

**状态**：已修复

### 问题 #16 补充：第一版修复无效，第二版方案

**第一版修复为什么无效**

第一版在进入编辑模式时 `stop()` + `wait()` 杀死采集线程，返回时 `wait()` + `start()` 重启线程。日志显示：

```
[MODE] wait 完成, 准备 start
[MODE] start 完成, isRunning: false    ← start() 调用了但线程没启动
采集帧超时
```

原因：V4L2 的 `run()` 方法在退出时调用了 `stopStreaming()`（即 `VIDIOC_STREAMOFF`），这会取消所有 buffer 的排队。再次 `start()` 时 `run()` 开头调用 `startStreaming()`（即 `VIDIOC_STREAMON`），但此时所有 buffer 都已被取消排队，没有可用的 buffer 给驱动填充数据，导致采集失败。

```
第一次启动：
  open() → requestBuffers() → QBUF 所有 buffer → run() → STREAMON ✅

stop + start 后：
  run() 退出 → STREAMOFF（所有 buffer 被取消排队）
  run() 重新开始 → STREAMON（但没有 buffer 在队列中）→ 采集失败 ❌
```

要修复需要在 `startStreaming()` 前重新 QBUF 所有 buffer，但这改动较大且容易引入新问题。

**第二版方案：pause/resume 代替 stop/start**

核心思路：**线程始终运行，不杀死**。用 `pause()`/`resume()` 控制是否处理帧。暂停时线程仍然 grabFrame + returnBuffer 保持 V4L2 流活跃，只是不 emit 帧数据。

**videosource.h — 新增 pause/resume，新增 m_quit 标志**

改动前：

```cpp
    void stop() { m_running = false; }
    bool isRunning() const { return m_running; }

    // ...
    std::atomic<bool> m_running;
```

改动后：

```cpp
    // 暂停采集：线程保持运行但不处理帧
    void pause() { m_running = false; }

    // 恢复采集
    void resume() { m_running = true; }

    // 停止线程：线程完全退出
    void stop() { m_quit = true; m_running = false; }

    bool isRunning() const { return m_running; }

    // ...
    std::atomic<bool> m_running;
    std::atomic<bool> m_quit{false};
```

**v4l2capture.cpp — run() 循环用 m_quit 控制退出，m_running 控制暂停**

改动前：

```cpp
void V4l2Capture::run() {
    if (!startStreaming()) return;
    m_running = true;
    // ...
    while (m_running) {
        if (!grabFrame(&data, &size, &bufIndex)) {
            if (m_running)
                qDebug() << "采集帧超时";
            continue;
        }

        perfTimer.start();
        // ... 处理帧 ...
    }
    stopStreaming();
}
```

改动后：

```cpp

    if (!startStreaming()) return;
    m_running = true;
    // ...
    while (!m_quit) {
        if (!grabFrame(&data, &size, &bufIndex)) {
            if (!m_quit)
                qDebug() << "采集帧超时";
            continue;
        }

        // 暂停时只归还 buffer，不处理帧
        if (!m_running) {
            returnBuffer(bufIndex);
            continue;
        }

        perfTimer.start();
        // ... 处理帧 ...
    }
    stopStreaming();
}
```

**filecapture.cpp — run() 同样改为 m_quit 控制退出**

改动前：

```cpp
void FileCapture::run() {
    m_running = true;
    while (m_running) {
        // ... 读帧 + emit ...
    }
}
```

改动后：

```cpp
void FileCapture::run() {
    m_running = true;
    while (!m_quit) {
        if (!m_running) {
            QThread::msleep(50);
            continue;
        }
        // ... 读帧 + emit ...
    }
}
```

**mainwindow.cpp — switchToMode() 用 pause/resume 代替 stop/start**

改动前：

```cpp
// 离开编辑模式
if (m_currentMode == 2 && mode != 2) {
    if (m_source) {
        m_source->wait();
        m_source->start();
    }
}

// 进入编辑模式
case 2:
    if (m_source) {
        m_source->stop();
        m_source->wait();
    }
```

改动后：

```cpp
// 离开编辑模式
if (m_currentMode == 2 && mode != 2) {
    if (m_source)
        m_source->resume();
}

// 进入编辑模式
case 2:
    if (m_source)
        m_source->pause();
```

**两版方案对比**

| | 第一版（stop/start） | 第二版（pause/resume） |
|---|---|---|
| 线程状态 | 杀死再重启 | 始终运行 |
| V4L2 流 | STREAMOFF → STREAMON（失败） | 始终保持 STREAMON |
| 暂停时 CPU | 线程退出，零开销 | 线程仍在 grabFrame + returnBuffer |
| 恢复速度 | 慢（需重启线程） | 即时（设置标志位） |
| 可靠性 | ❌ V4L2 重启失败 | ✅ 稳定 |

## 问题 #18：M4 按键中断无响应 — A7 设备树抢占 GPIO

**日期**：2025-07

**来源**：M4 固件开发，按键功能验证阶段

**现象**

M4 固件中配置了 PB13 (KEY1) 和 PH7 (KEY2) 为 EXTI 下降沿中断，编译烧录后：
- KEY2 (PH7) 按下时亮度似乎有变化（后确认为误判，初始亮度 500/999 太高，变化不明显）
- KEY1 (PB13) 按下完全无反应
- 最终确认：**两个按键的中断回调都没有触发**

**排查过程**

### 第一步：排除代码逻辑问题

尝试了多种代码修改：
1. 简化 KEY1 回调为直接 `SET_COMPARE(0)` → 无效
2. 加长消抖时间到 200ms → 无效
3. 中断回调只做计数，主循环根据计数器更新 LED → 无效
4. 在中断回调中加 `HAL_Delay` 做闪烁测试 → 导致系统卡死（中断中不能用 HAL_Delay）

> **教训**：`HAL_Delay` 依赖 SysTick 中断，在 EXTI 中断回调中调用会导致死锁。中断回调中只能做快速操作（设标志位、改寄存器），不能调用阻塞函数。

### 第二步：确认硬件正常

在开发板 A7 侧 Linux 上读取 GPIO 状态，确认按键硬件没问题：

```bash
cat /sys/kernel/debug/pinctrl/soc:pin-controller@50002000/pinmux-pins | grep "PB13\|PH7"
# pin 29 (PB13): GPIO GPIOB:29
# pin 119 (PH7): GPIO GPIOH:119
```

引脚存在且未被其他外设（I2C/SPI 等）复用。

### 第三步：发现 A7 侧拦截了中断（根本原因）

查看 A7 侧的中断注册情况：

```bash
cat /proc/interrupts | grep -i "Key\|gpio\|13"
```

输出：
```
89:         18          0  stm32gpio   7 Edge      Key 1
90:          8          0  stm32gpio  13 Edge      Key 0
```

**关键发现**：
- A7 的 Linux 已经注册了 PB13 和 PH7 作为 `gpio-keys` 设备
- `Key 1` 计数 18，`Key 0` 计数 8 — 说明按键硬件正常，A7 收到了中断
- A7 拦截了这些 GPIO 中断，M4 根本收不到

### 第四步：确认设备树中的按键配置

```bash
find /sys/firmware/devicetree -name "*key*" -o -name "*button*" 2>/dev/null
# /sys/firmware/devicetree/base/gpio-keys

cat /sys/firmware/devicetree/base/gpio-keys/status 2>/dev/null | tr '\0' '\n'
# okay

cat /sys/firmware/devicetree/base/gpio-keys/Key0/label 2>/dev/null | tr '\0' '\n'
# Key 0

cat /sys/firmware/devicetree/base/gpio-keys/Key1/label 2>/dev/null | tr '\0' '\n'
# Key 1
```

确认 `gpio-keys` 节点状态为 `okay`（启用中）。

### 第五步：找到 overlay 来源

查看启动配置文件：

```bash
cat /boot/uEnv.txt | grep key
# dtoverlay=/usr/lib/linux-image-4.19.94-stm-r1/overlays/stm-fire-key.dtbo
```

`stm-fire-key.dtbo` 这个设备树 overlay 在启动时加载，注册了 PB13 和 PH7 为 A7 侧的 gpio-keys 设备。

**原因总结**

A7 侧 Linux 通过 `stm-fire-key.dtbo` 设备树 overlay 注册了 PB13 (Key 0) 和 PH7 (Key 1) 为 gpio-keys 设备。A7 的 GPIO 中断控制器优先处理这些中断，M4 的 EXTI 配置被覆盖，中断回调永远不会触发。

```
按键按下 → GPIO 中断 → A7 的 gpio-keys 驱动处理 → M4 收不到
```

**修复方法**

在 `/boot/uEnv.txt` 中注释掉 `stm-fire-key.dtbo`，禁用 A7 侧的按键驱动：

```bash
# 修改前
dtoverlay=/usr/lib/linux-image-4.19.94-stm-r1/overlays/stm-fire-key.dtbo

# 修改后
#dtoverlay=/usr/lib/linux-image-4.19.94-stm-r1/overlays/stm-fire-key.dtbo
```

具体命令：

```bash
sed -i 's|^dtoverlay=/usr/lib/linux-image-4.19.94-stm-r1/overlays/stm-fire-key.dtbo|#dtoverlay=/usr/lib/linux-image-4.19.94-stm-r1/overlays/stm-fire-key.dtbo|' /boot/uEnv.txt
```

修改后需要 **重启开发板** 才能生效。

**验证方法**

重启后确认按键不再被 A7 占用：

```bash
cat /proc/interrupts | grep -i "Key"
# 应该没有输出
```

**影响评估**

禁用 `stm-fire-key.dtbo` 后：
- ✅ M4 可以正常接收 KEY1/KEY2 的 EXTI 中断
- ❌ A7 侧 Linux 不再能通过 `/dev/input/` 读取按键事件
- ✅ 对摄像头、LCD、触摸屏等其他功能无影响
- ✅ 我们的内窥镜项目中按键由 M4 控制，A7 不需要直接读取按键

**文件**：`/boot/uEnv.txt`（开发板）

**状态**：已修复（禁用 A7 按键 overlay），但 M4 EXTI 中断仍无法触发，详见问题 #19

## 问题 #19：M4 EXTI 中断无法触发 — 双核 EXTI 路由问题

**日期**：2025-07

**来源**：问题 #18 修复后，M4 按键中断仍然不工作

**现象**

禁用 A7 侧 `stm-fire-key.dtbo` 后，GPIO 引脚已释放（UNCLAIMED），但 M4 的 EXTI 中断回调仍然不触发。轮询方式可以正常读取 GPIO 电平。

**排查过程**

### 第一步：确认按键硬件和电平逻辑

通过轮询 `HAL_GPIO_ReadPin` 测试，用 LED 亮度编码显示两个按键状态：
- PB13 (KEY1)：不按=低电平，按下=高电平 → **高电平有效**
- PH7 (KEY2)：不按=低电平，按下=高电平 → **高电平有效**

原始 CubeMX 配置为上拉+下降沿（假设低电平有效），已修正为**下拉+上升沿**。

### 第二步：确认 EXTI 寄存器配置

在 M4 代码中直接读取 EXTI 寄存器，用 LED 闪烁次数显示 6 个配置位：

| 检查项 | 寄存器 | 期望值 | 实际结果 |
|--------|--------|--------|----------|
| EXTI13→PB 映射 | EXTICR[3] bit[8:15] | 1 (PB) | ✅ 正确 |
| EXTI7→PH 映射 | EXTICR[1] bit[24:31] | 7 (PH) | ✅ 正确 |
| KEY1 中断使能 | EXTI_C2->IMR1 bit13 | 1 | ✅ 正确 |
| KEY1 上升沿触发 | EXTI->RTSR1 bit13 | 1 | ✅ 正确 |
| KEY2 中断使能 | EXTI_C2->IMR1 bit7 | 1 | ✅ 正确 |
| KEY2 上升沿触发 | EXTI->RTSR1 bit7 | 1 | ✅ 正确 |

**6 个配置位全部正确**（LED 闪烁 6 次确认）。

> **注意**：STM32MP1 的 EXTI 寄存器名与普通 STM32 不同：
> - M4 侧中断屏蔽：`EXTI_C2->IMR1`（不是 `EXTI->IMR1`，那是 A7 侧的）
> - 上升沿挂起：`EXTI->RPR1`（不是 `EXTI->PR1`）
> - 下降沿挂起：`EXTI->FPR1`

### 第三步：检查 EXTI 挂起寄存器

按下按键后检查 `EXTI->RPR1` 的 bit13 和 bit7：
- **结果：挂起位没有被置位**
- 同时轮询 GPIO 电平可以检测到按键按下

**说明 EXTI 控制器没有检测到 GPIO 边沿变化**，尽管寄存器配置正确。

### 第四步：分析设备树中的 EXTI 架构

查看主设备树 `stm32mp157a-basic.dts`：

```dts
&m4_rproc {
    interrupt-parent = <&exti>;
    interrupts = <68 1>;        // 只有 line 68（看门狗）
    interrupt-names = "wdg";
};
```

EXTI 控制器属性：
```
compatible = "st,stm32mp1-exti", "syscon"
hwlocks = <0x24 0x01>    // 使用硬件锁仲裁 A7/M4 访问
```

**关键发现**：
1. M4 的 remoteproc 节点只声明了 EXTI line 68，没有 line 7 和 line 13
2. EXTI 控制器使用 `hwlocks` 进行 A7/M4 仲裁
3. 野火源码中的按键中断示例（`button_interrupt`）是 A7 侧 Linux 驱动，通过 `<&gpiob 13 IRQ_TYPE_EDGE_RISING>` 注册，走的是 Linux GPIO 中断子系统，不是 M4 直接操作 EXTI

**根本原因分析**

STM32MP1 的 EXTI 控制器由 A7 侧 Linux 内核管理。即使 M4 通过 HAL 库配置了 EXTI 寄存器，A7 的 Linux EXTI 驱动可能：
1. 在初始化时覆盖了 M4 的 EXTI 配置
2. 通过 hwlock 机制阻止了 M4 对 EXTI 的写入
3. EXTI 的 GPIO 选择（EXTICR）需要通过 A7 侧的 pinctrl 子系统配置才能真正生效

**M4 能读到正确的寄存器值，但 EXTI 硬件实际上没有按这些值工作**，可能是因为 A7 侧的 EXTI 驱动持有 hwlock，M4 的写入被静默忽略。

**可能的解决方案**

| 方案 | 难度 | 风险 | 说明 |
|------|------|------|------|
| A. 轮询方式 | 低 | 无 | 当前可用，但浪费 CPU |
| B. 设备树 overlay 声明 M4 使用 EXTI7/13 | 中 | 低 | 需要研究 stm32mp1-exti 驱动的设备树绑定 |
| C. M4 使用 TIM 定时器中断做周期性轮询 | 低 | 无 | 比主循环轮询更优雅，不阻塞主循环 |
| D. 通过 RPMsg 让 A7 转发按键事件给 M4 | 中 | 低 | A7 注册 gpio-keys，通过 RPMsg 通知 M4 |

**当前状态**：使用轮询方式作为临时方案，EXTI 中断问题待后续通过设备树方案解决

## 问题 #20：M4 OpenAMP RPMsg 通道无法建立 — Linux 找不到 resource table

**日期**：2025-07

**来源**：M4 固件集成 OpenAMP 后，A7 侧 `/dev/ttyRPMSG0` 设备不出现

**背景**

设计文档步骤 6 要求实现 A7-M4 通信。原方案为物理 UART + DMA，后改为 OpenAMP RPMsg 虚拟串口（不占用物理引脚，协议层不变）。CubeMX 中启用 OpenAMP（Remote 模式）+ IPCC，生成代码后编译通过，M4 通过 remoteproc 加载成功（state=running），但 A7 侧没有创建 RPMsg 设备。

**现象**

```bash
root@lubancat:~# echo start > /sys/class/remoteproc/remoteproc0/state
# M4 启动成功

root@lubancat:~# ls /dev/ttyRPMSG*
# 无输出 — 设备不存在

root@lubancat:~# ls /sys/bus/rpmsg/devices/
# 无输出 — 没有 rpmsg 设备

root@lubancat:~# dmesg | grep rpmsg
# 无输出 — 内核没有任何 rpmsg 日志
```

dmesg 中的关键错误信息：

```
remoteproc remoteproc0: no resource table found for this firmware
remoteproc remoteproc0: header-less resource table
remoteproc remoteproc0: remote processor m4 is now up
```

M4 能启动运行（LED、按键、I2C 均正常），但 Linux 找不到 resource table，因此不会初始化 virtio/rpmsg 子系统。

**排查过程**

### 第一步：确认 A7 侧内核支持

```bash
root@lubancat:~# zcat /proc/config.gz | grep -i RPMSG
CONFIG_RPMSG=y
CONFIG_RPMSG_VIRTIO=y
CONFIG_RPMSG_TTY=y          # 关键：会创建 /dev/ttyRPMSGx

root@lubancat:~# ls /sys/bus/ | grep rpmsg
rpmsg                        # rpmsg 总线存在
```

**结论**：内核完全支持 rpmsg，问题不在 A7 侧内核。

### 第二步：用野火示例 ELF 对比验证

将野火提供的 `OpenAMP_raw_CM4.elf` 加载到开发板：

```bash
root@lubancat:~# echo start > /sys/class/remoteproc/remoteproc0/state
 m4@0#vdev0buffer: assigned reserved memory node vdev0buffer@10044000
 virtio_rpmsg_bus virtio0: creating channel M4 rpmsg-client addr 0x0
 virtio_rpmsg_bus virtio0: rpmsg host is online
 m4@0#vdev0buffer: registered virtio0 (type 7)
 virtio_rpmsg_bus virtio0: creating channel rpmsg-tty-channel addr 0x1
 remoteproc remoteproc0: remote processor m4 is now up
 rpmsg_tty virtio0.rpmsg-tty-channel.-1.1: new channel: 0x400 -> 0x1 : ttyRPMSG0

root@lubancat:~# ls /dev/ttyRPMSG0
/dev/ttyRPMSG0               # ✅ 设备出现
```

**结论**：野火 ELF 能正常建立 RPMsg 通道，说明硬件和内核都没问题，问题在我们的固件。

### 第三步：对比两个 ELF 的 resource table

使用 `arm-none-eabi-objdump` 对比 `.resource_table` section：

**Section 属性对比**：

| 属性 | 我们的 ELF | 野火 ELF |
|---|---|---|
| Section name | `.resource_table` | `.resource_table` |
| VMA | `0x10020128` | `0x10020118` |
| Size | `0x8c` (140 bytes) | `0x8c` (140 bytes) |
| Type | PROGBITS | PROGBITS |
| Flags | A (ALLOC) | A (ALLOC) |
| Segment | #02 (.data 同段) | #02 (.data 同段) |

**Resource table 内容对比**（hex dump）：

| 偏移 | 我们的 | 野火的 | 含义 |
|---|---|---|---|
| 0x00 | `01 00 00 00` | `01 00 00 00` | version = 1 ✅ |
| 0x04 | **`01 00 00 00`** | **`02 00 00 00`** | **num = 1 vs num = 2** |
| 0x10 | `18 00 00 00` | `18 00 00 00` | offset[0] = 0x18 (vdev) ✅ |
| 0x14 | `5c 00 00 00` | `5c 00 00 00` | offset[1] = 0x5c (trace) ✅ |
| 0x18-0x57 | vdev + vring0 + vring1 | 相同 | virtio 设备描述 ✅ |
| 0x58-0x8b | **全零** | **`cm4_log` trace entry** | **trace buffer 缺失** |

**关键差异**：
- 我们的 `num=1`（只有 vdev），野火的 `num=2`（vdev + trace）
- 野火的包含 `cm4_log` trace buffer entry，我们的该区域全零
- 但 `num=1` 是合法值，不应导致 "no resource table found"

**符号对比**：

野火 ELF 包含 VIRT_UART 相关符号（`VIRT_UART_Init`, `VIRT_UART_Transmit` 等），我们的 ELF 没有。但这影响的是 RPMsg channel 创建，不影响 resource table 识别。

### 第四步：分析 Linux 日志差异

**我们的 ELF**（失败）：
```
remoteproc0: Booting fw image EndoscopeM4_CM4.elf, size 2420788
remoteproc0: header-less resource table
remoteproc0: no resource table found for this firmware
remoteproc0: header-less resource table
remoteproc0: remote processor m4 is now up
```

**野火 ELF**（成功）：
```
remoteproc0: Booting fw image OpenAMP_raw_CM4.elf, size 2190736
m4@0#vdev0buffer: assigned reserved memory node vdev0buffer@10044000
virtio_rpmsg_bus virtio0: creating channel M4 rpmsg-client addr 0x0
virtio_rpmsg_bus virtio0: rpmsg host is online
m4@0#vdev0buffer: registered virtio0 (type 7)
virtio_rpmsg_bus virtio0: creating channel rpmsg-tty-channel addr 0x1
remoteproc0: remote processor m4 is now up
rpmsg_tty virtio0.rpmsg-tty-channel.-1.1: new channel: 0x400 -> 0x1 : ttyRPMSG0
```

关键线索：野火 ELF 触发了 `assigned reserved memory node vdev0buffer@10044000`，说明 Linux 的 STM32 remoteproc 驱动通过**设备树中的 reserved memory 区域**来定位共享内存和 resource table，而不仅仅依赖 ELF section。

### 第五步：初步定位根因

**假设**：STM32MP1 的 remoteproc 驱动（`stm32_rproc.c`）查找 resource table 的方式可能依赖设备树中定义的固定内存地址（如 `rsc-table@10048000`），而不是（或不仅是）ELF 文件中的 `.resource_table` section。

设备树中可能定义了：
```dts
reserved-memory {
    vdev0buffer@10044000 { ... };   // virtio buffer 共享内存
    rsc-table@10048000 { ... };     // resource table 固定地址
};
```

如果 Linux 去设备树指定的地址（如 `0x10048000`）查找 resource table，而我们的 resource table 实际在 `0x10020128`（SRAM2），地址不匹配就会报 "no resource table found"。

**待确认**：需要在开发板上查看设备树中 reserved-memory 的定义，确认 resource table 的期望地址。

**当前状态**：✅ 已解决

**已尝试的修改**

1. 启用 `VIRTUAL_UART_MODULE_ENABLED` 宏（`openamp_conf.h`），使用 VIRT_UART 接口替代底层 RPMsg API — 编译通过
2. 改写 main.c 使用 `VIRT_UART_Init` / `VIRT_UART_Transmit` / `VIRT_UART_RegisterCallback`，与野火示例 ELF 的接口一致

**根因分析**

问题由两个独立原因共同导致：

### 根因 1：GCC 13 链接器 segment alignment 变更

| | 野火 ELF (GCC 9.3) | 我们的 ELF (GCC 13.3) |
|---|---|---|
| Segment Align | `0x10000` (64KB) | `0x1000` (4KB) |
| Linux 识别 | ✅ 成功 | ❌ "no resource table found" |

GCC 13 的链接器默认 `max-page-size=0x1000`，而 STM32MP1 的 Linux remoteproc 驱动（基于较老内核）在解析 ELF 时依赖 `0x10000` 对齐来正确定位 resource table section。

**修复**：链接器参数添加 `-Wl,-z,max-page-size=0x10000`

### 根因 2：RPMsg service name 不匹配

| | 野火 ELF | 我们的 ELF |

| Service name | `"rpmsg-tty-channel"` | `"rpmsg-tty"` |
| Linux rpmsg_tty 匹配 | ✅ 创建 ttyRPMSG0 | ❌ 不匹配，无设备 |

**修复**：修改 `Middlewares/Third_Party/OpenAMP/virtual_driver/virt_uart.c` 中的宏：
```c
#define RPMSG_SERVICE_NAME  "rpmsg-tty-channel"
```

## 问题 #21：FreeRTOS 集成后 RPMsg 通信中断 — cat 读到一条消息后 EOF 退出

**日期**：2025-07

**来源**：M4 固件集成 FreeRTOS 后，A7 侧 `cat /dev/ttyRPMSG0` 只能收到一条残留消息后就退出

**背景**

步骤 6（RPMsg 虚拟串口通信）在裸机模式下完美工作：温度每秒上报、指令即时响应、按键事件正常。步骤 7 集成 FreeRTOS 后，将裸机主循环逻辑迁移到 FreeRTOS 任务中，RPMsg 通道能建立（`/dev/ttyRPMSG0` 出现），但数据通信完全中断。

**现象**

```bash
root@lubancat:~# echo start > /sys/class/remoteproc/remoteproc0/state
# M4 启动成功，ttyRPMSG0 出现

root@lubancat:~# cat /dev/ttyRPMSG0 &
[1] 2988
L75                          # ← 只收到一条残留数据
[1]+  Done  cat /dev/ttyRPMSG0   # ← cat 正常退出（EOF）

# 之后发送指令也无响应
root@lubancat:~# echo "L50" > /dev/ttyRPMSG0
# 无回复，LED 无变化
```

**关键对比**：
- 裸机版本（无 FreeRTOS）：✅ 温度每秒上报，指令即时响应
- FreeRTOS 版本（相同逻辑移到任务中）：❌ 无任何数据输出

**排查过程**

### 第一步：排除多任务问题

将所有逻辑合并到单个 FreeRTOS 任务（defaultTask）中，逻辑与裸机主循环完全一致：

```c
void StartDefaultTask(void const * argument)
{
  // VIRT_UART 初始化
  VIRT_UART_Init(&huart0);
  VIRT_UART_RegisterCallback(&huart0, ...);
  rpmsg_ready = 1;

  for(;;)
  {
    OPENAMP_check_for_message();
    // 处理指令、按键、温度...
    vTaskDelay(pdMS_TO_TICKS(10));
  }
}
```

**结果**：仍然不工作。说明问题不在多任务拆分，而在 FreeRTOS 本身与 OpenAMP 的交互。

### 第二步：排除栈空间不足

- 增大 `configTOTAL_HEAP_SIZE` 从 8192 到 15360
- defaultTask 栈 512 words，Task_Temperature 384 words，Task_Command 384 words

**结果**：无改善。

### 第三步：排除 VIRT_UART 初始化时机问题

将 `VIRT_UART_Init` 从 `main.c` 的 `USER CODE BEGIN 2`（调度器启动前）移到 `defaultTask` 开头（调度器启动后）。

**结果**：无改善。RPMsg channel 能建立（dmesg 显示 `creating channel rpmsg-tty-channel`），但数据不通。

### 第四步：分析 libmetal FreeRTOS 后端

编译时定义了 `-DMETAL_FREERTOS`，导致 libmetal 使用 `metal/system/freertos/` 后端而非 `metal/system/generic/` 后端。

FreeRTOS 后端的关键差异：
- **mutex 实现**：使用 `xSemaphoreCreateMutex()` 代替简单的关中断
- **内存分配**：使用 `pvPortMalloc()` 代替标准 `malloc()`

编译时有大量 warning：
```
metal/system/freertos/mutex.h:88:25: warning: implicit declaration of function
'xSemaphoreGetMutexHolder'; did you mean 'xQueueGetMutexHolder'?
```

`__metal_mutex_is_acquired()` 调用了 `xSemaphoreGetMutexHolder()`，这个函数在新版 FreeRTOS（v10.5+）中虽然存在但声明方式变了，导致 implicit declaration warning。

**假设**：libmetal 的 FreeRTOS mutex 后端可能导致 OpenAMP 内部死锁或数据丢失。在裸机模式下 libmetal 使用 generic 后端（关中断保护），简单可靠；切换到 FreeRTOS 后端后，mutex 操作可能在 IPCC 中断上下文中被调用，导致问题。

### 第五步：确认 `cat` 退出原因

`cat /dev/ttyRPMSG0` 退出状态为 `Done`（正常退出，非错误），说明读到了 EOF。可能原因：
1. M4 侧 RPMsg endpoint 在发送一条消息后被意外销毁
2. libmetal mutex 死锁导致 `OPENAMP_check_for_message()` 不再处理 IPCC 中断
3. `vTaskDelay` 期间 IPCC 中断触发但 OpenAMP 的中断处理依赖主循环轮询，导致消息丢失

**当前状态**：排查中（已定位根因方向）

**已确认的信息**

| 项目 | 状态 |
|---|---|
| RPMsg channel 建立 | ✅ dmesg 显示 `ttyRPMSG0` 创建成功 |
| FreeRTOS 调度器启动 | ✅（否则 channel 不会建立） |
| SysTick + HAL_IncTick | ✅ 在 SysTick_Handler 中同时调用 |
| xPortSysTickHandler | ✅ 在 SysTick_Handler 中调用 |
| configTOTAL_HEAP_SIZE | 15360 bytes |
| configUSE_MUTEXES | 1 |
| METAL_FREERTOS 定义 | ✅（CubeMX 自动添加） |
| libmetal mutex warning | 有（xSemaphoreGetMutexHolder implicit declaration） |

**下一步排查方向**

1. **去掉 `METAL_FREERTOS` 定义**：让 libmetal 使用 generic 后端（关中断），验证是否是 FreeRTOS mutex 后端的问题
2. **用 `while read` 或 `dd` 替代 `cat`**：确认是 A7 侧读取问题还是 M4 侧发送问题
3. **在 defaultTask 中只发送固定字符串**：排除 I2C/SHT30 相关问题
4. **检查 IPCC 中断优先级**：FreeRTOS 要求中断优先级 ≥ `configMAX_SYSCALL_INTERRUPT_PRIORITY`，IPCC 中断如果优先级太高可能导致 FreeRTOS API 调用异常

**相关文件**

- `CM4/Core/Src/app_freertos.c` — FreeRTOS 任务实现
- `CM4/Core/Src/main.c` — 外设初始化 + 共享函数

### 第六步：验证 FreeRTOS 调度器是否启动

在 `vTaskStartScheduler()` 之后的 `while(1)` 中加 LED 快闪（调度器失败指示），在 defaultTask 开头加 LED 慢闪 3 次（任务运行确认）。

**结果**：LED 慢闪 3 次 → **FreeRTOS 调度器正常启动，任务正在运行**。

### 第七步：验证 VIRT_UART_Transmit 返回值

用 LED 闪烁编码诊断：
- `rpmsg_ready` 状态：闪 1 次 = ready（初始化成功）
- `VIRT_UART_Transmit` 返回值：闪 2 次 = OK，闪 7 次 = ERROR

**结果**：
- `rpmsg_ready = 1` ✅（VIRT_UART_Init 成功）
- `VIRT_UART_Transmit` 返回 **ERROR** ❌

**结论**：endpoint 创建成功，但发送失败。`OPENAMP_send` 内部返回负值。

### 第八步：验证 endpoint ready 状态

在发送前等待 `is_rpmsg_ept_ready(&huart0.ept)` 返回 true。

**结果**：LED 常亮 — `is_rpmsg_ept_ready()` **永远返回 false**。

`is_rpmsg_ept_ready` 检查 `ept->dest_addr != RPMSG_ADDR_ANY`，即远端地址是否已确认。在裸机模式下这个地址在 `VIRT_UART_Init` 过程中就被设置了，但在 FreeRTOS 环境下没有。

### 第九步：排除多任务问题 — 单任务测试

将所有逻辑合并到单个 defaultTask 中（与裸机主循环完全一致的逻辑）。

**结果**：仍然不工作。排除多任务拆分导致的问题。

### 第十步：排除调度器影响 — 禁用调度器

注释掉 `MX_FREERTOS_Init()` 和 `vTaskStartScheduler()`，在 `while(1)` 中直接运行裸机逻辑。

**结果**：**仍然不工作！** 这证明问题不在 FreeRTOS 调度器本身。

### 第十一步：确认是 FreeRTOS 编译引入的问题

即使不启动调度器，只要工程中编译了 FreeRTOS 相关文件，RPMsg 就不工作。

**关键发现**：CubeMX 添加 FreeRTOS 后，`objects.list` 中新增了 4 个 libmetal FreeRTOS 后端文件：
```

./Middlewares/Third_Party/OpenAMP/freertos_init.o
./Middlewares/Third_Party/OpenAMP/freertos_io.o

```

这些文件提供了 libmetal 的系统抽象层实现（`_metal` 全局变量、`metal_sys_io_mem_map` 等）。编译时定义了 `-DMETAL_FREERTOS`，使 libmetal 头文件选择 FreeRTOS 后端（使用 `xSemaphoreCreateMutex` 等）。

### 第十二步：尝试去掉 METAL_FREERTOS

将 `.cproject` 中的 `METAL_FREERTOS` 改为 `METAL_FREERTOS_DISABLED`，让头文件选择 generic 后端。

**结果**：编译通过（warnings 从 21 降到 2），但**链接时仍然链接了 FreeRTOS 后端的 .o 文件**。由于头文件选择了 generic 后端但 .o 文件是 FreeRTOS 后端编译的，可能导致符号冲突或行为不一致。

尝试从 objects.list 中移除这 4 个 .o 文件 → 链接失败（缺少 `_metal`、`metal_sys_io_mem_map` 等符号），因为没有 generic 后端的 .o 文件替代。

### 第十三步：将 MX_OPENAMP_Init 移到任务中

假设 libmetal FreeRTOS 后端的 mutex 需要 FreeRTOS heap 可用才能正确创建，将 `MX_OPENAMP_Init` + `VIRT_UART_Init` 从 `main()` 移到 defaultTask 开头（调度器已启动后）。

**结果**：dmesg 显示 channel 创建成功，但 `cat /dev/ttyRPMSG0` 仍然立即 EOF 退出。数据发不出去。

### 根因分析（当前假设）

**libmetal FreeRTOS 后端的 mutex 实现与 OpenAMP 的 IPCC 中断处理不兼容**。

具体机制：
1. `OPENAMP_check_for_message()` → `MAILBOX_Poll()` → `rproc_virtio_notified()` → 内部需要获取 libmetal mutex
2. libmetal FreeRTOS 后端的 mutex 使用 `xSemaphoreTake(mutex->m, portMAX_DELAY)`
3. 如果 `mutex->m` 为 NULL（创建失败）或者在中断上下文中被调用，会导致死锁或 assert 失败
4. `IPCC_channel1_callback` 在中断中设置标志，但 `MAILBOX_Poll` 中的 `rproc_virtio_notified` 可能在处理过程中需要获取 mutex，而这个 mutex 在 FreeRTOS 环境下的行为与 generic 后端（简单关中断）不同

**为什么裸机版本能工作**：裸机版本没有 FreeRTOS 编译，libmetal 使用 generic 后端，mutex 实现是简单的关中断/开中断，不依赖任何 RTOS 功能。

### 待验证的解决方向

| 方案 | 思路 | 风险 |
|---|---|---|
| A | 从 CubeMX 移除 FreeRTOS，手动添加 FreeRTOS 源码（不添加 libmetal FreeRTOS 后端） | 低风险，但需要手动管理编译 |
| B | 替换 libmetal FreeRTOS 后端源文件为 generic 后端源文件 | 中风险，需要确认 generic 后端在 FreeRTOS 环境下是否安全 |
| C | 修复 libmetal FreeRTOS 后端的 mutex 实现（确保 mutex 正确创建和使用） | 高风险，需要深入理解 libmetal 内部 |
| D | 放弃 FreeRTOS，保持裸机架构（当前裸机版本功能完整） | 零风险，但失去 FreeRTOS 学习价值 |

**推荐方案 B**：将 `freertos_init.c`、`freertos_device.c`、`freertos_io.c`、`freertos_shmem.c` 替换为对应的 generic 版本。generic 后端使用关中断保护临界区，在单核 M4 上完全安全，且与 FreeRTOS 共存无冲突（FreeRTOS 的临界区也是关中断）。

### 最终根因

**问题 #21 实际上由两个独立问题叠加导致**：

**问题 A：内核 oops 导致 rpmsg_tty 驱动状态异常**

在排查过程中，`cat /dev/ttyRPMSG0` 在 M4 停止时触发了内核 oops（`Unable to handle kernel paging request`，见问题 #20 的 log）。这个 oops 破坏了 `rpmsg_tty` 驱动的内部状态，导致后续所有 RPMsg 通信都失败（`cat` 立即 EOF 退出，数据无法传输）。

**问题 B：FreeRTOS 的 libmetal 后端确实有兼容性问题**

libmetal FreeRTOS 后端的 `sys_irq_save_disable` / `sys_irq_restore_enable` 是空实现（模板文件未填写），导致临界区保护无效。同时 FreeRTOS 后端的 mutex 使用 `xSemaphoreCreateMutex`，静态初始化为 `{ NULL }`，可能导致未初始化的 mutex 被使用。

但由于问题 A 的存在，我们无法正确验证问题 B 的修复效果 — 每次测试都因为内核状态异常而失败。

**重启开发板后问题 A 消失，裸机版本立即恢复正常。**

### 经验教训

1. **开发板重启是排查嵌入式问题的重要步骤** — 内核 oops 后驱动状态可能不可恢复，必须重启
2. **避免在 M4 运行时直接 `cat /dev/ttyRPMSG0`** — 如果 M4 停止，`cat` 进程会触发内核 oops。应该先 `kill` 读取进程再停止 M4
3. **多个问题叠加时容易误判** — 问题 A（内核状态异常）掩盖了问题 B（FreeRTOS 兼容性）的真实状态，导致大量无效排查

### 当前状态

- ⬜ FreeRTOS 集成待重新验证（在干净的内核状态下）

### 最终解决：新工程 v1.2 + FreeRTOS 集成成功

在新工程 `EndoscopeM4_v1.2` 上，通过 CubeMX 添加 FreeRTOS（CMSIS_V1），配合以下修复：

1. `openamp_conf.h` — 启用 `VIRTUAL_UART_MODULE_ENABLED`
2. `virt_uart.c` — service name 改为 `"rpmsg-tty-channel"`
3. `sys.c` — 实现 `sys_irq_save_disable` / `sys_irq_restore_enable`（使用 PRIMASK）
4. main.c — 用 FreeRTOS 原生 API（`xTaskCreate`/`vTaskStartScheduler`）替代缺失的 CMSIS_V1 API
5. **开发板重启后在干净的内核状态下测试**

验证结果：

```

[M4] LED=50% (PWM=499)     ← ✅ A7→M4 指令
T244                        ← ✅ Task_Temperature 独立运行
```

### 问题 #22 更新：进一步排查

**重启后第一次启动 M4 的测试结果**：

### 问题 #22 根因确认

**LCD 背光由 TIM2 Channel 0 的 PWM 控制**（设备树 `timer@40000000` = TIM2 基地址 `0x40000000`）。

| C | M4 只配置 TIM2_CH3，不修改 PSC/ARR | 复用 A7 的 TIM2 配置，但 PWM 频率不可控 |

**当前状态**：根因已确认，待选择解决方案

### 问题 #22 最终解决

**解决方案**：移除 TIM2，LED 补光灯改用 TIM3 中断中的软件 PWM（GPIO 电平切换）。

- 占空比 0-100%，步进 10%

## 问题 #24：I2C1 与触摸屏引脚冲突

**日期**：2025-07

**来源**：M4 启动后触摸屏无响应

**根因**

触摸屏（Goodix 电容屏）和 SHT30 温湿度传感器共用 I2C1 总线（PF14/PF15）。M4 的 `MX_I2C1_Init()` 重新配置 I2C1 寄存器，破坏了 A7 侧触摸屏的 I2C 通信。

```
PF14(SCL) + PF15(SDA) ── I2C1 总线
  ├── 板载触摸屏（Goodix, addr 0x5D）── A7 Linux 驱动
  └── J1 扩展口 pin 13/14 ── SHT30（addr 0x44）── M4 HAL 驱动
```

**临时解决**：注释掉 `MX_I2C1_Init()`，禁用温度功能。

**永久解决方向**：
1. 确认 J1/J2 上空闲 GPIO 的物理位置（联系野火）
2. 用软件 I2C（GPIO 模拟）在空闲引脚上读取 SHT30
3. 或找到可用的 I2C2 引脚对（PD12+PD13 空闲但物理位置未知）

**当前状态**：温度功能暂时禁用，等待引脚确认

### 问题 #24 解决

**解决方案**：禁用硬件 I2C1，改用软件 I2C（GPIO 模拟）在 PA11/PA12 上读取 SHT30。

- 注释掉 `/boot/uEnv.txt` 中的 `stm-fire-can.dtbo`，释放 PA11(J1 pin 12) 和 PA12(J1 pin 11)
- CubeMX 中配置 PA11/PA12 为 GPIO Output Open Drain + Pull-up
- 实现软件 I2C（bit-bang）：`soft_i2c_start/stop/write_byte/read_byte`
- SHT30 接线从 J1 pin 13/14 改到 J1 pin 11(SCL)/12(SDA)

**验证结果**：温度 25.4°C / 湿度 63.2% 正常显示 ✅

**当前状态**：✅ 已解决

## 问题 #26：FreeRTOS 版本 M4 RPMsg 数据发送失败

**日期**：2025-07

**现象**

FreeRTOS 版本的 M4 固件启动后：
- RPMsg channel 创建成功（dmesg 显示 `ttyRPMSG0`）
- A7 侧 `RpmsgListener` 成功打开 `/dev/ttyRPMSG0`
- 但 A7 完全收不到 M4 的数据（无 `[RPMsg] read` 日志）
- A7 向 M4 发送数据时 `::write()` 阻塞（导致 Qt 主线程卡死）
- LED 灯不亮（软件 PWM 任务可能没有正常运行）

**对比**：之前有一次测试中温度正常显示（重启后第一次 start M4），但后续测试无法复现。

**已排除的原因**

| 排除项 | 验证方式 |
|---|---|
| 第二次 stop/start | 本次是重启后第一次 start |
| A7 侧代码问题 | `[RPMsg] Opened` 成功，`select()` 持续超时 |
| 网络/NFS 问题 | 文件复制成功 |

**根因分析**

FreeRTOS 版本中 `MX_OPENAMP_Init` 在调度器启动前调用，`VIRT_UART_Init` 也在调度器启动前。但 endpoint 的 `dest_addr` 需要通过 IPCC 中断处理 A7 的 name service reply 才能设置。

M4 的 endpoint 不会自动变为 ready，需要 A7 先发送一条消息触发 name service 流程。

**解决方案**

### A7 侧修复 1：sendCommand 非阻塞写入

`::write(/dev/ttyRPMSG0)` 在 M4 endpoint 没 ready 时会阻塞，冻住 Qt 主线程。改为临时设置 `O_NONBLOCK` 再写入：

```cpp
void RpmsgListener::sendCommand(const QByteArray &cmd) {
    if (m_fd < 0) return;
    int flags = fcntl(m_fd, F_GETFL, 0);
    fcntl(m_fd, F_SETFL, flags | O_NONBLOCK);  // 临时非阻塞
    ::write(m_fd, cmd.constData(), cmd.size());
    fcntl(m_fd, F_SETFL, flags);                // 恢复
}
```

### A7 侧修复 2：自动重试初始化定时器

MainWindow 构造函数中添加 500ms 间隔定时器，持续发送 LED 命令直到 M4 endpoint ready（收到回复后停止）：

```cpp
m_initRetryTimer = new QTimer(this);
m_initRetryTimer->setInterval(500);
connect(m_initRetryTimer, &QTimer::timeout, [this]() {
    if (++m_initRetryCount > 30) { m_initRetryTimer->stop(); return; }
    m_rpmsgListener->sendCommand("L10\n");
});
m_initRetryTimer->start();

// 收到 M4 回复后停止重试
connect(m_rpmsgListener, &RpmsgListener::temperatureChanged, [this]() {
    if (m_initRetryTimer->isActive()) m_initRetryTimer->stop();
});
```

### 验证结果

```

[RPMsg] read 14 bytes
[RPMsg RX] "[M4] LED=10%"     ← M4 endpoint ready，回复 LED 确认
[RPMsg] read 10 bytes
[RPMsg RX] "T243"              ← 温度正常上报
[RPMsg RX] "H571"              ← 湿度正常上报
```

温度、湿度、LED 亮度调节全部正常工作。

**文件**：`src/comm/rpmsglistener.cpp`、`src/ui/mainwindow.cpp`

**当前状态**：✅ 已解决
